# tsk-337: системный аудит инварианта order_position по ВСЕМ курсам платформы

**Дата:** 2026-07-21
**Скилл:** `/db-check`, режим чтения (read-only)
**Целевая БД:** прод — `learn_prod_db` (`learn`, хост `5.42.107.253`)
**Триггер:** переезд учеников с WP на LMS сегодня; вчера (tsk-332) нашли и починили точечную коллизию в дереве курсов 917/1080 (84 id) — этот аудит расширяет проверку на всю платформу.

## Вердикт: СИСТЕМА ЧИСТА

Инвариант `order_position` (внутри `course_id` не должно быть двух активных задач с одинаковой позицией) **не нарушен ни в одном курсе платформы**.

## Выполненные проверки

### 1. Полный охват — коллизии по ВСЕМ курсам (не только 917/1080)

```sql
SELECT course_id, order_position, COUNT(*) FILTER (WHERE is_active) AS active_collisions
FROM tasks
GROUP BY course_id, order_position
HAVING COUNT(*) FILTER (WHERE is_active) > 1;
-- => 0 строк
```

Проверено тремя независимыми формулировками (на случай нюансов группировки NULL/is_active):
- Дубли среди `is_active = true` — 0.
- Дубли без фильтра по `is_active` (включая мягко удалённые) — 0.
- Дубли среди строк с `order_position IS NOT NULL` (исключает риск ложной группировки NULL) — 0.
- `order_position IS NULL` строк в `tasks` — 0 (нечего было бы сгруппировать ложно).

Охват: **494 курса** с задачами, **7073** строк в `tasks` (6843 активных).

### 3. Побочная находка — пропуски `order_position` (НЕ баг, зафиксировано для полноты)

При проверке на пропуски (плотность 1..N) обнаружено **268 курсов** с неплотной нумерацией (например, курс 561: значения от 347 до 713 вместо 1..N). Это преимущественно тот же Крылов-инвентарь (25 курсов выше) плюс смежные.

**Это не нарушение инварианта из tsk-332 и не user-facing баг:**
- `order_position` используется ТОЛЬКО как внутренний ключ сортировки в `LearningEngineService._ordered_task_rows` (`ORDER BY order_position ASC NULLS LAST, id ASC`) и в `_order_key` — нигде не проверялось использование в качестве отображаемого номера "Задание N из M" (grep по `docs/frontend-contract-sa-com.md` — совпадений нет, значение не входит в контракт для фронтенда).
- Пропуски не создают неоднозначности для триггера `trg_set_task_order_position` (в отличие от дублей) — сортировка детерминирована независимо от плотности номеров.
- Фикс (уплотнение 268 курсов) не был запрошен, содержит риск (можно случайно оспорить группировку "урок → вопросы" там, где она осмысленна) и не относится к цели tsk-337 (коллизии перед переездом). Оставлено как есть.

## Заключение

- Коллизий `order_position` в рамках `course_id` — **0 по всей платформе** (494 курса, включая все курсы с сигнатурой множественного импорта).
- Фикс tsk-332 (курсы 1163/1164/1179) держится, регрессии нет.
- Новых фиксов не потребовалось — read-only разведка не нашла ничего для исправления.
- Побочная находка (пропуски у 268 курсов) — не блокер для переезда учеников, задокументирована на будущее.
