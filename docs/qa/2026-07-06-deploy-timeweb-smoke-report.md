# QA-отчёт: формальный прогон тест-плана деплоя Timeweb Cloud

**Источник тест-плана:** [docs/qa/2026-07-04-deploy-timeweb-smoke-testplan.md](2026-07-04-deploy-timeweb-smoke-testplan.md)
**Дата прогона:** 2026-07-06
**Связанные задачи:** tsk-005 (деплой), tsk-159 (аудит клиентов, ревью Кодекса)

## QA Mode
`paranoid` — прод-окружение с реальными учениками, риск release-critical путей.

## Execution Posture
`report-only`. Найденные дефекты не исправлялись в рамках этого прогона.

## Scope
Полный прогон тест-плана против **фактической прод-инфраструктуры**. Важное расхождение:
сам тест-план написан 2026-07-04 под архитектуру **App Platform**, которая была отброшена
2026-07-05 из-за неподдерживаемой связки App Platform+Load Balancer на стороне Timeweb
(см. `docs/ai/operator-runbook.md` R-006). Фактическая архитектура — VPS + Nginx + systemd
(`lms-spw-vds` 5.42.102.20, `tg-lms-vds` 72.56.247.22). Пункты плана, специфичные для App
Platform (кнопка deploy, встроенный rollback), не имеют буквального аналога — разобраны как
gap, а не как FAIL, если функциональный эквивалент отсутствует.

Также на момент написания плана предполагалось 5 TG-ботов (включая marketer) — по факту
их 4 (`admin/methodist/teacher/student`), у marketer нет entrypoint в `src/`.

## Environment
Прод. Единственное окружение — staging/тестового контура нет. Проверки — read-only API-вызовы,
SSH-диагностика (`systemctl`, `journalctl`, `redis-cli`), один контролируемый `systemctl restart`
живого бота (student), без выключения LMS/SPW.

## Blocking Issues
Нет S1, блокирующих текущую эксплуатацию.

## Non-Blocking Issues

1. **Нет отката (rollback) для VPS-деплоя** (пункт 16). `deploy/vps/deploy.sh` в LMS и SPW —
   однонаправленный (`git reset --hard origin/main` + restart), без сохранения предыдущей
   версии/тега для быстрого отката. При падении прод-версии после деплоя восстановление —
   вручную (`git reset --hard <sha>` по памяти/логу коммитов), не описано в
   `docs/ai/operator-runbook.md`. Severity: Significant (не критично при текущей частоте
   деплоев вручную, но при следующем факапе деплоя это будет мешать оператору под давлением).
2. **`alembic upgrade head` на чистой БД по-прежнему сломан** (пункт 10) — известный
   технический долг, уже зафиксирован в `TODOS.md` LMS. Не новая находка, отчёт лишь
   подтверждает, что untestable-статус не изменился.

## Untested or Blocked Areas
- Живой `/start` от реального Telegram-аккаунта на все 4 бота — нет доступа к Telegram-клиенту
  в этой сессии; заменено на проверку systemd + app-логов (polling активен, ошибки сети
  самовосстанавливаются).
- Полный TMA-логин с валидным `initData` — заменено на проверку границы (отклонение мусора).
- Реальный обрыв сети LMS↔TG_LMS — не воспроизводился намеренно (пришлось бы положить live LMS,
  неоправданный риск для учеников); заменено проверкой кода/конфига таймаута.
- End-to-end студенческий сценарий (magic-link → задание → ответ) — не повторялся в этом
  прогоне; последнее подтверждение — ручной smoke оператора 2026-07-05/06 после фикса cookie.

## Known Error Regression Check
Ни одна из известных прошлых проблем (cookie без `Domain`, ConfigParser `%`-баг, VK link-flow
redirect_uri, split-brain content_hub) не воспроизвелась повторно — конфиги и заголовки
подтверждены текущими значениями на серверах.

## Recommended Next Step
**Условный PASS** — release-критичные пути, которые можно было проверить без риска для живых
пользователей, подтверждены. Формально нельзя закрыть тест-план на 100%, т.к. часть пунктов
(живой Telegram-клиент, реальный обрыв LMS) принципиально требует либо операторского участия,
либо контролируемого окна с риском для прод-трафика.

Рекомендация: не блокировать текущую эксплуатацию, но:
1. Завести отдельный TODO на rollback-скрипт для VPS-деплоя (не блокер, но стоит сделать
   до следующего рискованного релиза).
2. При следующем удобном окне — попросить оператора вручную пройти живой `/start` на все
   4 бота и один полный TMA-логин с телефона, чтобы закрыть оставшиеся ⚠️-пункты уже не
   через прокси-évidence, а напрямую.
3. `/review-gate` по-прежнему не проводился для самого факта миграции на VPS — если политика
   проекта требует его перед объявлением деплоя полностью завершённым, стоит прогнать отдельно.
