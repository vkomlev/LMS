# tsk-355: сверка `difficulty_id` с встроенной меткой сложности в тексте книжных заданий Крылова

Дата: 2026-07-21. Продолжение бонус-находки из [tsk-354](2026-07-21-tsk354-difficulty-reconciliation.md#бонус-находка-вне-исходного-скоупа-wpтг--встроенная-метка-в-тексте-книжных-заданий).

## Контекст

92 из 108 заданий «Сборник Крылова» (`external_uid LIKE 'crylov:%'`, курсы 138-165) содержат
собственную метку сложности прямо в тексте условия (`task_content->>'stem'`), скопированную
из книги при импорте: `"Задание N Сборник Крылова С.С. вариант M Уровень {слово}"`. Это
третий независимый источник (помимо WP-навигатора и ТГ-разборов из tsk-354), похоже, никогда
не сверявшийся с `difficulty_id`. У `pdf:...crylov` заданий (440 шт., другая партия импорта)
такой метки нет вообще (0 из 440) — не покрыты этой сверкой.

## Метод

```sql
WITH stem_labels AS (
  SELECT id, course_id, external_uid, difficulty_id, order_position,
    (regexp_match(task_content->>'stem', 'Уровень\s+([А-Яа-яёЁ]+)'))[1] AS stem_word
  FROM tasks
  WHERE course_id BETWEEN 138 AND 165 AND external_uid LIKE 'crylov%'
    AND task_content->>'stem' ~ 'Уровень\s+(простой|легкий|лёгкий|легкая|лёгкая|средний|средняя|сложный|сложная)'
),
mapped AS (
  SELECT *, CASE lower(stem_word)
    WHEN 'простой' THEN 2 WHEN 'легкий' THEN 2 WHEN 'лёгкий' THEN 2 WHEN 'легкая' THEN 2 WHEN 'лёгкая' THEN 2
    WHEN 'средний' THEN 3 WHEN 'средняя' THEN 3
    WHEN 'сложный' THEN 4 WHEN 'сложная' THEN 4 ELSE NULL END AS stem_difficulty_id
  FROM stem_labels
)
SELECT * FROM mapped WHERE stem_difficulty_id IS NOT NULL AND stem_difficulty_id <> difficulty_id;
```

92 задания со встроенной меткой → **39 расходятся (42%)** с текущим `difficulty_id`
(словарь: простой/лёгкий→EASY(2), средний→NORMAL(3), сложный→HARD(4)).

## Анализ надёжности источника

**Систематическое направление**: в 36 из 39 случаев `difficulty_id` в БД ЗАВЫШЕН относительно
метки книги (на 1-2 уровня); только в 3 случаях (курсы 148, 162) — наоборот. Это не похоже на
случайный шум — скорее на импорт, где сложность проставлялась пакетно по номеру задания ЕГЭ
(типовая сложность задания N), без сверки с фактической меткой конкретного варианта в книге.
Само по себе это не делает метку книги безусловно надёжнее — для большинства из 39 это
**единственный источник** против БД, тот же случай, что «WP-only»/«ТГ-only» в tsk-354, которые
там сознательно не применили автоматом.

**Критерий tsk-354 для автоприменения — двойное независимое согласие двух источников против
БД.** Сопоставление 39 `task_id` с уже собранными в tsk-354 данными ТГ-канала дало:

## Не применено — 37 расхождений, решение за оператором

Полный список (task_id, курс, external_uid, было/по книге) — воспроизводится SQL-запросом
из раздела «Метод» (фильтр `stem_difficulty_id <> difficulty_id`, исключить task_id 9478 и 9564).
Систематическое направление (БД завышает на 1-2 уровня в 36/39, включая 34 из этих 37) наводит
на мысль о политике «доверять книге по умолчанию для `crylov:`», аналогично гипотезе о политике
«доверять WP для средний/лёгкий на kompege/sdamgia» из tsk-354 — но это решение масштаба
политики, не поштучный технический вызов, оставлено оператору.

## Ограничения проверки

- `pdf:...crylov` (440 заданий) — встроенной метки нет вообще, не покрыты этой сверкой.
- Пересечение с данными tsk-354 (ТГ/WP) проверено только для 39 расходящихся, не для всех 92
  заданий со встроенной меткой — возможны дополнительные тройные согласия за пределами 39
  (метка книги = ТГ = БД, т.е. уже всё верно) или другие сочетания, не относящиеся к текущему
  скоупу «расхождение с БД».
