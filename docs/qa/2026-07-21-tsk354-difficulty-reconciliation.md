## Метод

1. **WP-навигаторы** (22 из 25 курсов имеют блочную структуру простые/средние/сложные) —
   скачаны напрямую (curl), распарсены по `<h2/h3 id="...">` заголовку блока (два варианта разметки:
   `prostoy-uroven`/`sredniy-uroven`/`slozhnyy-uroven` и `prostye`/`srednie`/`slozhnye`) до следующего
   заголовка, извлечены все ссылки на kompege.ru/sdamgia.ru/kpolyakov.spb.ru/education.yandex.ru.
2. **ТГ-канал @cyberguru_ege** (1091 сообщение, source_id=1701256430 в ContentBackbone) —
   528 постов содержат explicit-разметку "Уровень {слово}". Словарь синонимов:
   EASY = простой/простая/легкий/лёгкий/базовый; NORMAL = средний/средняя (в т.ч. опечатка "срелний");
   HARD = сложный/сложная/трудный/трудная. Источник задачи в посте определён по ключевым словам
   (Комп ЕГЭ/КЕГЭ→kompege, Решу ЕГЭ→sdamgia, Поляков→polyakov, Яндекс→yandex, Крылов→krylov)
   плюс numeric ID в формате "{номер_задания}_{ID}". Возможно несколько постов на одно задание
   (переклассификация) — берётся ПОСЛЕДНИЙ по дате.
3. **Связка WP↔ТГ↔БД** — по числовому ID источника (kompege task?id / sdamgia problem?id /
   polyakov topicId), который совпадает во всех трёх местах (external_uid в БД, ссылка на
   WP-странице, число в тексте поста ТГ). Для книжных заданий (Сборник Крылова) — по паре
   (вариант, номер задания ЕГЭ), т.к. отдельного числового ID нет.

## Почему WP-only/ТГ-only не применены автоматически

Декомпозиция tsk-354 определяет «однозначный случай» как **WP = последняя ТГ-метка**,
не совпадающие с БД. Здесь конфликта между источниками нет, но подтверждение только ОДНИМ
источником (а не двумя независимыми) — риск того, что сам сопоставление (ID→задание)
дало ложное совпадение, ниже, чем у случаев с двойным согласием, но не нулевой. Оставлено
на решение оператора вместе со спорными случаями, а не применено по строгой букве правила.

## Бонус-находка (вне исходного скоупа WP/ТГ) — встроенная метка в тексте книжных заданий

92 из 108 заданий «Сборник Крылова» (`crylov:vXtY`) содержат СОБСТВЕННУЮ метку сложности
прямо в тексте условия (`"Задание N Сборник Крылова С.С. вариант M Уровень {слово}"`) —
это третий независимый источник, скопированный из самой книги при импорте, но, похоже,
никогда не сверявшийся с `difficulty_id`. **39 из 92 (42%) расходятся** с текущим
`difficulty_id`. Это НЕ входит в заявленный скоуп «WP vs ТГ», поэтому не проверялось и не
применялось здесь — рекомендуется отдельная задача (сверка `difficulty_id` с меткой,
встроенной в `task_content->>'stem'`, для всех `crylov:`/`pdf:...crylov` заданий, 548 шт.,
из которых только 108 `crylov:` имеют встроенную метку — `pdf:...crylov` её не содержат
вообще, там 0 из 440).

**Follow-up выполнен** — см. [tsk-355](2026-07-21-tsk355-difficulty-crylov-stem-reconciliation.md):
2 задания с двойным независимым согласием (книга = ТГ, оба против БД) применены; остальные 37 —
на решении оператора.

## Побочная находка: битый скрейп условия у части `wp_nav:`-заданий (40 шт.)

При чтении превью текста для таблиц выше замечено: у части заданий `task_content->>'stem'`
содержит не условие задачи, а служебный текст сайта-источника — "We're sorry but KEGE
doesn't work properly without JavaScript enabled" (kompege.ru, JS-заглушка) или шапку
sdamgia.ru ("СДАМ ГИА: РЕШУ ЕГЭ Образовательный портал..."). Проверено по всему диапазону
курсов 138-165: **40 заданий** с таким битым условием (в отчёте выше это, например,
task_id 2948-2954, 3759, 3779-3799). Ученик, открыв такое задание, видит служебный текст
вместо условия — это отдельный P0/P1-дефект данных, не связанный с `difficulty_id`
напрямую, но затрагивающий часть тех же заданий. Не исправлялось здесь (вне скоупа
tsk-354) — вынесено отдельным предложением на doing-фикс.
