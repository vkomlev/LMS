## Контракт неизменяемости ID ролей

### Правила

1. **ID ролей в таблице `roles` считаются неизменяемыми.**  
   Значения `id` закреплены за конкретными бизнес-ролями. Код и внешние системы (боты, интеграции) могут опираться на эти ID.

2. **Добавление новых ролей допускается только по согласованию при изменении бизнес-логики.**  
   Перед добавлением новой строки в `roles` необходимо:
   - обновить этот документ (раздел «Закреплённые роли»);
   - при необходимости обновить код (фильтры по ролям, маппинг имён в `app/repos/users_repo.py` и т.д.);
   - обеспечить заполнение ролей при развёртывании (скрипт `scripts/seed_roles.sql` / `scripts/seed_roles.py`).

3. **Запрещается:**
   - менять `id` существующих ролей;
   - удалять роли, на которые есть ссылки из `user_roles` или `access_requests`;
   - вводить новые роли без отражения в документации и, при необходимости, в коде.

## Закреплённые роли

Соответствие `id` и имени роли зафиксировано контрактом с фронтом (ТГ бот). Инициализация — см. [Восстановление БД](restore-database.md), шаг 4.

### Соответствие с ТГ ботом (обязательно к соблюдению)

| id | name в БД   | Роль (бот/UI)   | Бот (фронт)           |
|----|-------------|------------------|------------------------|
| 1  | `admin`     | Админ            | main_bots_admin        |
| 2  | `methodist` | Методист         | main_bots_methodist    |
| 3  | `teacher`   | Преподаватель    | main_bots_teacher      |
| 4  | `student`   | Студент          | main_bots_student      |
| 5  | `marketer`  | Маркетолог       | —                      |
| 6  | `customer`  | Заказчик         | —                      |

Роли 1–4 используются ботом для выбора сценария (админ/методист/преподаватель/студент). Роли 5–6 — дополнительные, для фильтров API.

В коде при фильтрации по роли допускаются русские синонимы (см. `app/repos/users_repo.py`): администратор/админ↔admin, методист↔methodist, преподаватель↔teacher, студент↔student, маркетолог↔marketer, заказчик↔customer. В таблице `roles` хранятся **английские** имена (`admin`, `methodist`, `teacher`, `student`, `marketer`, `customer`).

## Эндпоинты, связанные с ролями

Базовый префикс: `/api/v1`. Аутентификация: query-параметр `api_key` (см. [api-reference.md](api-reference.md)).

### 1. Справочник ролей (CRUD)

| Метод | Путь | Описание |
|-------|------|----------|
| GET   | `/roles/` | Список всех ролей (пагинация: `skip`, `limit`). |
| GET   | `/roles/{id}` | Одна роль по `id`. |
| POST  | `/roles/` | Создание роли (тело: `{"name": "..."}`). **Не использовать для стандартных ролей** — они создаются скриптом seed. |
| PUT   | `/roles/{id}` | Полное обновление (тело: `{"name": "..."}`). Допустимо только для смены отображаемого имени, **не менять `id`**. |
| PATCH | `/roles/{id}` | Частичное обновление. |
| DELETE| `/roles/{id}` | Удаление роли. Не применять к закреплённым ролям (1, 2, 3), если на них есть ссылки. |

**Схемы:** `RoleRead` (id, name), `RoleCreate` (name), `RoleUpdate` (name, опционально).

---

### 2. Роли пользователя (назначение / снятие)

Префикс: `/users/{user_id}/roles`.

| Метод | Путь | Описание |
|-------|------|----------|
| GET   | `/users/{user_id}/roles/` | Список ролей, назначенных пользователю. Ответ: массив `RoleRead`. |
| POST  | `/users/{user_id}/roles/{role_id}` | Назначить роль `role_id` пользователю `user_id`. Ответ: 204 No Content. |
| DELETE| `/users/{user_id}/roles/{role_id}` | Снять роль `role_id` у пользователя. Ответ: 204 No Content. |

**Ошибки:** 404 — пользователь или роль не найдены.

---

### 3. Список и поиск пользователей с фильтром по роли

| Метод | Путь | Параметр | Описание |
|-------|------|----------|----------|
| GET   | `/users/` | `role` (query, опционально) | Список пользователей с пагинацией и сортировкой. `role` — фильтр по имени роли: `student`, `teacher`, `methodist` (или русские эквиваленты). |
| GET   | `/users/search` | `q`, `role` (query) | Поиск по фрагменту имени (`full_name`). `role` — тот же фильтр по роли. |

Допустимые значения `role`: `student`, `teacher`, `methodist`, `admin`, `marketer`, `customer` (и русские: студент, преподаватель, методист, администратор, маркетолог, заказчик). Регистр не важен. Если роль не найдена в БД, возвращается пустой список (без ошибки).

---

### 4. Курсы пользователя с учётом роли

| Метод | Путь | Параметр | Описание |
|-------|------|----------|----------|
| GET   | `/users/{id}/courses` | `role` (query, опционально) | Курсы пользователя. `role=teacher` — только из `teacher_courses`; `role=student` — только из `user_courses`. Без `role` — объединённый список. |

Допустимые значения: `teacher`, `student` (или преподаватель, студент).

---

### 5. Заявки на доступ (access_requests)

Заявка связывает пользователя и **запрашиваемую роль** по `role_id`.

| Метод | Путь | Описание |
|-------|------|----------|
| POST  | `/access_requests/` | Создать заявку. Тело: `user_id`, **`role_id`** (обязательно), `flag` (опционально, по умолчанию `not_ready`). |
| GET   | `/access_requests/` | Список заявок (пагинация). |
| GET   | `/access_requests/{id}` | Одна заявка по id. |
| GET   | `/access_requests/flag/{flag}` | Список заявок с заданным статусом (`completed`, `rejected`, `not_ready` и т.д.), с полями пользователя и роли. |
| PUT/PATCH/DELETE | `/access_requests/{id}` | Обновление / удаление заявки. |

**Важно:** при создании заявки `role_id` должен существовать в таблице `roles` (например, 1–6). Иначе — нарушение внешнего ключа (500). После восстановления пустой БД необходимо выполнить заполнение ролей: `python scripts/seed_roles.py` (см. [restore-database.md](restore-database.md)).

## Инициализация ролей при развёртывании

После применения схемы БД (в т.ч. по [restore-database.md](restore-database.md)) таблица `roles` должна быть заполнена закреплёнными ролями:

- **Скрипт:** `python scripts/seed_roles.py` (из корня проекта, использует `DATABASE_URL` из `.env`).
- **SQL (альтернатива):** выполнить `scripts/seed_roles.sql` в целевой БД.

Скрипты вставляют строки с `id=1..6` и именами `student`, `teacher`, `methodist`, `admin`, `marketer`, `customer`; при повторном запуске обновляют имя по `id` (`ON CONFLICT (id) DO UPDATE`).
