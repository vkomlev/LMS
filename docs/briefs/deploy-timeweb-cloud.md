### Timeweb Cloud — обзор релевантных продуктов
- **App Platform** (`/docs/apps`) — git-based автодеплой, поддерживает Python/FastAPI и Node/Next.js
  из коробки, healthcheck-path, бесплатный технический домен + Let's Encrypt. Явной поддержки
  long-running воркеров без HTTP-порта (тип бота на long polling) в документации не найдено;
  общая рекомендация Timeweb и практика сообщества — Telegram-боты размещать на VDS, не PaaS.
- **DBaaS** (`/docs/dbaas`) — managed PostgreSQL, несколько логических БД и пользователей в
  одном кластере (значит `learn` и ContentBackbone можно на одном managed-инстансе), Valkey
  (Redis-совместимый) как managed-сервис, физические/логические бэкапы, приватная сеть.
- **Облачные серверы (VDS)** (`/docs/cloud-servers`) — root-доступ, NVMe, cloud-init,
  снапшоты; естественный вариант для TG_LMS (5 процессов) и/или self-hosted Postgres/Redis
  вместо DBaaS.
- **Домены/DNS/SSL** — управление в панели (`/docs/domains`, `/docs/ssl`), поддомены
  3 уровня (`api.learn.victor-komlev.ru`, `learn.victor-komlev.ru`) — стандартная DNS-запись,
  ограничений не обнаружено.
- Точные тарифы (цены VDS/DBaaS) не вытянулись из документации (динамическая страница/калькулятор) —
  нужна ручная проверка в панели Timeweb оператором на этапе выбора конкретного тарифа.

## 4. Ключевые находки по разделам

- Архитектура: 4 решения приняты оператором —
  1) конфигурация хостинга: **B — App Platform (LMS+SPW) + отдельный VDS (5 ботов TG_LMS) + DBaaS**;
  2) топология БД: **один кластер DBaaS, разные БД + разные пользователи/права** (learn / content_backbone);
  3) супервизия ботов на VDS: **systemd unit на каждый из 5 ботов, Restart=always**;
  4) автодеплой: **ручной триггер на первые недели** (пока Phase Y-1 в активной разработке),
     переключить на авто-деплой по push в main отдельным решением позже.
- Секреты: рекомендация (без голосования, нет реального trade-off) — сгенерировать НОВЫЕ
  prod-значения `MAGIC_LINK_SECRET`/`SESSION_SIGNING_KEY`/`FERNET_MASTER_KEY`/API-ключей,
  не переносить dev-значения из локального `.env`. Хранить: App Platform — через встроенный
  раздел "Переменные" (LMS/SPW), VDS — `.env` с правами 600 + `EnvironmentFile=` в systemd unit,
  никогда не коммитить.
- DRY по инфраструктурным файлам: 3 Dockerfile/CI пишутся отдельно на проект (не строить общий
  инфра-репозиторий-абстракцию для первого деплоя — преждевременно при одном цикле переноса).
- Ошибки и обработка: критический пробел закрыт решением по супервизии (см. Архитектура п.3).
- Безопасность: TG_LMS требует `.env.example` (сейчас только `.env` с реальными токенами) —
  обязательный шаг перед публикацией репозитория/передачей на VDS; `requirements.txt` TG_LMS
  в UTF-16 — обязательный фикс кодировки перед деплоем (encoding-guard).
- Данные и граничные случаи: Redis TG_LMS (FSM, db=0) и Redis LMS (rate-limit/session, db=2) —
  уже разведены по номеру БД в текущих `.env.example`; при переносе на VDS/managed Valkey важно
  сохранить разделение (не завести общий db=0 для обоих).
- Качество кода: см. DRY выше; иных находок, специфичных для инфраструктурного ревью, не выявлено
  (это не ревью diff, а ревью стратегии — код проектов не менялся).
- Тесты: тест-план см. `docs/qa/2026-07-04-deploy-timeweb-smoke-testplan.md` (создаётся ниже).
- Производительность: точные тарифы VDS/DBaaS не вытянулись из документации (динамическая
  страница-калькулятор) — решение по размеру тарифа вынесено в раздел 5 «Открытые решения»
  как operator-only (выбор платного тарифа = финансовое решение, не чисто техническое).
- Наблюдаемость: `SENTRY_DSN` уже в `.env` TG_LMS (опционально, не настроен) — рекомендуется
  включить на этапе деплоя, чтобы сбои ботов не были тихими даже при работающем systemd.
- Деплой: ручной триггер на старте (см. Архитектура п.4); риск — забыть передеплоить после
  merge, снижается тем, что процесс всё равно активно ведётся Виктором вручную на этой стадии.
- Долгосрочная траектория: выбор B (App Platform+VDS+DBaaS) обратим для веб-частей
  (LMS/SPW можно позже мигрировать на VDS без потери домена), но переход ботов с polling на
  webhook (опция C, отклонена) — отдельная задача на будущее, если 5 отдельных VDS-процессов
  станут узким местом.

## 5. Открытые решения
- **Размер тарифа App Platform / VDS / DBaaS** — не решено в этом ревью: цены Timeweb отдаются
  через динамический калькулятор/личный кабинет, не через статическую документацию. Операторское
  действие: зайти в панель Timeweb Cloud → подобрать минимальный тариф под ожидаемую нагрузку
  (учебный проект, десятки-сотни учеников) для каждого из 3 сервисов (App Platform LMS,
  App Platform SPW, VDS для ботов) + DBaaS (2 БД, 1 кластер). Согласовать с бюджетом до старта
  `executor-pro`.
- **VPC/приватная сеть между App Platform и DBaaS — РЕШЕНО (факт, не выбор):** по документации
  Timeweb Cloud, App Platform **пока не поддерживает** подключение к VPC — только публичный
  адрес для БД при использовании Apps (приватные сети для Apps заявлены как "план на будущее").
  Следствие: `DATABASE_URL`/`REDIS_URL` для LMS на App Platform будут указывать на публичный IP
  DBaaS. Публичный IP для БД — платная опция (ориентировочно 150 руб/мес за адрес, уточнить в
  панели). Требование безопасности: сильный пароль/сертификат на DBaaS-подключение, при
  возможности — ограничить доступ по IP-allowlist (firewall Timeweb), не полагаться только на
  пароль. VDS (боты) может подключаться к DBaaS/друг к другу через VPC бесплатно — это доступно
  сразу.
- **Домен ContentBackbone — РЕШЕНО:** только БД `content_backbone` переезжает на DBaaS,
  приложение (пайплайны) остаётся локальным на машине Виктора, подключается к prod БД по
  публичному DBaaS-хосту. Перенос самого приложения — см. TODO-кандидаты.
