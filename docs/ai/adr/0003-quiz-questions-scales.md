# ADR-0003: Квиз-вопросы со шкалами (SC_Qw / MC_Qw) — scoring, интерпретация, триггеры

- **Статус:** Accepted (согласовано оператором 2026-06-25)
- **Дата:** 2026-06-25
- **Связано:** tsk-053 (routing-квизы LMS/SPW — потребитель), tsk-031 (триггеры
  назначения — расширяем), tsk-120 / ADR-0042 ContentBackbone (публикатор).

## Контекст (проблема)

Типы `SC`/`MC` требуют **ровно один правильный** вариант. Для вопросов-предпочтений
и routing-квизов это бессмысленно: «выбери трек», «какой курс тебе подходит» не имеют
«правильного» ответа. Сейчас приходится назначать один вариант «верным» (костыль).

Нужны **квиз-вопросы**: за каждый выбор начисляются баллы **по шкалам** (scales).
Шкал может быть несколько. После прохождения квиза баллы по шкалам **интерпретируются**
в результат (рекомендация/трек), и по результату срабатывает **триггер** назначения курса.

## Решение

### 1. Новые типы вопросов
`SC_Qw` (одиночный выбор-квиз), `MC_Qw` (множественный выбор-квиз). Без «правильного»
варианта — вместо проверки начисляют баллы по шкалам.

### 2. Модель данных (task_content)
```jsonc
{
  "type": "SC_Qw",                       // | MC_Qw
  "stem": "Что тебе ближе?",
  "scales": ["информатика", "python"],   // объявление шкал (валидация ключей scores)
  "options": [
    {"id": "a", "text": "Разгадывать и быстро соображать", "scores": {"информатика": 2}},
    {"id": "b", "text": "Делать игры и ботов",            "scores": {"python": 2}},
    {"id": "c", "text": "И то, и другое",                 "scores": {"информатика": 1, "python": 1}}
  ]
}
```
- `scores` у варианта: вклад в одну или несколько шкал.
- Для `MC_Qw` суммируются баллы всех выбранных вариантов.

### 3. Scoring (без «правильности»)
- SC_Qw/MC_Qw — `auto_check=true`, но НЕ pass/fail (нет «верного» варианта,
  `is_correct=null`). На ответ движок проверки считает **вклад по шкалам** из
  выбранных вариантов и пишет в результат: `task_result.scale_scores = {scale: points}`.
- `solution_rules` для квиза: без `correct_options`; хранит `quiz.scales` (объявление)
  и режим (`single`/`multiple`).
- **`score` (важно для прогрессии):** Learning Engine гейтит завершённость задачи и
  курса по отношению `score/max_score >= 0.5` (`compute_task_state` /
  `compute_course_state` / `resolve_next_item`), а НЕ по `is_correct`. Поэтому квиз
  трактуется как обычная auto-задача на уровне прогрессии: при непустом ответе
  `score = max_score` (отвечен = выполнен), при пустом — `score = 0`. Иначе ученик
  зацикливается на вопросе-предпочтении, а курс с квизом не завершается.
  Маршрутизация (триггер `quiz_scale`) опирается на отдельный `scale_scores`, а не
  на `score`; `max_score` остаётся справочным (held at 1 публикатором).

### 4. Интерпретация (накопление по курсу)
Квиз = набор SC_Qw/MC_Qw в одном курсе (узле графа). После завершения попытки/курса
движок **суммирует `scale_scores` по всем квиз-задачам курса** у ученика → итоговые шкалы.
Интерпретация — это вычисление выигравшей шкалы (`argmax`) или прохождения порога.
Гранулярность накопления — **по курсу** (один вопрос — частный случай).

### 5. Триггеры (расширение assignment_rule)
Новый `trigger_event = "quiz_scale"`:
- `condition`: `{"scale": "информатика", "mode": "argmax"}` (эта шкала максимальна)
  или `{"scale": "информатика", "min_score": N}` (порог).
- `course_id` правила = курс-квиз (где копятся шкалы); `target_course_uid` = что назначить.
- Срабатывает на завершении попытки/курса (как `course_failed`), оценивает накопленные шкалы.
- Пример: квиз выбора трека → шкалы {информатика:2, python:0} → argmax=информатика →
  назначить `wp:vvodnaya-informatika`.

### 6. Сторона публикатора (ContentBackbone, отдельно)
- Контракт: блок `{type:"quiz", qtype:"SC_Qw"|"MC_Qw", stem, scales, options:[{text, scores}]}`.
- Эмиссия SC_Qw/MC_Qw задач с `scores` у вариантов.
- Триггеры: `assign_rules:[{on_scale:"информатика", mode:"argmax", assign:"wp:..."}]` →
  правила `quiz_scale`.

## Затрагиваемое в LMS
- `task_content` схема: типы SC_Qw/MC_Qw + `options[].scores` + `scales`.
- `solution_rules`: секция `quiz` (scales, mode) вместо `correct_options`.
- checking/scoring сервис: ветка для SC_Qw/MC_Qw — считать `scale_scores`, не «верно/неверно».
- `task_result`: хранение `scale_scores` (JSONB; есть `answer_json`/доп. поле).
- Движок правил (`assignment_rules_service`): `evaluate_rules_for_attempt` + `quiz_scale`
  (накопление шкал по курсу, argmax/threshold).
- Миграция: `trigger_event` CHECK — добавить `quiz_scale`.
- Валидация типов задач (enum) — добавить SC_Qw/MC_Qw.

## Решения (согласовано 2026-06-25)
1. **Гранулярность интерпретации — накопление по курсу.** Баллы по шкалам суммируются
   по всем квиз-задачам курса; интерпретация и триггер — по итогу прохождения. Один
   вопрос — частный случай.
2. **Объявление шкал — явное `scales:[...]`** в задаче; ключи `scores` вариантов
   валидируются против него (неизвестная шкала = ошибка валидации).
3. **Хранение `scale_scores`** — отдельное поле `task_result.scale_scores` (JSONB).

## Этапы
1. Типы SC_Qw/MC_Qw + scoring по шкалам + хранение scale_scores (+ миграция enum/поле).
2. Интерпретация (накопление по курсу) + триггер `quiz_scale` (+ миграция CHECK).
3. Публикатор CB: контракт quiz + эмиссия + триггеры on_scale (отдельная ветка).

## Обратная совместимость
SC/MC остаются как есть. Новые типы — аддитивны. Курсы без квизов не затронуты.
