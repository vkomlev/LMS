# Tech-Spec Y-4.2 — фикс фильтра «pending review» в LMS (R-3)

**Дата:** 2026-04-30
**Авторитет:** [LMS Y-4 backend spec](2026-04-30-tech-spec-Y4-sa-com-teacher-queue-backend.md), [ADR-0012 SA_COM manual review](../ai/adr/0012-sa-com-manual-review-in-mvp.md), [teacher-queue-states.md](../../../ContentBackbone/docs/design/teacher-queue-states.md), [cross-project R-3 finding](../../../ContentBackbone/docs/cross-project/contracts/lms-api.md#open-questions-cross-project-gaps)
**Trigger:** TG_LMS Y-4 S3 manual smoke 2026-04-30 — обнаружен rid=12 (`type=MC, is_correct=false, score=0/15, checked_at=NULL`), который попадал в `claim_next_review`. Преподаватель потенциально мог «Принять (правильно)» MC-задачу через grade endpoint и **переопределить автопроверку** — data corruption в `task_results.score / is_correct`.

---

## 1. Цель

Запретить попадание авто-проверяемых задач (SA / SC / MC) и уже-оценённых SA_COM в очередь ручной проверки teacher-бота. Это предотвращает data corruption через переопределение автопроверки и очищает SPW history-список «На проверке» от зомби-записей.

Пользовательский результат:
- Преподаватель в TG_LMS видит в «Следующая проверка» **только** SA_COM/TA-задачи, ожидающие ручной оценки.
- Ученик в SPW `/me/history` НЕ видит автопроверенные MC/SC/SA как `pending_review`.
- Существующие зомби-записи (старые task_results с `is_correct IS NOT NULL AND checked_at IS NULL`) санируются через M9 data-migration.

## 2. Контекст

**Затронутый репозиторий:** `D:\Work\LMS\` (FastAPI + SQLAlchemy 2.x async + Alembic + PostgreSQL 13+).

**Текущее поведение (verified из smoke 2026-04-30):**

- `app/services/teacher_queue_service.py::claim_next_review` (line 269) фильтрует `WHERE tr.checked_at IS NULL AND <ACL>` без проверки `is_correct` или типа задачи. JOIN на `tasks t ON t.id = tr.task_id` уже есть.
- `app/services/teacher_queue_service.py::get_pending_count` (Y-4 backend) — аналогичный фильтр.
- `app/services/task_results_service.py::list_pending_review` (используется `GET /api/v1/task-results/by-pending-review`) — фильтр без типа задачи.
- `app/services/teacher_workload_service.py` (или файл, реализующий `pending_manual_reviews_total` для `GET /api/v1/teacher/workload`) — фильтр без типа задачи.
- Автопроверка SA / SC / MC в `attempts/finish` flow создаёт `task_results` с `is_correct IS NOT NULL AND score IS NOT NULL`, но **`checked_at` остаётся NULL** (поле зарезервировано под ручную проверку). Это design choice, не меняется в этом ТЗ.

**Внешние зависимости:** PG 13+, Alembic head `m8_inbox` (Y-4). Нет внешних API.

## 3. Границы задачи

### 3.1. В скоупе

- 4 SQL-сайта (`claim_next_review`, `get_pending_count`, `list_pending_review`, `pending_manual_reviews_total`) — дополнить фильтр.
- M9 Alembic data-migration `m9_zombie_sanitize` — санация существующих зомби.
- Тесты: 4 regression-теста (по одному на каждый endpoint) + миграционный тест (upgrade + downgrade roundtrip + data-effect).
- Cross-project memory backsync (`contracts/lms-api.md` R-3 → resolved).

### 3.2. Вне скоупа

- **Producer-side фикс автопроверки** (выставление `checked_at = NOW()` при автопроверке SA/SC/MC): рассматривался как вариант B в R-3 анализе, отложен — потенциальная регрессия SPW history больше, чем риск R-3 после фикса. Возможный candidate для Y-5/Y-6.
- TG_LMS-сторона: после применения этого фикса не требует изменений (TG-бот только потребитель endpoint'ов; round-2 hotfix уже удалил legacy /manual-check escape hatch).
- SPW: не требует изменений (потребляет /me/history, который теперь будет показывать корректный status после M9).
- Изменение grade endpoint (`POST /teacher/reviews/{id}/grade`): не нужно — фильтр в claim_next предотвращает попадание автопроверенных, и без `lock_token` (никто не claim'ил) grade всё равно вернёт 409.

### 3.3. Что не трогать

- Логика автопроверки в `attempts/finish` (см. §3.2 — это producer-side, отдельный candidate).
- Schema `task_results` — без новых столбцов / индексов / constraints.
- ACL (REVIEW_ACL_SQL): уже Y-4.1 hierarchical, не меняется.
- /grade endpoint: атомарная транзакция остаётся как есть.

- LMS Python convention: type hints, RU docstrings, `logging` (не print), pytest async через `pytest-asyncio`.

"""
WITH cand AS (
    SELECT tr.id
    FROM task_results tr
    JOIN tasks t ON t.id = tr.task_id
    WHERE tr.checked_at IS NULL
      AND tr.is_correct IS NULL                          -- НОВОЕ Y-4.2

SET review_claimed_by = $2, review_claim_token = $3, review_claim_expires_at = $4

WHERE tr.id = cand.id

### 9.4. S4 — M9 Alembic data-migration

**Исполнитель:** `/db-check`
**Ревью:** `/techlead-code-reviewer` (data integrity, downtime risk)

**Файл:** `D:\Work\LMS\alembic\versions\20260430_020000_m9_zombie.py` (имя revision ≤ 32 символов: `m9_zombie_sanitize`).

**revision/down_revision:**
- `revision = 'm9_zombie_sanitize'`
- `down_revision = 'm8_inbox'`

**Логика upgrade:**

```python
def upgrade():
    op.execute("""
        UPDATE task_results
        SET checked_at = COALESCE(received_at, submitted_at, now())
        WHERE is_correct IS NOT NULL
          AND checked_at IS NULL
    """)
```

**Логика downgrade:**

```python
def downgrade():
    # Невозможно откатить: мы не знаем, какие checked_at были оригинально NULL
    # vs какие были выставлены этой миграцией. Помечаем downgrade как no-op
    # с явным предупреждением в логах.
    op.execute("SELECT 'm9 downgrade is no-op: zombie sanitization не откатывается'")
```

**Pre-implementation check:** через MCP подсчитать масштаб:

```sql
SELECT count(*) AS zombie_count,
       min(submitted_at) AS oldest,
       max(submitted_at) AS newest
FROM task_results
WHERE is_correct IS NOT NULL AND checked_at IS NULL;
```

Если `zombie_count > 10000` — обсудить с оператором (батч-миграция вместо single UPDATE). Если ≤ 10000 — single UPDATE безопасен (lock на 5-30 секунд).

**Concurrency / Idempotency:**
- Миграция идемпотентна: повторный upgrade не делает ничего (после первого `is_correct IS NOT NULL AND checked_at IS NULL` уже не выполнится).
- Lock на `task_results` coarse-grained (UPDATE без WHERE id IN — full scan). Проводить в downtime или off-peak hours.
- Не блокирует claim_next_review (тот использует `FOR UPDATE SKIP LOCKED` row-level), но во время M9 могут быть задержки на write-path к task_results.

## 10. Concurrency & Idempotency

| Источник риска | Митигация |
|---|---|
| **Race claim_next_review во время фикса** | SQL-фильтр работает в той же транзакции с `FOR UPDATE SKIP LOCKED`. Race-семантика не меняется. |
| **Двойной grade автопроверенной задачи** | После фикса задача не попадает в claim → нет lock_token → /grade вернёт 409 «не захвачена». Data corruption невозможна. |
| **Concurrent INSERT новой автопроверенной задачи во время M9 миграции** | M9 — single UPDATE без WHERE по id; lock на task_results coarse-grained. Concurrent INSERT блокируется до завершения миграции (1-30 сек). Acceptable для off-peak. |
| **Idempotency M9 upgrade** | `WHERE is_correct IS NOT NULL AND checked_at IS NULL` — после первого upgrade match'ит 0 строк. Повторный запуск безопасен. |
| **Idempotency M9 downgrade** | No-op с warning в логах (см. §9.4). Восстановление зомби невозможно — это design choice. |
| **Inflight grade на задаче с истекшим lock'ом** | До фикса: 409 (lock_token mismatch). После фикса: тот же 409. Поведение не меняется. |
