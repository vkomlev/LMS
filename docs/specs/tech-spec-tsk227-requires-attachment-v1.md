# Tech Spec tsk-227 — обязательное вложение проектных заданий (`requires_attachment`)

**Статус:** DRAFT (спек, до реализации) · **Дата:** 2026-07-15 · **Приоритет:** P1
**Задача:** tsk-227 (класс А3 фидбека флагмана, tsk-224) · **Автор:** spec-writer
**Репозитории:** LMS (foundation) · SPW · TG_LMS · ContentBackbone (cross-project docs)

---

## Цель

Сделать так, чтобы проектное задание, требующее артефакта (скриншот/файл), **нельзя было
зачесть без реального вложения** — форс на стороне LMS (источник истины), обязательная загрузка
файла в клиентах (SPW, TG_LMS). Авто-проверка ответа сохраняется, но зачёт наступает только
при наличии вложения.

## Границы

### Входит
1. **LMS:** поле `requires_attachment: bool = False` в `SolutionRules`; серверный форс на сдаче
   ответа; проброс флага клиенту в ответе task-state; регенерация openapi; тесты.
2. **Данные:** проставить `requires_attachment=true` целевым заданиям через `/db-check`
   (проектные миссии + скриншот-задания). Выполняется ПОСЛЕ деплоя клиентов.
3. **SPW:** обязательный upload-контрол для задач с флагом; блокировка submit без файла;
   контексты web и tg-app (wp-embed — read-only, форса нет).
4. **TG_LMS:** форс вложения (фото/документ) в диалоге бота перед приёмом ответа.
5. **Cross-project:** обновить `ContentBackbone/docs/cross-project/{contracts/lms-api.md,
   contracts/spw.md, CHANGELOG.md, STATE.md}`.

### Не входит
- Проверка **подлинности** вложения (что на скрине именно результат ученика) — вне scope;
  форс поднимает планку до «есть артефакт», не верифицирует его содержимое.
- Перевод заданий в ручную проверку (`manual_review_required`) — отклонён (очередь
  преподавателя не операционна на проде; сломает самотемповых).
- Множественные вложения на одну попытку / привязка вложения к конкретной задаче внутри
  много-задачной попытки — MVP работает на модели «одно активное вложение на попытку»
  (см. Риск R3).
- Переписывание формулировок миссий / ключевых слов (это отдельный класс, не входит в форс).
- Изменение TA/manual-логики и teacher-queue.

## Ограничения
- **Security-critical:** гейт сдачи ответа. `review-gate` обязателен до push (MANDATORY-триггер:
  публичный API schema/поведение + cross-project). Деплой прод — **operator-gated**.
- **Не ослаблять проверку:** отсутствие вложения при `requires_attachment=true` НИКОГДА не даёт
  зачёт (в т.ч. не проходит оптимистичный авто-пасс SA_COM — см. attempts.py:424-427).
- **Обратная совместимость:** default `false`; существующие задания без флага ведут себя как
  сейчас. Регенерация openapi не должна ломать существующие схемы.
- **Порядок деплоя (критично):** backend → клиенты → деплой → данные-флаги. Проставление флагов
  до готовности клиентов заблокирует учеников (форс без способа загрузить).
- **Координация:** чипы tsk-220/221/222/223 правят рабочие деревья SPW/LMS. Работать на своих
  ветках, коммитить pathspec'ом только свои файлы, не захватывать чужой uncommitted WIP
  (в т.ч. регенерированный `docs/openapi.json` — не давать пре-хуку затянуть чужие правки).

## Точка форса (архитектура)

**Сервер — источник истины.** Гейт в `app/api/v1/attempts.py::submit_attempt_answers`
(POST `/attempts/{id}/answers`), в цикле по ответам, ПОСЛЕ резолва задачи и ДО фиксации зачёта:

```
если task.solution_rules.requires_attachment И не attempt.time_expired:
    есть_вложение = _attempt_attachment_files(attempt_id) не пусто
                    # ТОЛЬКО реальный файл на диске; meta.attachments из тела запроса
                    # НЕ используется (клиентские данные, подделываются → обход форса)
    если не есть_вложение:
        # НЕ зачёт: не отдаём is_correct=True/score=max
        check_result = CheckResult(score=0, is_correct=None|False,
            feedback="Прикрепите файл-подтверждение — без вложения задание не засчитывается")
        # ответ сохраняется (answer_json), статус = не пройдено; ученик может дозагрузить и пересдать
```

- Форс срабатывает и для авто-SA_COM (перекрывает оптимистичный пасс attempts.py:424-427),
  и для SA/manual.
- **Проброс клиенту:** `requires_attachment` должен возвращаться в task-state, который читает
  клиент (`GET /learning/tasks/{id}/state` или эквивалент, через который SPW/TG получают правила
  задачи) — иначе клиент не знает, что показывать upload. Добавить поле в соответствующий
  response-schema (аддитивно, Optional).
- **Детект вложения (security):** ТОЛЬКО `_attempt_attachment_files(attempt_id)` — реальные файлы
  с префиксом `{attempt_id}_` в `settings.attempt_attachments_upload_dir` (кладёт эндпоинт
  `POST /attempts/{id}/attachments`). `answer.response.meta.attachments` НЕ используется: это
  клиентские данные из тела запроса, их можно подделать (`meta:{attachments:[{}]}`) и обойти форс
  без единого файла. Оба клиента грузят реальный файл до сдачи, поэтому доверие только диску
  честные пути не ломает. Вложение — per-attempt (см. R3).

**Клиент — UX-слой (не источник истины):** показывает обязательный upload, блокирует submit до
загрузки. Даже при обходе клиента сервер не зачтёт (гейт выше).

## План (фазы)

### Фаза 0 — Данные: точный список целевых заданий (read-only, `/db-check`)
1. Селектор целевых заданий: активные задания, где stem требует артефакта
   (`ILIKE '%приложи%'/'%скрин%'/'%прикрепи%'/'%скриншот%'`) И тип SA/SA_COM. Сверить с уже
   найденными ~37 миссиями + скриншот-задания чат-бот-курсов. Зафиксировать список ID.
2. Отдельно отметить задания, которые уже `manual_review_required=true` (форс для них — доп.
   слой поверх ручной проверки).

### Фаза 1 — LMS foundation (`/fastapi-api-developer`)
3. `app/schemas/solution_rules.py`: `requires_attachment: bool = Field(default=False, …)`.
4. `app/api/v1/attempts.py::submit_attempt_answers`: гейт форса (см. «Точка форса»). Явное
   поведение «нет вложения → не зачёт + feedback», без ослабления.
5. Проброс `requires_attachment` в task-state response-schema, который читает клиент.
6. Регенерация `docs/openapi.json` (`scripts/export_openapi.py`) — в том же коммите.
7. Тесты: `test_requires_attachment_gate.py` — (а) требует-вложение + нет файла → не зачёт;
   (б) требует-вложение + есть файл → авто-зачёт по слову; (в) флаг false → поведение как
   раньше (регресс); (г) SA_COM-миссия без вложения не получает оптимистичный пасс; (д)
   task-state отдаёт флаг. Прогон против dev-БД.

### Фаза 2 — Клиенты (параллельно после Фазы 1)
8. **SPW** (`/executor-pro`): в форме задачи с `requires_attachment` — обязательный upload
   (reuse POST `/attempts/{id}/attachments`), блок submit без файла, статус/подсказка. Контексты
   web + tg-app; wp-embed read-only (форса нет). `lib/api-types.ts` регенерировать. Unit/e2e.
9. **TG_LMS** (`/telegram-ux-flow-designer` → `/executor-pro`): в диалоге задачи с флагом —
   требовать фото/документ до приёма ответа; понятное сообщение; recoverability (переслать файл).

### Фаза 3 — Ревью + деплой (operator-gated)
10. `review-gate` PASS по всем трём репозиториям (security-critical + cross-project + public API).
11. Cross-project docs (lms-api.md, spw.md, CHANGELOG, STATE) — в коммитах.
12. **Деплой в порядке:** LMS backend → SPW + TG_LMS клиенты → верификация UX (upload виден,
    submit блокируется). Все деплои — go оператора.

### Фаза 4 — Данные-флаги (ПОСЛЕ деплоя клиентов, `/db-check`)
13. Скрипт `LMS/scripts/set_requires_attachment_tsk227.py` (dry-run→--apply, транзакция, verify):
    проставить `requires_attachment=true` заданиям из списка Фазы 0. Независимая MCP-верификация.
14. Smoke на 1-2 заданиях: без вложения не зачитывается, с вложением — проходит.

## Критерии готовности
- [ ] `SolutionRules.requires_attachment` есть, default false, openapi отражает (аддитивно).
- [ ] POST `/attempts/{id}/answers`: задание с `requires_attachment=true` без вложения → **не
      зачёт** (is_correct≠true, score<max) + внятный feedback; с вложением → штатный зачёт.
- [ ] Оптимистичный авто-пасс SA_COM НЕ обходит гейт (тест (г) зелёный).
- [ ] Флаг `requires_attachment` виден клиенту в task-state response.
- [ ] Флаг false / отсутствует → поведение идентично текущему (регресс-тесты зелёные).
- [ ] SPW: задача с флагом требует файл, submit заблокирован без него (web+tg-app), e2e зелёный.
- [ ] TG_LMS: диалог требует вложение до приёма ответа.
- [ ] Данные: список Фазы 0 помечен `requires_attachment=true`; проставлено ПОСЛЕ деплоя
      клиентов; verify через MCP = ожидаемое число строк.
- [ ] Cross-project docs обновлены; cross-repo grep старых контрактов = 0.
- [ ] review-gate PASS; деплой выполнен оператором в предписанном порядке.

## Риски
- **R1. Данные-флаги до клиентов → ученик заблокирован** (форс без upload). Мера: жёсткий
  порядок (Фаза 4 строго после деплоя клиентов); feedback объясняет, что нужно вложение.
- **R2. Регресс на существующих заданиях** (флаг по умолчанию поменял поведение). Мера: default
  false + регресс-тест (флаг отсутствует → как раньше); данные не трогаем до Фазы 4.
- **R3. Вложение per-attempt, не per-task** (много-задачная попытка: одно вложение «закрывает»
  все requires_attachment-задачи попытки). Мера: для флагмана миссии — одно-задачные попытки;
  зафиксировать ограничение в контракте; если всплывёт много-задачный кейс — доработка отдельно.
- **R4. Оптимистичный SA_COM-пасс обойдёт гейт** (attempts.py:424-427 ставит is_correct=True до
  проверки вложения). Мера: гейт выполняется ДО/ВМЕСТО оптимистичного пасса; тест (г).
- **R5. Захват чужого WIP чипов** (openapi.json, общие файлы SPW/LMS). Мера: свои ветки,
  pathspec-коммиты, контроль пре-хука openapi (не затягивать чужие правки — как в tsk-224).
- **R6. Клиент показал upload, но сервер поле не отдаёт** (рассинхрон контракта). Мера: проброс
  флага в task-state — часть Фазы 1, до клиентов; e2e сверяет реальный ответ API.
