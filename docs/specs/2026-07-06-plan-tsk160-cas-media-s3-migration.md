# Изменение: перенос CAS media-хранилища (ADR-0040) на S3-совместимое хранилище

**Дата:** 2026-07-06
**Статус:** Ready for review (нужно решение оператора по Q1/Q4 ниже)
**Задача:** tsk-160
**Проекты:** LMS, ContentBackbone

---

## Целевая возможность

LMS отдаёт студенту медиафайлы и файлы-вложения внешних задач (изображения sdamgia/
polyakov/wp_nav, .xls/.ods из polyakov) из S3-совместимого хранилища Timeweb Cloud —
корректно работает при физически раздельном размещении LMS (VPS) и ContentBackbone
(локальная машина оператора). Заменяет допущение ADR-0040 «CB и LMS на одной машине»,
явно помеченное как временное (Q4 плана tsk-110).

## Карта влияния

```
ContentBackbone (Windows, локально)                    LMS (lms-spw-vds, Linux VPS)
  cas_downloader.download_to_cas()
      ├─ sha256 (как сейчас, tmp-файл)
      └─ upload в S3 ──────────────────▶ Timeweb S3 bucket
                                            <bucket>/<sha[:2]>/<sha>.<ext>
                                                       │
  enricher.py → payload                                │
    stem_images["/api/v1/media/{sha}.{ext}"]           │
                                                       │
  bulk-upsert (POST /tasks/import) ──────────────▶ LMS DB (task_content)
                                                       │
  GET /api/v1/media/{sha}.{ext} ◀── SPW <img>/TG_LMS ─┘
         │
         └─ 307 Redirect → https://<bucket>.s3.timeweb.cloud/<shard>/<sha_ext>
            (браузер/httpx следует редиректу автоматически, без правок клиента)
```

Затронуты: **LMS** (эндпоинт + конфиг), **ContentBackbone** (downloader + конфиг +
секреты). **SPW/TG_LMS не затронуты** (подтверждено выше).

## Допущения и открытые вопросы

**Q1: Редирект (307 на публичный S3 URL) или проксирование (LMS сам стримит через себя, как сейчас)?**
→ **Рекомендация архитектора: РЕДИРЕКТ.** Совпадает с исходным решением ADR-0040 Q2 («без
авторизации — сha256-имена непрактично перебирать»), минимальная нагрузка на VPS (LMS не
гоняет файловый трафик через себя), кардинально проще код (нет нужды в S3 SDK на LMS
вообще — см. Q2). Trade-off: URL bucket становится виден клиенту напрямую — не критично,
т.к. контент и так публичный (та же логика, что и public bucket в Q2).
Альтернатива (проксирование) оправдана только если нужно скрыть факт использования S3
или добавить кэширующий слой на LMS — не нужно на данном этапе.

**Q2: Bucket публичный (public-read) или приватный + presigned URL?**
→ **Рекомендация архитектора: PUBLIC-READ bucket.** Прямое продолжение уже принятого
решения ADR-0040 Q2 (endpoint и так был public, без auth). При public bucket LMS-сторона
сводится к: валидация `sha_ext` (уже есть, оставить как есть) → детерминированное
построение URL по шаблону `https://<bucket>.s3.timeweb.cloud/<sha[:2]>/<sha_ext>` →
`RedirectResponse(307)`. **Не нужен S3 SDK, не нужны credentials на LMS вообще** — резко
снижает поверхность атаки (LMS prod не хранит S3-секреты). Presigned URL добавил бы
приватность, но требует генерации подписи на каждый запрос (доступ к secret key в рантайме
LMS, доп. задержка) — не оправдано для контента, который и так был public.
LMS всё равно НЕ проверяет существование файла в S3 перед редиректом (нет вызова S3 API) —
если файла нет, S3 сам отдаст 403/404 на прямой запрос; это ухудшение по сравнению с
текущим явным `404 "Медиафайл не найден"` от LMS — see риск ниже.

**Q3 (новый, не в исходном ADR-0040):** Оставлять ли CAS_MEDIA_ROOT/локальную запись в CB
как fallback для локальной dev-среды (Windows-машина оператора, тесты)?
→ **Рекомендация: оставить.** CB продолжает писать на локальный диск (как сейчас, без
изменений в `download_to_cas`) **и дополнительно** аплоадить тот же файл в S3 — двойная
запись, минимальная доп. сложность, сохраняет текущий dev-workflow (локальные тесты без
сети) и даёт локальный кэш на случай проблем с S3.

## Решение по дублированию

CB и LMS — разные кодовые базы. Общая точка знания — **имя bucket'а и базовый URL** —
хранить как одну env-переменную в обоих `.env` (`S3_MEDIA_BUCKET_URL` или аналог),
не как отдельный shared-пакет (нецелесообразно ради 1 URL-паттерна). Shard-логика
(`sha[:2]`) уже задублирована в CB (`cas_downloader.py`) и потребуется в LMS (URL builder)
— это уже существующий, обоснованный дубль (2 разные среды выполнения), фиксируется явно
в новом ADR как инвариант, а не оставляется implicit.

## Риски и меры снижения

| Риск | Вероятность | Митигация |
|---|---|---|
| LMS-редирект на несуществующий файл в S3 (LMS больше не проверяет существование сама) — вместо явного `404 "Медиафайл не найден"` клиент получит родное S3 403/404 с менее понятным телом ответа | Средняя | Задокументировать в ADR как принятый trade-off; при необходимости — добавить опциональный HEAD-запрос к S3 перед редиректом отдельной итерацией (не в этой фазе, чтобы не тянуть S3 SDK на LMS без необходимости) |
| Секреты S3 (access/secret key) утекут в git/логи | Низкая | Только через `.env` (gitignored), `encoding-guard`/review проверяют перед коммитом; тот же паттерн, что уже используется для остальных секретов CB |
| CB локальный upload успел, S3-upload упал (сеть/квота) — файл есть локально, но недоступен через прод LMS | Средняя | Best-effort upload с WARNING-логом (Фаза C, Шаг C2); при систематических сбоях — отдельный алерт/ретрай-механизм вне охвата этой фазы, но заложить точку расширения |
| Публичный bucket раскрывает больше, чем предполагалось (нет auth вообще) | Низкая | Не новый риск — уже принято решение ADR-0040 Q2 («endpoint public, sha256-имена не перебираются»); S3 наследует то же свойство |
| Stale-каталог `monolith/media_store` удалён по ошибке, если там были важные тестовые данные | Низкая | Явное подтверждение оператора перед удалением (Фаза D, Шаг D2) |

## Критерии Go/No-Go

**Go (переход к Фазе B и далее):**
- Фаза 0 завершена — bucket + ключи получены
- Q1/Q2 подтверждены оператором (или приняты рекомендации архитектора по умолчанию)

**No-Go (стоп, эскалация оператору):**
- LMS media-эндпоинт после Фазы B ломает существующий dev-режим (regression на старые тесты)
- CB upload в S3 систематически падает (не единичные сетевые сбои) — до диагностики причины
- review-gate FAIL перед закрытием

## Решение по UX-сложности

Медиафайлы отдаются как и раньше — через публичный URL, без нового UX (экранов/диалогов).
Единственное наблюдаемое клиентом изменение — HTTP 307 вместо прямого 200 (браузер и
`<img>`/`<a>` обрабатывают это прозрачно, без видимой разницы для пользователя).
Guard пройден: новых экранов нет, дополнительный «шаг» (редирект) не имеет UX-стоимости
(невидим пользователю), проверка не требуется.
