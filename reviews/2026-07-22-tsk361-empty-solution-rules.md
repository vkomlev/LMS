# tsk-361 — задания с пустым правилом проверки: 10 закрыто, тестовый артефакт снят, чек добавлен

Дата: 2026-07-22 · БД: прод `learn` (5.42.107.253) · Протокол: `/db-check` (read → dry-run → apply → независимая верификация)

## Что было

11 активных заданий на проде без правила проверки: 10 с `solution_rules` = JSON-null и 1 с SQL NULL.
Предикат `solution_rules IS NULL` JSON-null не ловит — отсюда более раннее ложное «на платформе чисто».

Из 10 JSON-null девять — отложенный хвост [[tsk-100]]: задания Яндекса, у которых в `task_content.source_url`
не прямая ссылка на задачу (`/ege/inf/task/<uuid>`), а **подборка с позицией**
(`/ege/collections/<collection_uuid>/task/<N>`). UUID задачи неизвестен → партия tsk-100 их пропустила.
Десятое (3759) — задание с решением оператора. Одиннадцатое (1025) — тестовый артефакт.

## Как разрешили позицию в подборке → UUID задачи

Оператор авторизован в education.yandex.ru в управляемом Chrome. Иконку «скопировать ссылку» не понадобилось:
при открытии задания страница шлёт в Метрику событие с полем `site-info.task_id` — это и есть UUID задачи.
Переход между позициями — клиентский (`history.pushState` + `popstate`), без перезагрузки, поэтому весь обход
трёх подборок — 3 загрузки страницы вместо 9.

Ответы взяты тем же способом, что в tsk-100: `POST /api/v5/gpttr`, тело —
`[{type:"get_task_by_id", task_id:<uuid>, params:{subject_id:"ac7328ca-…"}}]`, заголовок `x-csrf-token`
из `/api/v5/get-csrf-token`. Ответ отдаёт `markup.answer_control_layout[].content.correct_answers`.

последних 160 символов. Все 9 совпали (расхождение длины 3–9 символов — markdown-звёздочки Яндекса).

## Что записано на проде

**1. `solution_rules` для 10 заданий** — скрипт `scripts/backfill_solution_rules_yandex_tsk100.py`
(механика tsk-325/tsk-100, схема `SolutionRules`, нормализация `["trim","lower"]`),
маппинг `scripts/tsk361_answers.json`.
8 auto (`accepted_answers`, `manual_review_required=false`) + 2 manual (табличные, без `accepted_answers`).
Верификация после COMMIT независимым чтением MCP: у всех 10 `solution_rules` — объект, `max_score` совпадает
с `tasks.max_score`, у 8 ровно один `accepted_answers`, у 2 `manual_review_required=true`.

**2. 3759: `difficulty_id` 4 (HARD) → 2 (EASY) + реордер курса 158** — скрипт
`scripts/fix_task3759_difficulty_and_test_artifact_tsk361.py`. Решение оператора: задание элементарное
(840 бит ÷ 8 = 105 байт при 1 байт/с).
Реордер — та же ROW_NUMBER-логика THEORY→EASY→NORMAL→HARD→PROJECT с тайбрейком по текущему `order_position`,
триггер заглушён session-variable `app.skip_task_order_trigger` (не `ALTER TABLE … DISABLE TRIGGER` —
тот берёт ACCESS EXCLUSIVE лок на всю `tasks`). Переставлено 28 строк, 3759: 100 → 73 (конец блока EASY).
Проверено после COMMIT: 0 коллизий `order_position`, 0 нарушений межгруппового порядка, `order_position`
плотный 1..118.

**Координация с параллельным чипом.** Курс 158 одновременно пересортировывал tsk-354. Скрипт проверяет
`2262.difficulty_id = 4` (маркер завершения того чипа) ДО работы и падает, если чип не закончил, — иначе
реордер закрепил бы промежуточное состояние. Контроль пройден до и после.

**3. 1025 (курс 3) — `is_active=false`.** Подтверждено, что это тестовый артефакт, а не учебный контент:
`task_content` = `{"stem":"y4ps5-test","type":"SC"}` (тип SC без вариантов ответа), `max_score=10` при
единице у соседей, единственная попытка по курсу — тоже тестовая (`cancel_reason=test_isolated`).
Не «чинили» правило проверки: задания как учебной единицы не существует. Скрипт падает, если `stem`
окажется не `y4ps5-test`.

Состояние после: активных заданий с пустым `solution_rules` — **0** (было 11); единственная оставшаяся
запись — деактивированное 1025.

## Durable-проверка

`scripts/check_ungradable_tasks.py` (read-only, коды выхода 0/1/2) + еженедельная обёртка
`scripts/check_ungradable_tasks_weekly.ps1` (по образцу `check_section_order_weekly.ps1`, лог в
`logs/ungradable_tasks_check.log`, прод-DSN из `.mcp.json`, значение не печатается).

Чек ловит **три** формы пустого правила, а не одну:
1. SQL NULL;
2. JSON-null — `jsonb_typeof(solution_rules)='null'` (прежний `IS NULL` его пропускал, отсюда весь tsk-361);
3. правило-объект, но пустое — `auto_check=true`, нет `accepted_answers`/`correct_options`/`text_answer`/
   `custom_scoring_config`, при этом `manual_review_required=false`.

Чтобы чек попал в расписание, нужен разовый шаг оператора (Task Scheduler), команда — в разделе ниже.

## Находка сверх задачи

Форма 3 на проде — **231 активное задание** (228 `SA_COM` + 3 `SC_Qw`, 28 курсов). У всех нет ни
`source_kind`, ни `answer_raw`, поэтому ни партия tsk-325 (по `answer_raw`), ни tsk-100 (Яндекс) их не
захватывали. Ученик отвечает верно — система засчитывает неверно. Не чинилось здесь (нужен источник
ответов, а выдумывать нельзя): заведена **tsk-362**, P1. Пока она открыта, чек возвращает код 1 —
это осознанно, а не поломка чека.

Отдельно замечено и не трогалось: у всех этих заданий `task_content.difficulty_code` остался равен
`NORMAL`, хотя `difficulty_id` уже правился вручную (в т.ч. 3759, 3477). Это эхо импортного payload;
при повторном импорте из того же источника оно способно вернуть сложность назад. Класс системный
(касается всех ~2500 заданий ЕГЭ), не точечный — фиксирую как наблюдение.

## Что от оператора (1 шаг, ~1 минута)

Поставить чек в расписание рядом с существующим `check_section_order_weekly.ps1`
(запустить в PowerShell **от администратора**):

```
schtasks /Create /TN "LMS ungradable tasks weekly" /TR "powershell -ExecutionPolicy Bypass -File D:\Work\LMS\scripts\check_ungradable_tasks_weekly.ps1" /SC WEEKLY /D MON /ST 09:10
```

Ожидаемый результат: `SUCCESS: The scheduled task "LMS ungradable tasks weekly" has successfully been created.`
Результат прогонов — в `D:\Work\LMS\logs\ungradable_tasks_check.log`. Пока открыта tsk-362, в логе будет
строка «НАЙДЕНЫ непроверяемые задания: 231» — так и задумано.
