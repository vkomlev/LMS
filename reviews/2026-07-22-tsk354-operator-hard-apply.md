## Реордер `order_position`

Прямая запись идёт мимо `TasksService.bulk_upsert`, поэтому durable-хук из tsk-345
не срабатывает — реордер вызван явно в той же транзакции той же ROW_NUMBER-логикой
(`PARTITION BY course_id ORDER BY difficulty_id, тип-ранг, текущий order_position, id`),
тайбрейк по текущему `order_position` сохраняет ручные перестановки методистов внутри
группы сложности.

Курсы: 138, 139, 144, 154, 155, 157, 158. Переставлено строк: **111**.

Триггер `trg_set_task_order_position` заглушён **только** session-variable
`set_config('app.skip_task_order_trigger','true', true)` (is_local), НЕ через
`ALTER TABLE ... DISABLE TRIGGER` — последнее берёт ACCESS EXCLUSIVE лок на всю таблицу
`tasks` и блокирует живой трафик учеников (урок tsk-345/tsk-346).

## Дефект сверки, найденный при разборе (важно для оставшихся случаев)

ТГ-метки для заданий **без числового ID источника** (Яндекс, `wp_nav:`) сопоставлялись
в отчёте `docs/qa/2026-07-21-tsk354-difficulty-reconciliation.md` нечётким сравнением
текста и промахивались. Два подтверждённых промаха:

- **3477** — отчёт заявлял ТГ = EASY @2026-06-12; реальный пост 569 говорит
  «Уровень сложный». Спор источников был ложным, источники согласны, БД верна.
- **3759** — отчёт заявлял ТГ = EASY @2026-06-16; ТГ-поста для этого задания
  **не существует вовсе**.

Где числовой ID источника есть (kompege / sdamgia / polyakov) — 5 из 5 проверенных
совпали идеально.

**Следствие для оставшихся ~64 случаев EASY↔NORMAL:** доверять можно только строкам
отчёта с числовым ID источника; строки по Яндекс- и `wp_nav:`-заданиям требуют
переверификации, их выводы ненадёжны.

Дополнительно: 13 из 93 позиций отчёта — задания «Сборник Крылова», уже закрытые
в tsk-355; по ним отчёт устарел.

## Побочный дефект данных (зафиксирован, не исправлялся)

У задания **3759** не записан правильный ответ: `solution_rules` = JSON `null`,
то есть нет ни `accepted_answers`, ни правил проверки вообще. Проверка по всему
диапазону курсов 138–165: таких заданий **10**, все `wp_nav:`, все типа `SA_COM` —
`3470, 3471, 3472, 3473, 3474, 3475, 3476, 3477, 3478, 3759`. Ученик отправляет
ответ, но проверять его не по чему.

Это **уже открытая tsk-361** (заведена в тот же день, туда же входит 11-е задание `1025`
с SQL-null). Там же зафиксирована первопричина, из-за которой класс долго не находился:
предикат `solution_rules IS NULL` его **не ловит** — у сломанных заданий лежит JSON-null,
искать надо `jsonb_typeof(solution_rules) = 'null'`. Отдельная задача не заводилась.

**Важно про 3759:** здесь он оставлен HARD только потому, что был исключён из списка
оператора. В tsk-361 по нему уже принято решение — сложность **EASY** и ответ `105`.
Текущее HARD подтверждением не является; при применении tsk-361 понадобится реордер
курса 158.
