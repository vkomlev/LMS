## Что было

231 активное задание, у которых `solution_rules` — объект, но проверить им ответ нечем:
`auto_check=true`, ни `accepted_answers`, ни `correct_options`, ни `text_answer`, при этом
`manual_review_required=false`. Найдено durable-чеком из [tsk-361](reviews/2026-07-22-tsk361-empty-solution-rules.md).

**Что это значило для ученика (уточнение по коду, а не по догадке).** В
[checking_service.py](app/services/checking_service.py:558) ветка «нечем проверять» возвращает
`is_correct=None` и балл 0 — то есть ответ **не** помечается неверным. Но в **обязательную**
очередь преподавателя такое задание не попадает: [teacher_queue_service.py](app/services/teacher_queue_service.py:129)
берёт туда `SA_COM` только с `manual_review_required=true`. Итог — ответ зависает непроверенным
с нулём, и никто об этом не узнаёт. Поэтому вторая половина работы (перевод в ручную проверку) —
не косметика, а способ вернуть задание в поле зрения преподавателя.

Состав: 201 `tg:*` (**вся партия импорта из Telegram-канала целиком** — 201 из 201 без ответа),
21 `ext:calib:*`, 6 `lms:*` (авторские практикумы), 3 `authored:*` типа `SC_Qw`.

## Где взялись ответы

Задания `tg:*` — это разборы задач из канала «Подготовка к ЕГЭ по информатике». Сам пост ответа
не содержит (разбор — в видео), но содержит **каноническую ссылку на источник**. Отсюда четыре
канала восстановления:

| Источник | Как берётся ответ | Публично |
|---|---|---|
| kompege | `GET https://kompege.ru/api/v1/task/<id>` → поле `key` | да, без авторизации |
| sdamgia (Решу ЕГЭ) | `GET /problem?id=<id>` → `div.answer` («Ответ: X») | да |
| kpolyakov | `GET gen.php?action=viewTopic&topicId=<id>` → `div.hidedata` | да |
| Яндекс.Учебник | `POST /api/v5/gpttr`, `get_task_by_id` / `public_get_variant_request_item` | под учёткой Виктора ([tsk-100](reviews/../../reviews)) |
| близнец внутри LMS | то же условие у задания из другой партии, где ответ уже есть | — |

**Ключ источника** брался по убыванию надёжности: `external_uid` (21) → **ссылка из самого
поста Telegram** (116) → разбор шапки условия «Задание NN_<id> (КЕГЭ)» (58). Ссылки лежат в
ContentBackbone: канал импортирован дважды, и сущности сообщения (а с ними ссылки) есть только
в партии `telegram_desktop_json:*`, тогда как LMS ссылается на `tg_parser:*` — склейка идёт по
номеру сообщения, а не по `global_uid`.

## Сверка (гейт, без него ответ не пишется)

ID — сильный ключ, но пост мог быть про «аналог задания», а условие в LMS переписано. По каждой
паре сверялись **два независимых признака**:

1. **дословный фрагмент 60 букв** из середины условия LMS присутствует в тексте источника
   (сравниваются только буквы: KaTeX в LMS дублирует формулу — и отрисованную, и LaTeX-исходник,
   поэтому посимвольно тексты не совпадают никогда);
2. **все значимые числа источника** (3+ цифр) присутствуют в условии LMS.

Одного текста мало: задачи одного типа делят преамбулу дословно. Проверено на живом примере —
задание 2947 «Значение арифметического выражения…» текстом совпало с `wp_nav:14` (ответ 837),
хотя верный ответ 13; различает их только начинка. Поэтому близнец без единого значимого числа
не засчитывается вовсе.

Гейт сработал и в обратную сторону: у задания 3108 пост ссылался на `sdamgia:68243`, а там
«Поиск символов в текстовом редакторе» вместо условия про маску подсети — ответ не записан.

**Перекрёстная проверка.** У 49 заданий ответ независимо дали и сайт-источник, и близнец внутри
LMS. Совпало **49 из 49**, расхождений нет. Плюс ручная сверка выборки (3299/kompege 21604 → 264,
3324/kompege 32 → 14, 3014/kompege 1 → zyxw): условие и ответ совпадают, поле `number` источника
равно номеру задания ЕГЭ в шапке LMS.

## Второй проход (та же сессия, по решению оператора)

### Табличные ответы теперь хранятся
Решение оператора: таблицу хранить как последовательность значений через пробельные символы
(ячейки — пробелом, строки — переводом строки), а под ввод завести отдельный тип задания
`TBL_COM` по образцу `SA_COM` — задача **tsk-366**, внесена в пульт (`Root/docs/lms-product-roadmap.md`,
Эпик A).

**47 табличных ответов** сохранены в `task_content.answer_raw` (скрипт `tsk366_store_table_answers.py`):
22 из [[tsk-100]], 2 из [[tsk-361]], 23 из этой задачи. Правила проверки не менялись — задания
остаются на ручной проверке до появления `TBL_COM`; это сохранение данных, а не смена поведения.
Пример (3475, ЕГЭ №25): `1237678 95206 / 12300678 946206 / …` — 9 строк по 2 значения.

### Найден крупный дефект соседней партии → tsk-367
Задание 3285 получило «верный ответ» **`er`** — его дал близнец 2685 из партии Крылова.
Проверка партии: из 648 активных заданий `pdf:*crylov*` **222** имеют в `accepted_answers`
обрывок латиницы (`ae`, `oe`, `as`, `er`), а не ответ; у 224 слово «Ответ:» попало внутрь
условия. Причина — брак OCR при импорте из PDF, видно и по тексту («F(n) = 2- Еп - 1) - п»,
«Ответ: ss —"'(''CCS»). Это **хуже** пустого правила: ответ ученика сравнивается с мусором и
признаётся неверным.

Аварийная мера в рамках этой задачи: три задания, чей ответ пришёл из партии Крылова (3285 `er`,
3058 `42`, 3169 `2809`), возвращены на ручную проверку, прежний ответ сохранён в `answer_raw`
с пометкой источника (`fix_crylov_twin_answers_tsk367.py`). Разбор партии — **tsk-367** (P0).

### Добор: ещё 10 ответов
- **Перебор источника** (`tsk362_probe_unknown_source.py`) — там, где в шапке есть числовой ID,
  но нет имени источника («Задание 13_23749 Демоверсия 2026»): ID прогоняется по kompege,
  sdamgia и kpolyakov, засчитывается только если ровно у одного совпало условие. Нашлось 4
  (3042, 3055, 3057, 3059 — все kompege). Показательно, что 3056 с тем же ID 23749 **не** совпал:
  в посте ID указан от другого задания, гейт это поймал.
- **Позиция в подборке Яндекса** — `ext:calib:yandex:…:<подборка>:<позиция>` (6 заданий).
  Оказалось, подборка открывается тем же `public_get_variant_request_item`, что и вариант, и
  отдаёт все 27 задач с ответами. Разрешённые так UUID **совпали** с полученными в [[tsk-361]]
  совсем другим способом (поле `task_id` в событии Метрики) — независимое подтверждение обоих
  методов. Плюс закрыта новая позиция `a97d888a:13` → ответ 240.

8 из них записаны авто-ответом (`tsk362_upgrade_manual_to_auto.py`, guard: только задания,
которые сейчас на ручной проверке и без `accepted_answers`), 2 табличных — в `answer_raw`.

### Автор поста иногда сам предупреждает, что ответ источника неверен
Разбор остатка вскрыл ещё один класс: в тексте задания встречается прямое предупреждение —
«‼️ Внимание, предварительно есть ошибка в ответе!» (3119, kompege:27615, записали 736) и
«Внимание, в ответе на сайте ошибка. Ошибка идёт от решения Кабанова, где он округляет вверх,
где нужно округлять вниз» (3343 про polyakov:7926 — тот же ID дал ответ заданию 2937, записали 33).

Основной гейт проверял, что по ID лежит **та же задача**, но не читал оценку качества ответа.
Формально ключ верный, по существу автор курса с ним не согласен. Оба задания возвращены на
ручную проверку, ответ источника сохранён в `answer_raw` с пометкой
(`fix_source_error_warning_tsk362.py`). Не тронуты задания, где «ошибка» относится к ходу
решения в видео («долго искал ошибку», «ищем ошибки и исправляем») — там ответ под сомнение
не ставится.

## Третий проход: открываем файлы-приложения (решение оператора)

Ключ, которого не хватало раньше: **у источника файл есть**. `GET kompege.ru/api/v1/task/<id>`
возвращает `files[].url`, и файл качается без авторизации. Это дало гейт сильнее прежнего —
скачать данные, **решить задачу самому** и сверить с ключом источника: сходятся и данные, и
ответ, значит и задача та самая, и ответ верный.

Так закрыты: 3127/3128 (ЕГЭ-24 по файлу 2.8 МБ → 2657017), 3275 (ЕГЭ-3, база «Молочные
продукты» → 513000), 3409 (ЕГЭ-22, 13 процессов → 5), 3026 и 3035 (ЕГЭ-26, табличные).

Задачи без файла решены по полному условию; девять из них совпали с ключом источника
(3011, 3032, 3282, 3287, 3300, 3177, 3281, 3378, 3418), три подтверждены дублем задания
(3290 → 17 и 3291 → 22 совпали с 3472 и 3470 из [[tsk-361]]; 3329 → 91 — с вариантом Яндекса),
семь решены без внешнего подтверждения — расчёт записан в `origin` аудит-файла.

**Вариант 3 сборника Крылова** закрыт по таблице ответов книги (скан прислал оператор):
2834 → 165, 2836 → 23774, 2837 → 331, 2839 → 17. Та же таблица заодно подтвердила чистую
копию партии: вариант 1, задания 1–9 совпали **9 из 9**.

### Найдена системная причина остатка → tsk-369 (P0)
233 активных задания ссылаются в условии на файл-приложение, и у **108** из них `task_content.media`
пуст. Ученик такое задание решить не может в принципе — это не «нет ответа», а «нет самой
задачи». Механизм прикрепления рабочий (у 125 заданий файл на месте), беда в конкретных
партиях импорта.
