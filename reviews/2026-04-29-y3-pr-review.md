## Ревью перед мержем: 5 проблем (1 КРИТИЧЕСКАЯ, 4 ИНФОРМАЦИОННЫХ)

### Проход 1 — КРИТИЧЕСКИЕ

#### [КРИТИЧЕСКОЕ] `app/services/me_service.py:265-319` — `get_last_position` возвращает LAST activity вместо NEXT item (расхождение со спецификацией)

**Что:** Согласно CB tech-spec Y-3 §5.3 шаг 4 и LMS-side spec §5.3 шаг 4:

> 4. Иначе вызвать `learning_engine_service.resolve_next_item(user_id)` → next material/task в этом курсе → вернуть его

Текущая реализация возвращает информацию о ПОСЛЕДНЕЙ активности (`tr.received_at` или `smp.completed_at` MAX), а не о СЛЕДУЮЩЕМ шаге. UX-цель endpoint'а — виджет «Продолжить» (D5 восхищение), который должен направить ученика на следующее задание/материал, а не назад на уже-решённое.

**Эффекты:**
- SPW Continue widget после успешной задачи отправит ученика на ту же task → нажатие "Продолжить" возвращает к завершённому материалу
- При is_correct=true на последнем task — ученик видит уже-PASSED задание в виджете
- Расхождение с CB authority spec §5.3 — drift от контракта

**Концепт фикса:**
```python
from app.services.learning_engine_service import LearningEngineService

async def get_last_position(db: AsyncSession, user_id: int) -> dict | None:
    # ... existing last_activity SQL для определения course_id ...
    if course_row["state"] == "COMPLETED":
        return {... "type": "course_completed" ...}

    # NEW: вызов learning engine для NEXT item в этом курсе
    engine = LearningEngineService()
    next_item = await engine.resolve_next_item(db, student_id=user_id)
    # next_item.type ∈ {'material', 'task', 'none', 'blocked_dependency', 'blocked_limit'}
    # mapping next_item → LastPositionRead
    ...
```

**Решение требуется:** A) исправить сейчас (нужен дополнительный resolve через engine + extra тест на NEXT vs LAST) | B) пометить как Y-3.1 follow-up + добавить TODO в spec.

### Проход 2 — ИНФОРМАЦИОННЫЕ

#### [ИНФОРМАЦИОННОЕ] `app/api/v1/me.py:35` — `from app.services.fernet_service import encrypt_token` нарушает slim-controller паттерн

**Что:** `me.py` (controller layer) импортирует криптографический helper `encrypt_token` напрямую и вызывает его в `link_identity_vk:329-330`. Логика VK token encryption дублируется с `vk_oauth_service.get_or_create_user_by_vk`. По layered architecture эта логика должна жить в сервисном слое (например, в `vk_oauth_service` отдельным методом `link_vk_to_existing_user(...)`).

**Эффекты:**
- Дублирование business logic между `vk.py:get_or_create_user_by_vk` и `me.py:link_identity_vk`
- Если encryption pattern изменится (например, key rotation) — нужно править в 2 местах

**Авто-исправление невозможно:** требует архитектурного решения о новом сервисе `vk_link_service.py` или метода в `vk_oauth_service`. Не блокер для merge — текущий код корректен.

**Решение:** B) Skip (отложить как S3 техдолг). Пометить как «возможный рефакторинг при добавлении phone identity».

#### [ИНФОРМАЦИОННОЕ] `app/services/me_service.py:251` — magic string `'completed'` для `student_material_progress.status`

**Что:** Литерал `'completed'` хардкоден в трёх SQL-блоках (`_LAST_ACTIVITY_SQL`, `_COURSES_PROGRESS_SQL` materials_done CTE, `last_active_per_root` smp filter). Также используется в `learning_engine_service._first_incomplete_material:339`. Нет общей константы.

**Эффекты:** Опечатка в одном месте → silent bug. Risk низкий (тесты ловят), но единая константа `MATERIAL_STATUS_COMPLETED = 'completed'` была бы более defensive.

**Решение:** B) Skip (S3 — minor; существующий код Y-1 тоже хардкодит, сохраняем стиль).

#### [ИНФОРМАЦИОННОЕ] `app/api/v1/me.py:303-307` — `last_active_at` в response при `course_completed` может быть устаревшим

**Что:** При `state='COMPLETED'` мы возвращаем `last_active_at` из таблицы активности. Если course был completed давно, а user недавно начал другие курсы, этот `last_active_at` может вводить в заблуждение (datestamp old course's last activity, но виджет «Продолжить» предложит другой курс).

**Эффекты:** UX подозрение для frontend — поле семантически неоднозначно для course_completed состояния. SPW dev должен знать.

**Решение:** B) Skip — документировать в `contracts/lms-api.md`. Уже частично описано в `LastPositionRead.last_active_at: datetime`. Запись в FIXME для frontend tech-spec Y-3.

#### [ИНФОРМАЦИОННОЕ] `app/api/v1/auth/link_token.py:35` — отсутствует валидация `body.kind` дополнительно к Pydantic Literal

**Что:** Pydantic `Literal["email", "tg", "vk"]` гарантирует валидацию на входе FastAPI. Дополнительная проверка не нужна. Это **не находка**, а подтверждение корректности.

**Решение:** N/A — корректная реализация.
