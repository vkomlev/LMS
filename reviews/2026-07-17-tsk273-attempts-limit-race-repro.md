## Модель дефекта

Гейт 2.3b в `POST /api/v1/attempts/{id}/answers` (`app/api/v1/attempts.py:499-569`):
1. читает счёт попыток — `learning_engine_service.compute_task_state` →
   `SELECT COUNT(*) FROM task_results …` (`learning_engine_service.py:291-305`);
2. решает 409 при `BLOCKED_LIMIT`;
3. **потом** пишет `task_result` — `task_results_service.create_from_check_result`
   → `repos/base.py:144` `create(commit=True)` (коммит на КАЖДУЮ запись).

Между чтением счёта и записью нет сериализации. Каждый запрос идёт по своей сессии
`get_bare_db` → своё соединение (в тестах NullPool, на проде QueuePool), READ COMMITTED.
N одновременных ответов при «лимит-1» читают ОДИН и тот же счёт, все проходят гейт,
все пишут → `task_results` > limit. Перебор ограничен числом параллельных запросов,
но серверное правило слабеет.

## Почему очевидный прецедент не переносится (проверено пробником)

`pg_advisory_xact_lock` (`learning.py:302-305`) — транзакционный: отпускается на первом
же `commit`, а `repos/base.py` коммитит каждую запись. Значит замок отпустится ДО записи
результата и гонку не закроет.

Наивная замена «взять `pg_advisory_lock` (session-scope) наверху запроса» **тоже не
работает** — проверено пробником `/tmp/probe_lock.py`:

```
PROBE  try_advisory_lock got = True  => СЕССИОННЫЙ ЗАМОК ПОТЕРЯН после commit
CONTROL xact-lock after commit       => отпущен на commit (ожидаемо, метод пробы верен)
```

Держатель взял session-замок, сделал `commit` (как per-row `create`), и независимая
сессия смогла взять тот же замок. Причина: SQLAlchemy на `commit` возвращает соединение
в пул и берёт для следующего statement другое — session-замок остаётся висеть на старом
соединении, а сессия его уже не держит. Оба «дешёвых» варианта отпадают именно из-за
коммита-на-каждую-запись.

## Варианты фикса (цена) — вынесены оператору

- **A (рекомендую).** Точечная критическая секция на КАЖДЫЙ ответ: `pg_advisory_xact_lock`
  (ключ `user_id` × `hashtext("task:root")`) → пересчёт `compute_task_state` ПОД замком →
  409 при `BLOCKED_LIMIT` → `create(commit=False)` → один `commit` в конце секции. Чтение
  и запись атомарны в одной транзакции, замок держится всю секцию. Радиус: только шаг
  записи `task_result`; соседние гейты (квиз, вложения, optimistic-PASSED, таймлимит,
  правила назначения) не трогаются.
- **B.** Отдельное соединение под `pg_advisory_lock` на весь запрос (замок вне транзакции
  сессии), снять в `finally`. Логику эндпоинта не меняем вовсе; цена — +1 соединение на
  запрос и замок держится дольше (включая checking/диск).
- **C.** Весь эндпоинт в одну транзакцию + `commit=False` везде + один `xact_lock`.
  Максимальная целостность, но задевает soft-fail `db.rollback()` в правилах назначения
  (снёс бы записи всего запроса), `finish_attempt` (коммитит в середине) и вложения —
  ровно риск смежных гейтов. Не рекомендую.

## Реализация (вариант A — выбран оператором)

Атомарная критическая секция на КАЖДЫЙ ответ, `app/api/v1/attempts.py` шаг 2.4
(между гейтом вложений 2.3e и записью):

```
if effective_root_id is not None:
    pg_advisory_xact_lock(user_id, hashtext("task_id:root"))   # замок
    locked_state = compute_task_state(..., root=effective_root_id)  # ПЕРЕСЧЁТ под замком
    if BLOCKED_LIMIT: raise 409                                  # гонку выиграл конкурент
    create_from_check_result(..., commit=False)                 # запись без коммита
    await db.commit()                                           # коммит+отпуск замка атомарно
else:
    create_from_check_result(...)                               # null/ambiguous root — лимит и так не форсится
```

Замок xact-scoped и держится от пересчёта до записи, поэтому второй запрос пересчитает
счёт уже ПОСЛЕ записи первого и получит 409. Ключ `(user_id, hashtext("{task}:{root}"))` —
контендят только ответы на одно задание в одном корне.

**Важная деталь (грабли):** в ветке 409 НЕ делаем `db.rollback()`. Откат экспайрит
ORM-объекты `attempt`/`task` (rollback экспайрит всегда, даже при `expire_on_commit=False`),
и следующее же обращение `task.id` в логе дёрнуло бы ленивую загрузку вне greenlet-контекста
→ `MissingGreenlet`. Замок xact-scoped отпускается сам при закрытии сессии на выходе из
запроса — как и существующие 409/400 в 2.3b, которые тоже просто `raise` без отката.

`commit=False` прокинут в `task_results_service.create_from_check_result` → напрямую в
`repo.create(commit=commit)` (мимо `BaseService.create`, который коммитит всегда). Общий
базовый сервис не тронут.

## Risks / Follow-ups

1. Замок сериализует только ответы одного ученика на одно задание в одном корне —
   пропускная способность прочих ответов не страдает.
2. Живой прод не проверялся — правка не развёрнута. После развёртывания — живая проверка
   (ветвь А): два быстрых сабмита при исчерпанном лимите не должны дать перебор.
3. TG_LMS передаёт `root_course_id` не всегда (порождённая задача из tsk-269): на
   переиспользуемом узле `effective_root_id` может остаться None — там замок не берётся,
   но лимит там и так не форсится (осознанная граница tsk-264/269).
