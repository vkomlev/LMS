# tsk-395 — Устранение дублирования заданий ОГЭ-14 в курсе 1179

Дата: 2026-07-24. Прод `learn` (5.42.107.253). Продолжение [[tsk-392]] (находка о дублировании).
Профиль: /db-check (read → план → dry-run → транзакция → верификация → бэкап) + /fastapi-api-developer.

## Итог

Курс 1179 (ОГЭ-14) держал каждую из 25 задач источника в двух формах одновременно. По
решению оператора оставлена форма «целиком», форма подвопросов деактивирована. Курс приведён
к структуре соседнего 1178.

| | Было | Стало |
|---|---|---|
| Активных заданий | 78 | 28 |
| «целиком» (`sdamgia:oge:14`) | 25 | 25 |
| подвопросы (`oge:reshu:t14`, по 2 на задачу) | 50 | 0 (деактивированы) |
| авторские | 3 | 3 |

Соседний курс 1178 (ОГЭ-13): те же 28 = 25 «целиком» + 3 авторских — структуры совпали.

## Решение оператора

Из трёх предложенных вариантов оператор выбрал четвёртый: оставить форму «целиком» (одно
задание = одна задача ОГЭ, три вопроса сразу: 1–2 точный ответ + 3 диаграмма), сделать её
**SA_COM с обязательной ручной проверкой** — диаграмму принимает преподаватель, без неё
задание не зачтено.

## Что сделано (одна транзакция, только курс 1179)

**A. Дедуп.** 50 заданий-подвопросов `oge:reshu:t14:%` → `is_active=false`. Перед
деактивацией проверено, что на них нет данных ученика: 0 в `task_results`,
`student_task_progress`, `guest_attempt`. Деактивация ничего не рушит.

**B. Форма.** 24 задания «целиком» типа SA → **SA_COM**. SA_COM добавляет ученику
поле-комментарий («пояснение, код или фрагмент решения») — туда он опишет построенную
диаграмму. `manual_review_required` (уже был `true` на всех 25) и эталон ответа «32 546,82»
**сохранены**. Задание 7170 (`sdamgia:oge:14:11044`) — тип `TBL_COM` (таблица перелётов),
форма другая, тип не трогали; ручная проверка на нём уже стояла.

## Важное уточнение про «два ответа автопроверяются»

Движок (`app/services/checking_service.py:545-555`) при `manual_review_required=true`
**не выставляет авто-вердикт вовсе**: возвращает `is_correct=None, score=0`, ответ уходит в
очередь преподавателя (это сделано намеренно, чтобы оптимистичный авто-пасс Y-6 не зачёл
ученика до ручной оценки). Гибрида «два числа автопроверяются на стороне ученика + диаграмма
гейтит вручную» в движке сейчас **нет**.

Поэтому в текущей реализации два числа сверяет **преподаватель** по сохранённому эталону
«32 546,82» (эталон виден в очереди проверки), а не мгновенная авто-проверка у ученика.
Эталон сохранён специально: (а) преподаватель сразу видит верные числа; (б) если однажды
снять флаг ручной проверки — числа заработают на авто-проверке без доработок данных.

**Если нужен именно гибрид** (ученик сразу видит, что два числа сошлись, но задание всё
равно ждёт подтверждения диаграммы преподавателем) — это доработка scoring-движка
(новый режим «частичная авто-проверка + ручной гейт»), затрагивающая общий механизм всех
курсов. Отдельной задачей, не тихо в рамках дедупа. Здесь такого не делалось.

  (deep-link не режется по `is_active`, как и `recommended`-элементы); в поток курса дубль
  не возвращается, попыток по этим URL нет.
