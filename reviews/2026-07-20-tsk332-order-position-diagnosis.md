### 3. Источник — прослежено частично

`external_uid` обоих курсов имеет префикс `authored:<course-slug>:<lesson-slug>#q<N>`
(917 → `authored:chat-boty-tg-vk-max:...`, 1080 → `authored:oge-informatika:...`) —
это НЕ паттерн Google Sheets импорта (`PY-EGE-T1-03-SC` и т.п., см.
`app/services/sheets_parser_service.py`) и не паттерн Яндекс-извлечения. Похоже на
формат ручного/сгенерированного авторского контента (`/methodist`-подобный пайплайн).

Ключевая находка по семантике `order_position`: значения — это **номер вопроса внутри
урока** (`#q1→1`, `#q2→2`, `#q3→3`...), а НЕ курс-сквозная позиция. Для курса 1080 видно
явно: `zadanie-11: q1..q4 → 1,2,3,4`, `zadanie-12: q1..q4 → 1,2,3,4` (снова с 1) и т.д.

Это противоречит документированному контракту:
- `TaskUpsertItem` (`POST /api/v1/tasks/bulk-upsert`, схема в
  `~/.claude/skills/methodist/references/lms-wp-export.md` § 1.2.A) **вообще не содержит
  поля `order_position`** для заданий (оно есть только у Materials) — по контракту
  позиция должна была остаться `NULL`, и триггер сам проставил бы `MAX+1` без коллизий.
- Импорт через Google Sheets (`app/api/v1/tasks_extra.py`) тоже не выставляет
  `order_position` явно.

Значит, explicit `order_position` = "номер вопроса в уроке" пришёл из какого-то ДРУГОГО
инструмента/скрипта — либо более старой версии пайплайна авторского контента, либо
ручного payload'а, который скопировал конвенцию Materials (`order_position` = позиция
внутри сущности) туда, где её не должно быть. Прямого коммита/скрипта в `d:\Work\LMS`,
создавшего эти строки, не найдено (`git log --all -S "917"/"1080"/"authored:"` — пусто);
`audit_event` таких действий не логирует (только login/profile события). Значит импорт
шёл вне этого репозитория (внешний тул уровня ParseCourse/`methodist`, без git-трейла в
LMS) — дальше без доступа к истории того инструмента источник не сузить.

**Как физически получилась коллизия** (а не просто "неправильное число"): нормальный
путь (`TasksService.bulk_upsert` → `self.create()`) идёт через живой триггер
`trg_set_task_order_position`, и при explicit `order_position` триггер САМ сдвигает
существующие строки (`>= NEW.order_position` → `+1`) — коллизий в норме быть не может,
даже если каждый новый item явно просит `position=1`. Коллизия технически возможна
только если строки создавались/перезаписывались с активным
`app.skip_task_order_trigger='true'` (легитимный путь — `TasksRepository.reorder_tasks`,
`POST /courses/{id}/tasks/reorder`, см. `docs/database-triggers-contract.md` §15) —
т.е. кто-то вызвал bulk-reorder с "позицией внутри урока" вместо курс-сквозной позиции.
Это официальный API, использованный с неверными данными, а не обход защиты.

### 4. User-facing влияние — подтверждено, но не "непредсказуемо", а "неверная группировка"

`LearningEngineService._ordered_task_rows` (`app/services/learning_engine_service.py:807-821`)
— источник порядка для SPW/TG_LMS ("what's next") — сортирует строго
`ORDER BY order_position ASC NULLS LAST, id ASC`. При коллизии `id` — детерминированный
tie-break, не рандом. Но результат для курса 917/1080: **все "первые вопросы" всех
уроков идут одним блоком раньше любых "вторых вопросов"**, а не
`урок1.q1, урок1.q2, урок2.q1, урок2.q2, ...`, как задумано программой курса.

Пример по курсу 1080 (текущий эффективный порядок, восстановлен из `order_position ASC, id ASC`):
`zadanie-11#q1, zadanie-12#q1, zadanie-13#q1, zadanie-14#q1, zadanie-15#q1, zadanie-11#q2, zadanie-12#q2, ...`
— вместо `zadanie-11#q1, #q2, #q3, #q4, zadanie-12#q1, ...`.

`_order_key` (используется для "что после текущего задания" при возобновлении) тоже
опирается на этот порядок — застревания/зацикливания нет (ключ строго возрастает), но
последовательность вопросов внутри курса перемешана между уроками.

## Важный нюанс для плана бэкфилла

Наивный `ROW_NUMBER() OVER (PARTITION BY course_id ORDER BY id ASC)` (формула из
исходного бэкфилла миграции `20260521_120000` и из теста `test_t14`) **удовлетворит
инвариант БД** (уникальность 1..N), но **не восстановит** урок-за-уроком порядок:
вторая партия вопросов (`#q2`) создавалась отдельным проходом, и её `id`-порядок
НЕ соответствует программе курса (например, в 917: `bot-zdorovaetsya#q2` идёt раньше
`pridumay-svoego-bota#q2`, при том что между ними по `id` есть ещё десяток чужих
`#q2` — сам батч не отсортирован по уроку). Т.е. id-ROW_NUMBER technically закроет
`test_t14` для этих двух курсов, но оставит содержательный порядок "все q1, потом все
q2" как есть — то же самое поведение, что наблюдается сейчас (просто без коллизий).

Семантически верный порядок (урок → его вопросы по порядку) требует группировки по
слагу урока (часть `external_uid` до `#q`), с сортировкой уроков по позиции их первого
вопроса, и вопросов внутри урока — по номеру после `#q`. Это уже не чисто DB-инвариант,
а содержательное решение о порядке курса — нужен явный выбор оператора, см. вопрос ниже.

## ИСПРАВЛЕННЫЙ РАЗБОР (прод, дерево курсов)

### 1. Структура: 917 и 1080 — контейнеры, не листья

```sql
SELECT cp.course_id AS child_id, c.title
FROM course_parents cp JOIN courses c ON c.id = cp.course_id
WHERE cp.parent_course_id IN (917, 1080);
```

917 → 11 детей ("Тема 0..10"), 1080 → 16 детей ("Задание 1..16"), у части детей есть ещё
и свои дети (материалы-темы, переиспользуемые несколькими родителями — legit many-to-many
в `course_parents`). Задачи (`tasks`) лежат у детей/внуков, не у 917/1080 напрямую.
Traversal `LearningEngineService` идёт по дереву `course_parents` — родитель без
собственных материалов/заданий просто не задерживает ученика.

### 2. Полный охват — коллизия только в 3 курсах, все под 1080

Проверка `order_position` коллизий по ВСЕМ children + grandchildren обеих ветвей
(917: 11 детей + 34 внука; 1080: 16 детей + 21 внук) на проде:

```sql
SELECT course_id, count(*) FILTER (WHERE dup.order_position IS NOT NULL) AS collided_rows
FROM tasks t JOIN (
    SELECT course_id, order_position FROM tasks
    WHERE course_id IN (<все id детей+внуков 917 и 1080>) AND is_active
    GROUP BY course_id, order_position HAVING COUNT(*) > 1
) dup ON dup.course_id = t.course_id AND dup.order_position = t.order_position
GROUP BY t.course_id;
```

| course_id | title | collided_rows |
|---|---|---|
| 1163 | Задание 11. Поиск файла по содержимому | 6 |
| 1164 | Задание 12. Поиск файлов по маске | 4 |
| 1179 | Задание 14. Обработка данных в электронной таблице | 42 |

Ветка 917 (чат-боты) — чисто, 0 коллизий. Остальные 13 детей + 21 внук 1080 — чисто.

### 3. Источник — два прохода импорта внешних банков

`external_uid` в затронутых курсах — смесь трёх происхождений:
- `authored:oge-informatika:zadanie-N#qM` — исходный авторский контент (тот же, что
  раньше был диагностирован как "плоские строки под 917/1080" — на самом деле это ЕГО
  первые вопросы, остальные фактически лежат в детях 1163/1164/1179 и т.д.)
- `oge:reshu:tN:<id>` — первый импорт из внешнего банка (РешуЕГЭ)
- `sdamgia:oge:N:<id>` — второй импорт из другого банка (СдамГИА)

Паттерн коллизий: НЕ "всё в одну позицию" (как ошибочно предполагалось для плоских
917/1080), а точечные пары на РАЗНЫХ позициях по всему диапазону — например, в 1179
`order_position=19` занят одновременно `oge:reshu:t14:10568_2` (id 6951) и
`sdamgia:oge:14:10571` (id 7151). Похоже на то, что импорт `sdamgia` считал
`MAX(order_position)+1` по снимку ДО завершения импорта `oge:reshu` (или оба шли без
общей сериализации) — классическая гонка при вычислении `MAX+1` под read committed,
либо импортер явно указывал `order_position` из какого-то внешнего индекса без сверки
с уже занятыми значениями в курсе.

На dev этого импорта ещё нет вообще: `1163`=30 строк (прод 34), `1164`=30 (прод 34),
`1179`=50 (прод 78), `1178`/`1180` отсутствуют на dev целиком, `1181`=5 (прод 35).
**Прод в этих курсах ОПЕРЕЖАЕТ dev**, а не наоборот.

### 4. User-facing влияние — подтверждено, коллизия живая

`LearningEngineService._ordered_task_rows` сортирует `order_position ASC NULLS LAST,
id ASC` — при коллизии двух РАЗНЫХ вопросов на одной позиции результат детерминирован
(id-тайбрейк), явного дублирования/пропуска в выдаче нет, НО сам факт коллизии означает,
что при следующей вставке/удалении в этих курсах триггер `trg_set_task_order_position`
будет сдвигать соседей от НЕОДНОЗНАЧНОЙ базы (два разных `id` с одинаковым
`order_position`) — поведение с этого момента непредсказуемо (какая из двух строк
"считается" эталоном для сдвига — не специфицировано). Это уже нарушение инварианта
БД, который другой код (`reorder`, `after_task_id` resume) полагает истинным без
проверки на дубли.

### 5. План бэкфилла (прод) — принят минимальный, консервативный вариант

В отличие от плоских 917/1080 (где лесенка "внутри урока" была найдена и группировка
по уроку восстанавливала педагогический порядок), здесь такой структуры нет — просто
два слитых банка вопросов. Правильный минимальный фикс: **сохранить порядок, который
уже реально отдаётся ученикам** (`order_position ASC, id ASC` — та же сортировка, что
использует `LearningEngineService`), просто убрать коллизии и пропуски:

```sql
ROW_NUMBER() OVER (
    PARTITION BY course_id
    ORDER BY order_position ASC NULLS LAST, id ASC
)
```

Превью на проде (read-only) подтвердил: результат = 100% та же относительная
последовательность, что учащиеся видят сегодня (id-тайбрейк уже разрешает все
коллизии для чтения) — просто без дублей и без пропусков в номерах. Ничего не
переставляется по смыслу, только схлопываются пропуски и разводятся дубли.

**Требует отдельного подтверждения оператора перед записью — прод, `/db-check` Режим
записи, `DBCHECK_OK=1`.**
