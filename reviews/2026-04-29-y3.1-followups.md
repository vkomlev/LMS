### `app/services/me_service.py` — **критический bug fix в streak SQL**

`test_streak_logic.test_streak_continuous_3_days` обнаружил, что streak SQL из spec работает некорректно. Исходный шаблон:

```sql
-- БЫЛО (математически неверно для DESC-ордера):
d - (ROW_NUMBER() OVER (ORDER BY d DESC))::int * INTERVAL '1 day' AS grp
```

Для DESC-упорядоченных последовательных дней `{T, T-1, T-2}` с `rn={1,2,3}`:
- `T - 1*1d = T-1`
- `(T-1) - 2*1d = T-3`
- `(T-2) - 3*1d = T-5`

Все grp разные → каждый день в своей группе → streak всегда = 1.

Правильная формула для DESC: `d + rn*1d`:
- `T + 1*1d = T+1`
- `(T-1) + 2*1d = T+1`
- `(T-2) + 3*1d = T+1`

Все grp одинаковы → один run → streak=3. **Без новых edge-тестов этот bug ушёл бы на прод** — Y-3 acceptance §11 включал «`/me/streak` корректно считает streak с edge cases» только на словах.

```sql
-- СТАЛО:
d + (ROW_NUMBER() OVER (ORDER BY d DESC))::int * INTERVAL '1 day' AS grp
```

1. **R1 (Y-3.1 spec backsync)**: LMS-side spec `2026-04-29-tech-spec-Y3-learning-loop-backend.md` §5.4 содержит исходный (буггованный) SQL шаблон с `d - rn*1d` для DESC. CB authority spec §7.2.4 — тот же шаблон.

2. **R2 (production smoke fail-secure)**: новый path `LinkTokenServiceUnavailableError` не тестируется автоматически. В тестах ENV != "production", поэтому fallback на in-memory срабатывает как раньше. Smoke в production-like окружении был бы желателен (gated тест с `ENV=production` + mock-Redis-down).
   - **Митигация:** добавить gated test `pytest -m production_simulate` с моком `aioredis.Redis.set → Exception` + `monkeypatch.setenv("ENV", "production")` — Y-3.2 follow-up если стрезс-тест нужен.

   - **Митигация:** добавить per-user rate-limit на linking endpoints (Y-3.2 follow-up). Для текущего уровня атак — приемлемо.
