**Скиллы:** `/db-check` (привязка, прод), `/fastapi-api-developer` (эндпоинт), `/telegram-ux-flow-designer` (экран)
**Data Impact:** read (новый read-эндпоинт; миграций нет)

## Контекст

Шаг 1 (привязка Виктора id=2 к ученикам 4497/4498/4499 через `student_teacher_links`) выполнен и верифицирован ранее. Экраны teacher-бота показывали лишь общий свод + грубый список заданий («попыток: N, ср.балл»), без эталона оператора: **по каждому заданию — результат, с какой попытки сдал, баллы, подсказки; плюс свод по курсу**.

## Изменения

### LMS (backend)
- `app/services/task_results_service.py`
  - `get_detail_by_user(db, user_id, course_id=None)` — детальный разбор: per-task (is_correct, score/max, число попыток, **solved_on_attempt**, hints_used) + свод по корневым курсам + общий свод.
  - `_build_task_label(...)` — подпись задания (external_uid → title/stem → id), как в боте.
- `app/api/v1/task_results_extra.py`
  - `GET /task-results/detail/by-user/{user_id}?course_id=` — тонкий роутер, ACL идентична `/stats/by-user` (ученик — только свой разбор; service-key и роли admin/methodist/teacher — сквозной доступ).

**Ключевые семантические решения:**
- «С какой попытки сдал» = порядковый номер (1-based) первой ВЕРНОЙ строки `task_results` по паре (user, task), упорядоченной по `(submitted_at, id)`. Поле `count_retry` в данных = 0 у всех и не используется; повтор задания — это НОВАЯ строка результата (тот же attempt_id).
- Группировка по **корневому курсу** (`COALESCE(attempts.root_course_id, attempts.course_id)`), а не `tasks.course_id` — иначе подкурсы дробят свод (у Риты 74 задания → 6 подкурсов vs 1 корневой курс 1248).
- Учитываются только незакрытые попытки (`attempts.cancelled_at IS NULL`) — консистентно с остальной статистикой (этап 6).
- Баллы задания = лучший результат (`MAX(score)`), статус = `bool_or(is_correct)`.

### TG_LMS (teacher/methodist боты)
- `src/common/models.py` — модели `UserDetailRead` / `UserDetailOverall` / `UserDetailCourseSummary` / `UserDetailTaskItem`.
- `src/common/api_client.py` — `get_user_detail(user_id, course_id=None)` (auth через query `api_key`, как у `get_user_stats`).
- `src/common/services/task_results_service.py` — обёртка `get_user_detail`.
- `src/bots/common/dialogs/user_stats_base.py` — геттер `user_stats_task_list_getter` переписан на детальный эндпоинт; строка задания: `✅/❌ {label} · {score}/{max} · с N-й попытки · 💡K`; шапка — свод общий + по курсам. Удалён мёртвый код (`_task_label_from_content`, `USER_STATS_TASK_LIST_LIMIT`, N+1-загрузка меток через `tasks_service`).
- Экран доступен обоим ботам (teacher и methodist) — оба используют `user_stats_base`.

## Живой баг на проде и фикс (после первого деплоя)

Живой прогон teacher-бота под Виктором (Claude in Chrome, Telegram Web) вскрыл баг, который не ловился прямым curl эндпоинта:

- **Симптом:** экран «Детали по заданиям» не открывался (оставался на сводном экране). LMS-эндпоинт при этом отдавал корректные 200/15 КБ (nginx-лог подтвердил).
- **Диагностика по логам бота** (`/var/log/tg_lms/teacher.log`): `ValidationError: 1 validation error for UserDetailRead — user_id Field required, input_value={'items': [], 'total': 0, 'skip': 0, 'limit': 0}`. То есть клиент видел ПУСТОЙ page-envelope, а не тело ответа.
- **Первопричина:** `api_client._normalize_page_payload` (TG_LMS) имеет легаси-эвристику: ответ, где одновременно есть `user_id` и `courses` и НЕТ `items`, трактуется как список курсов студента (`UserCourseListResponse`) и схлопывается в `Page` — извлекая вложенные `course`-объекты, которых в моём формате нет → `{items:[], ...}`, все поля разбора теряются. Мой ответ `{user_id, overall, courses, tasks}` случайно попал под эту эвристику.
- **Фикс:** переименован ключ `courses` → `course_summaries` в ответе эндпоинта (LMS сервис + пример) и в модели/рендере бота. Эвристика больше не срабатывает (`if "items" not in data: return data`). Добавлен предупреждающий комментарий в сервис, чтобы ключ не переименовали обратно.
- **Валидация фикса:** сквозной прогон реального `_normalize_page_payload` над живым ответом LMS — не схлопывает (assert `normed == data`), `UserDetailRead` парсит, `user_id` сохранён, рендер корректен для 4497/4498/4499.
- **Урок:** прямой curl эндпоинта не эквивалентен клиентскому пути — клиент бота имеет слой нормализации пагинации, который надо учитывать при дизайне формата ответа. Живой прогон (не только smoke эндпоинта) обязателен.

### Второй живой баг: `reply markup is too long`

- **Симптом (после фикса #1):** экран падал с `aiogram.exceptions.TelegramBadRequest: reply markup is too long`.
- **Первопричина:** экран рендерил каждое задание отдельной инлайн-кнопкой (`ListGroup`). У Риты 74 задания × длинная подпись → инлайн-клавиатура превышает лимит размера Telegram.
- **Фикс:** отказ от кнопок-заданий в пользу ТЕКСТА с пагинацией — 20 заданий на страницу (`TASK_DETAIL_PAGE_SIZE`), кнопки «◀ Пред. / След. ▶ / ⬅️ Назад» (макс 3 кнопки). Каждая страница ~1900 символов (лимит текста 4096). Драйлдаун в per-task-stats убран (для per-student разбора не нужен; аггрегат по заданию остаётся отдельным экраном task_stats).
- **Валидация:** живой прогон под Виктором — экран открывается, показывает эталон (статус/баллы/«с N-й попытки»/💡), футер «Стр. 1/4 (задания 1–20 из 74)», навигация Пред/След работает.
- **Урок:** экраны со списком неизвестной длины должны пагинироваться с самого начала (риск был отмечен в follow-up, но реализован только после живого бага).

## Редизайн «по урокам» (по фидбеку оператора)

Первая версия показывала техничный `external_uid` без времени — «непонятно, какие задания и когда сделал студент; на уроке преподаватель должен видеть прогресс». Переработано:

- **Эндпоинт** теперь группирует ПО УРОКАМ (подкурс `tasks.course_id` + `courses.title`). Ответ: `{user_id, overall, lessons:[{lesson_id, lesson_title, tasks_total, tasks_solved, hints_used, first_at, last_at, tasks:[...]}]}`. По заданию добавлены: `question_no` (из `external_uid …#qN`), `text` (короткий `title`/`stem`), `solved_at`/`last_at` (ISO/UTC). Уроки — по времени начала, задания — по номеру вопроса.
- **Экран бота**: заголовок урока `📚 {название} ({решено}/{всего}) · {интервал} · 💡`, строки `✅ Вопрос N «текст…» · с K-й попытки · [💡] · ВРЕМЯ`. Время конвертируется в местное (Europe/Moscow, фолбэк +3), формат `HH:MM` (или `DD.MM HH:MM` если не сегодня). Пагинация — по урокам целиком (не разрывая), ~24 строки на страницу.
- **Валидация:** сквозное превью на живых данных (Рита 6 уроков/6 стр, Михаил 8 уроков/2 стр), нормализатор не схлопывает, страница ~1.3–1.6 КБ. Живой прогон под Виктором: экран открывается, показывает уроки с названиями, вопросы с текстом, время в МСК, пагинация работает.
- Коммиты: LMS `1d5c5df`, TG_LMS `c79f73f`.
