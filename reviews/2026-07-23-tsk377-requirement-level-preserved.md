# tsk-377 — переиздание через bulk-upsert больше не сбрасывает `requirement_level`

Дата: 2026-07-23 · Проекты: LMS (+ контракт для ContentBackbone) · Скилл: `/fastapi-api-developer`
Diff: [2026-07-23-tsk377-requirement-level-preserved.diff](2026-07-23-tsk377-requirement-level-preserved.diff)

## Objective

Уровень обязательности, проставленный методистом, обязан переживать любое переиздание
элемента курса — по всем путям записи, а не только через один конвейер.

## Дефект (как было)

`TaskUpsertItem.requirement_level` и `MaterialsBulkUpsertItem.requirement_level` имеют
дефолт `required`. Эндпоинт заданий отдавал в сервис `item.model_dump()` (без
`exclude_unset`), сервис материалов брал `item.requirement_level` напрямую — то есть
слой записи физически не отличал «поле не передали» от «передали `required`».

При этом поля нет ни у одного клиента: `TaskPayload` / `MaterialPayload` в
`D:\Work\ContentBackbone\monolith\lms_client\contracts.py`, парсеры Google Sheets
(`materials_sheets_parser_service.parse_material_row`, сборка `task_data` в
`api/v1/tasks_extra.py`) уровень не заполняют. Итог: любая доливка внешних задач,
round-trip гигиены стемов, backfill вложений молча возвращали элемент в `required`.
Так эродировала простановка tsk-112; tsk-347 нашла 37 сложных заданий, снова сидевших
в основном потоке ученика.

## Проверка материалов (пункт 1 задачи)

**Воспроизводится, тот же паттерн.** `MaterialsService.bulk_upsert` писал
`payload_data["requirement_level"] = item.requirement_level` и на UPDATE. Подтверждено
регресс-тестом: без правки `test_material_reissue_keeps_recommended` падает
(`recommended` → `required`). Дополнительно `_material_unchanged` считал материал
изменённым из-за дефолта, то есть возвращал `updated` там, где ничего не менял.

Путь Google Sheets для материалов уровня не касается вовсе — там дефекта нет.

## Выбор решения (пункт 2 задачи)

Взят **вариант (а)** — семантика «не передано = не менять», как у `order_position`.

| Вариант | Почему не он |
|---|---|
| (б) поле в payload-моделях ContentBackbone | Чинит один конвейер. Google Sheets, прямой вызов API и любой будущий клиент остаются с прежним дефектом — класс вернётся через необслуженный путь. Плюс требует, чтобы каждый экспортёр знал и заполнял чужое поле. |
| (в) хранить уровень отдельно от импортируемых полей | Новая таблица/колонка + миграция + правки во всех читателях (`learning_engine_service`, `me_service`, `manual_progress_service`, SPW, TG_LMS) ради задачи, которая решается семантикой записи. Цена несоразмерна. |
| **(а) `exclude_unset` + запись только при явной передаче** | Закрывает ВСЕ пути записи разом: CB, Google Sheets, прямой API. Нулевая миграция, контракт остаётся совместимым — клиент, который шлёт уровень явно, работает как раньше. |

Отличие от формулировки в задаче: поле **не** переведено в `Optional`. Дефолт
`required` в схеме сохранён (он правильный для CREATE и остаётся в OpenAPI как
документация), а различить «не передали» и «передали required» позволяет
`model_dump(exclude_unset=True)` на эндпоинте заданий и `item.model_fields_set` в
сервисе материалов (там эндпоинт передаёт сырые словари, и сервис валидирует их сам).
Это дешевле по контракту: нет ни `null` в схеме, ни изменения типа поля.

### Проверка `is_active` (предупреждение из задачи)

`exclude_unset` **не меняет** поведение активности: и до, и после правки UPDATE пишет
`data.get("is_active", True)`, то есть переиздание по-прежнему включает выключенный
элемент. Закреплено двумя сторожевыми тестами (`test_task_reissue_still_activates`,
`test_material_reissue_still_activates`) — они проходят и на старом, и на новом коде,
и упадут, если активность когда-нибудь поедет молча.

Сам по себе `is_active` — тот же класс дефекта (методист деактивировал дубликат →
доливка вернула его ученику), но чинить его здесь не стал: это отдельная смена
семантики публичного поля с более широким радиусом. Вынесено в follow-up.

## Стык с tsk-347 (не сломать блок сложных)

Тесты tsk-347 сразу поймали реальный конфликт: `test_downgrade_from_hard_returns_to_source`
упал. Инвариант блока сложных односторонний — он **ставит** `recommended` на входе, но
ничего не снимает на выходе. Раньше выход «чинил» сам дефект: дефолт `required` из
payload возвращал уровень. С «не менять» задание уезжало обратно в номерной курс, но
оставалось вне зачёта и вне next-item.

Добавлено симметричное исключение: если задание **выходит** из подкурса сложных
(`existing.course_id` — twin по `course_uid LIKE 'lms:tsk347:hard:%'`, целевой курс
другой), уровень снимается вместе с блоком → `required`. Проверка по данным, а не по
списку id, как и в `_resolve_hard_routing`. Запрос выполняется только при переезде
задания между курсами — на обычной доливке лишнего похода в БД нет.

Итог: `tests/test_tsk347_hard_twin_routing.py` — 6/6 зелёные, инвариант 1026 заданий
блоков сложных не тронут.

## Живой прогон на проде (после выката `e3ab273`)

Оба прогона — настоящим путём конвейера с сервера (`http://127.0.0.1:8000`, ключ из
`/opt/lms/.env`), payload собран из фактического содержимого элемента, как это делает
round-trip публикатора.

**Задание 3759** (`wp_nav:7:017606ca`, курс 158, `recommended`, вне блока сложных) —
прочитано через `GET /tasks/3759`, отправлено обратно ключами `task_payload_to_dict`
(без уровня):

```
BEFORE level=recommended course=158 diff=2 pos=73 active=True
UPSERT {'results': [{'external_uid': 'wp_nav:7:017606ca', 'action': 'updated', 'id': 3759}]}
AFTER  level=recommended course=158 diff=2 pos=73 active=True
```

Побочный эффект, **не связанный с правкой**: `task_content` после переиздания получил два
новых ключа-`null` (`table`, `scales`) — это обычная нормализация
`TaskContent.model_validate(...).model_dump()`, которая происходила и раньше на любом
переиздании. Смысловые поля (`stem`, `hints_video`, `accepted_answers: 105`) сохранены,
ключи не потеряны — сверено с соседним заданием того же импортёра.

**Материал 359** (`wp:mat:komlev:…:1`, курс 140, `recommended`) — payload публикатора CB
(`material_payload_to_dict`: с позицией и активностью, без уровня):

```
BEFORE level=recommended pos=1 active=True
UPSERT {'processed': 1, 'updated': 0, 'unchanged': 1, …}
AFTER  level=recommended pos=1 active=True   content identical: True
```

`unchanged` вместо `updated` — работает и вторая половина правки: отсутствие уровня в
payload больше не считается изменением.

## Risks and Follow-ups

1. **`is_active` при переиздании** (тот же класс, не тронут намеренно): доливка включает
   элемент, выключенный методистом. Прод-свидетельств срабатывания нет (деактивированные
   в tsk-112 `tg:ege:*` до сих пор выключены), но защиты тоже нет.
2. **`order_position` материалов**: `MaterialsService.bulk_upsert` кладёт в UPDATE
   `order_position` как есть, включая `None`. По контракту триггеров
   (`docs/database-triggers-contract.md` §7) `NULL` при UPDATE = «поставить следующий
   номер», то есть переиздание без позиции утаскивает материал в конец курса. У заданий
   этот случай закрыт (tsk-345), у материалов — нет. На проде `NULL`-позиций нет, но
   порядок мог разъезжаться незаметно.
3. Ручной `curl` с полным телом (включая `"requirement_level": "required"`) по-прежнему
   перезапишет уровень — это корректная семантика «явное побеждает», а не дефект.
