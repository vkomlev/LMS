## Контекст

Находка ревью tsk-264, за рамками её охвата. Подтверждено по коду до правки:
`POST /api/v1/attempts/{id}/answers` писал `task_result` без проверки
`attempts_used >= limit_eff`. На приёме были только: владелец попытки, вхождение
задания в дерево корня, совпадение типа ответа, повтор квиза (409), таймлимит.

`attempts_used` / `attempts_limit_effective` / `BLOCKED_LIMIT` считались только для
ВЫДАЧИ — `learning_engine_service.compute_task_state` (next-item, `/learning/tasks/{id}/state`)
и `me_service._SYLLABUS_TASKS_SQL` (syllabus). «Задание заблокировано» было гейтом
интерфейса и маршрутизации, а не серверным правилом: клиент, зовущий API напрямую,
отвечал сколько угодно раз.

**Решение оператора (2026-07-17):** это дыра, сабмит обязан отдавать 409.

## Что сделано

Гейт в `app/api/v1/attempts.py`, шаг 2.3b — после проверки квиза, до `check_task` и до
записи результата. Приём спрашивает **тот же** `compute_task_state`, что и выдача, и
отдаёт 409 при `state == "BLOCKED_LIMIT"`.

**Почему через сервис, а не своим SQL:** вторая копия формулы лимита неизбежно
разошлась бы с первой (`student_task_limit_override`, квиз-приоритет, `PASS_THRESHOLD_RATIO`),
и выдача с приёмом начали бы противоречить друг другу — интерфейс показывает
«попытка есть», сервер отказывает.

### История ревью (важно для читателя)

Первая версия правки была **ОТКЛОНЕНА** независимым рецензентом с двумя блокирующими
находками; обе он **воспроизвёл пробником**, а не вывел из чтения кода. Обе исправлены.

- **Б1 — гейт обходился удалением одного поля.** `course_id` в теле `POST /attempts`
  опционален; без него попытка создавалась с пустым корнем, и гейт (как и проверка
  дерева 2.1.1) молча не выполнялся. Клиент отвечал бесконечно на любое задание —
  ровно та модель угрозы, которую задача закрывает. Воспроизведено: коды
  `[200,200,200,200,200]` сверх лимита, `task_results` 3 → 8.
  **Фикс:** корень доопределяется по заданию (`resolve_attempt_root` по `task.course_id`),
  а не берётся только у попытки. После фикса — `[409,409,409,409,409]`, `task_results` 3 → 3.
- **Б2 — переиспользуемый узел оставался бесконечным.** При неоднозначном пути (узел
  под несколькими активными курсами ученика) корня нет → лимит не форсился. Моё
  обоснование «результаты без корня не попадают ни в один корень» было **неточным**:
  это верно для СЧЁТА, но прогресс (`PASSED`) корнем не фильтруется вовсе. Значит
  перебором добывался зачёт в том самом корне, где ученик заблокирован. Рецензент
  также показал, что развилка «пропустить либо считать по всем курсам (жалоба A7)»
  была **ложной** — есть третий вариант: спросить корень.
  **Решение оператора:** 400 «укажите root_course_id», но **только когда лимит на кону**
  (ученик заблокирован хотя бы в одном своём корне). Честный ученик с оставшимися
  попытками 400 не видит.

### Границы поведения (осознанные, не упущения)

| Случай | Ответ | Почему |
|---|---|---|
| Лимит исчерпан, задание не сдано | **409** | собственно дыра |
| Попытка без `course_id`, узел однозначен | **409** | Б1: корень доопределяется по заданию |
| Путь неоднозначен, лимит исчерпан в каком-то корне | **400** «укажите root_course_id» | Б2: гадать нельзя, пропускать нельзя |
| Путь неоднозначен, попытки есть | 200 | цена падает только на подозрительный случай |
| Сдавший ученик (`PASSED`), попытки исчерпаны | 200 | `compute_task_state` отдаёт `PASSED` раньше `BLOCKED_LIMIT` — приём ведёт себя как выдача |
| Тот же узел под другим корнем, там попытки есть | 200 | tsk-264: попытки не пересекаются, иначе возвращается жалоба tsk-261 A7 |
| Квиз (SC_Qw/MC_Qw) | своё 409 (tsk-124) выше | формулировка точнее — «одна попытка навсегда» |

Старые попытки с пустым корнем (на проде 7) лимит **не начинают расходовать задним
числом**: счёт идёт по `a.root_course_id = :root`, их результаты по-прежнему не попадают
ни в один корень. Семантика tsk-264 цела.

## Risks / Follow-ups

1. **Гонка (TOCTOU).** Гейт читает счёт и пишет результат не под блокировкой: два
   одновременных запроса при «2 из 3» оба пройдут проверку и запишут 4 результата.
   Перебор ограничен (не бесконечность), но правило слабеет. Прецедент
   `pg_advisory_xact_lock` (`learning.py:304`) сюда **не переносится**: он живёт в границах
   транзакции, а `repos/base.py` коммитит каждую запись — блокировка отпустится на первом
   коммите. Честный фикс требует переделки работы с транзакцией в этом эндпоинте →
   отдельная задача.
2. **TG_LMS и новый 400.** Боты корень не передают (ходят через start-or-get-attempt,
   который при неоднозначном узле сам вернёт пустой корень). На переиспользуемом узле
   при исчерпанном лимите ученик увидит 400 вместо понятного отказа → отдельная задача.
   SPW не затронут (`useRootCourseId`).
3. **Ученик без активной записи на курс отвечает и получает 200** — по данным независимого
   рецензента (`[P4] коды=[200,200,200,200]`). **Своими руками не проверял.** Это дыра
   авторизации старше tsk-269 (сюда бы вообще не доходить) → отдельная задача.
4. **Повторный неверный ответ после `PASSED` уронит прогресс** (`compute_task_state`
   смотрит последний `task_result`). Не регресс tsk-269 — так было и до правки. Стоит
   задачи, если оператор считает поведение нежелательным.
5. **Живой прод не проверялся** — правка не развёрнута. После развёртывания нужна живая
   проверка (ветвь А): ученик с исчерпанным лимитом на SPW.
