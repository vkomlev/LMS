# tsk-363 — VK-регистрация без почты роняла список пользователей (P0)

**Дата:** 2026-07-22
**Проект:** LMS (backend)
**Приоритет:** P0 — боевой сбой, бот методиста не открывал список студентов

## Что произошло

07:14:17 UTC новый пользователь зарегистрировался через VK OAuth на SPW.
VK userinfo вернул `email` **пустой строкой**, а не отсутствием ключа.
Проверка `if email:` в `fetch_vk_userinfo` пропускала нормализацию пустой
строки в `None`, и `''` записалась в `users.email`.

С 07:15:09 любой запрос `GET /api/v1/users/`, куда попадал этот пользователь,
падал с 500: схема ответа `UserRead.email: Optional[EmailStr]` не принимает
пустую строку. Симптом у оператора — бот методиста перестал загружать список
студентов. Бот исправен, ошибка целиком на стороне LMS.

Вторая, ещё не сработавшая мина: уникальный partial index
`users_email_unique_partial (email) WHERE email IS NOT NULL` держал занятым
слот `''` — следующая VK-регистрация без почты упала бы на конфликте
уникальности.

## Доказательства

- Лог бота: `role=student` → 500, `role=methodist` → 200 (в списке методистов
  пользователя нет).
- Лог LMS: `value is not a valid email address`, `loc=('response','items',9,'email')`,
  `input=''`.
- Аудит LMS: `user.registered.via_vk user_id=4505 email_provided=False`
  (`bool('')` = False — согласуется).
- БД (прод): ровно одна строка с `email = ''`, `id=4505`, `created_at=07:14:17`.

## Правки

| Файл | Суть |
|---|---|
| `app/services/auth/vk_oauth_service.py` | Источник течи: `email = (user.get("email") or "").strip().lower() or None` — пустая строка и пробелы приводятся к `None` |
| `app/schemas/users.py` | Подстраховка выдачи: валидатор `UserRead._blank_email_as_none` — историческая пустая строка в БД читается как `None`, а не роняет весь ответ |
| `tests/test_vk_blank_email_tsk363.py` | 8 тестов: разбор userinfo (пустая строка / пробелы / нет ключа / нормальная почта) + схема выдачи (пустота → None, валидная почта сохраняется, мусорная по-прежнему отвергается) |

Почему две правки, а не одна: первая закрывает конкретный источник, вторая
устраняет класс отказа «одна битая строка роняет весь список» — именно он
превратил единичную запись в P0.

## Правка данных на проде

Протокол `/db-check`, режим записи. Затронута 1 строка:

```sql
UPDATE users SET email = NULL WHERE id = 4505 AND email = '';
```

Выполнено в транзакции с проверкой «ровно 1 строка» и откатом при несовпадении.
После: `email IS NULL` = true, по таблице пустых строк 0 из 14 пользователей.
Данные пользователя не потеряны — почты у него не было изначально.

## Почему тесты не поймали раньше

`test_first_time_vk_no_email_creates_user_email_null` подавал `email=None`
напрямую в `get_or_create_user_by_vk`, минуя разбор ответа VK — а дефект был
именно в разборе. Новый тест бьёт по `fetch_vk_userinfo`.

## Остаточные риски

- Другие пути записи почты проверены и чисты: `tg_init_service` пишет `email=None`
  жёстко; `magic_link_service` получает адрес из уже провалидированной ссылки;
  `UserUpdate.email` отвергает пустую строку на входе (422).
- Валидатор добавлен только в `UserRead`. Если появится другая схема выдачи с
  `EmailStr` поверх той же таблицы — подстраховки там не будет.
