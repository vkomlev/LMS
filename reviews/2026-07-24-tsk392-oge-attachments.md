## Текстовый гейт tsk-369 здесь не работает — и это не признак чужого файла

Штатная сверка («дословный фрагмент условия» + «значимые числа источника») дала на партии
`match` всего 5 из 110: `weak` 53, `mismatch` 52. Причина не в файлах, а в том, что
**автор курса переписал условия своими словами**:

- «Скачай файл-таблицу задачи 10566 … Сколько учеников набрали более 600 баллов по
  информатике» вместо исходного условия с таблицей на пол-экрана;
- задание ОГЭ-14 разбито на два подвопроса (`_1`, `_2`) — у каждого свой кусок условия.

Ослаблять общий гейт нельзя: он защищает партии ЕГЭ. Поэтому принадлежность файла доказана
**другими признаками**, независимыми от формулировки (`scripts/tsk392_evidence.py`):

| Признак | Что означает | Сработал |
|---|---|---|
| `twin_sha` | скачанный по ID файл байт-в-байт равен файлу, который LMS уже отдаёт заданию с тем же ID источника | 50 (курс 1179) |
| `answer` | верный ответ в LMS совпал с ответом источника | 60 (курсы 1163/1164) |
| `tokens` | имена каталогов из условия («Проза», «DEMO-12», «Тургенев») есть в тексте источника | 60 (те же) |

Правило: `twin_sha` **либо** (`answer` И `tokens`) — одного признака мало. Подтверждено
110 из 110, отклонённых нет. Все признаки записаны поимённо в `reviews/tsk392-oge/evidence.json`.

Единственный случай неполного совпадения якорей — 6869 (доля 0.67): источник опускает имя
подкаталога («в подкаталоге каталога Проза» вместо «подкаталог „Достоевский“»). Прочитано
глазами: задача та же (Ф. М. Достоевский, «единственный сын» Варвары Петровны), ответ
источника «Николай» = ответу LMS.

## Прямая проверка: по файлу получается эталонный ответ задания

Все сверки выше косвенные. Для ОГЭ-14 сделана прямая: файл скачан **боевым**
`GET /api/v1/media/<sha>` и по нему пересчитан эталонный ответ задания
(`scripts/tsk392_verify_oge14.py`).

- автоматически, схема «округ | фамилия | предмет | балл»: **16 заданий, расхождений 0**;
- вручную, шесть других схем таблиц: 6969 (процент по физике > 60 → 30,8), 6970
  (наименьшая сумма баллов, округ Центральный → 68), 6972 (любят информатику → 12),
  6974 (любят алгебру → 25), 6976 (макс. вес в 10 классе → 80), 6980 (городов ближе
  4 часов → 103) — **все совпали точно**;
- остальные 28 — таблицы других схем («Дата|Температура|Осадки|Давление», «Город|Страна|
  Время в пути»), их условия каждый раз свои; автоматически не разбираются, скрипт честно
  сообщает «другая схема таблицы», а не выдаёт ложное расхождение.

Итого **22 из 50 заданий ОГЭ-14 проверены пересчётом ответа по файлу, расхождений 0.**
Именно этой проверки не хватило в tsk-390: там у авторских заданий сверялась лишь структура
файла, одинаковая у двух разных версий базы, — и чужой файл прошёл гейт.

## Грабли tsk-390: проверено, что не повторились

1. **Файл уже есть другим способом.** Критерий расширен на `rar` ещё до сбора кандидатов —
   25 заданий ОГЭ-13 отсеялись сами. Прямых внешних ссылок в охвате нет ни одной.
2. **Вторая ссылка вместо замены.** После записи: заданий с более чем одним файлом данных
   в охвате — **0**; ровно один файл у всех 110. Два задания курса 1178 имеют по три
   media-ссылки, но это `rar` + два `png` с образцом текста — они моей правкой не тронуты.
3. **Идемпотентность.** Пофайловая (по `sha_ext`), как после tsk-390.
4. **Расхождение текста и файла.** Не встретилось: и якоря условия, и пересчёт ответа сошлись.

## Три дефекта конвейера, найденных и починенных по пути

1. **Шаблонное вложение сайта.** В блоке задачи ОГЭ-14 лежит ссылка «Инструкция по
   обновлению сертификата windows 7.pdf» — она одинакова у всех задач и к ним не относится.
   Без фильтра приложилась бы к каждому заданию ОГЭ-14 второй ссылкой.
2. **Архив не проходил проверку типа файла.** `expected_ext` сопоставляла формулировку
   («текстовый редактор») с расширением, и `.rar` признавался несовпадением. Архив теперь
   согласуется с любой формулировкой: внутри лежит ровно то, что обещано условием.
3. **Кириллица в имени файла.** Имя из `Content-Disposition` Python отдаёт декодированным
   как latin-1: «Лермонтов.rar» превращалось в «Ð\x9bÐµÑ\x80…» и в таком виде уехало бы
   ученику в подпись ссылки. Плюс без имени подпись выходила «Файл к заданию: Файл к заданию».

Отдельно — кэш скачиваний по URL: 30 заданий ОГЭ-11 ссылаются на один `DEMO-12.rar` (33 МБ),
и прогон без кэша выкачал бы его 110 раз (3.6 ГБ).

## Регулярный чек научен видеть этот класс

`scripts/check_missing_attachments.py` ловил дефект **только по тексту условия** и потому
ОГЭ-11…14 не видел вовсе: переформулированные условия под `FILE_GATE_RE` не подходят.
Добавлена вторая ветка — **номер задания, где файл обязателен по типу** (ЕГЭ 3, 9, 10, 17,
18, 22, 24, 26, 27; ОГЭ 11, 12, 13, 14; номер берётся из `course_uid`), плюс `rar` в списке
расширений и allowlist `scripts/missing_attachments_allowlist.json` для заданий, которым
файл не нужен (19 ЕГЭ-24 из tsk-390 + 14 авторских ОГЭ) — с причиной у каждого ID.

Прогон на проде до работы: 110 находок (ровно тот же список, что собран вручную).
Прогон после: **0 находок по всей базе**.

## Верификация записи

- Каждый из 43 файлов проверен боевым `GET /api/v1/media/<sha_ext>` → 200 **до** правки БД
  (25 из них уже лежали в хранилище — те самые файлы близнецов).
- Запись — одной транзакцией, с бэкапом прежних условий на диск
  (`reviews/tsk392-oge/backup_stems.json`), построчной проверкой внутри транзакции и
  независимой построчной после COMMIT: **110 проверено, расхождений 0**.
- Пересчёт отдельным соединением через MCP `learn_prod_db`: без файла в охвате остались
  ровно 14 авторских заданий, у всех 110 изменённых — `has_attached_file=true` и ровно
  один путь в `attached_file_paths`.
- Файлы больше 20 МБ (7 штук, до 45 МБ) потребовали поднять `CAS_MAX_FILE_BYTES` на время
  загрузки — иначе помощник CAS молча их пропускает.
