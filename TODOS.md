### Перевод 5 TG-ботов с long polling на webhook
- **Что:** TG_LMS сейчас — 5 отдельных процессов (`main_bots_admin/methodist/teacher/student/marketer`)
  на long polling к Telegram. Перевод на webhook сделает их обычным HTTP-сервисом.

- **Контекст:** решение принято 2026-07-04 при ревью деплой-стратегии на Timeweb Cloud
  (`docs/briefs/deploy-timeweb-cloud.md`, tsk-005) — на первом этапе оставили long polling +

- **Зависит от / блокируется:** ничем не блокируется; целесообразно делать, когда VDS с ботами
  станет узким местом по ресурсам или потребуется горизонтальное масштабирование.

### Переключить LMS на VPC-доступ к DBaaS и балансировщикам, когда Timeweb добавит поддержку для Apps
- **Что:** сейчас App Platform не поддерживает подключение к приватной сети (VPC) — только
  публичный IP. Как только Timeweb добавит поддержку VPC для Apps — переключить (1)
  `DATABASE_URL`/`REDIS_URL` LMS с публичного адреса DBaaS на приватный, и (2) связь
  балансировщик↔приложение (LMS/SPW) с публичного IP приложения на приватный.
- **Почему:** приватная сеть убирает публичную поверхность атаки и не требует платных
  публичных IP (~150 руб/мес за адрес, уточнить точную цену в панели) — ни у БД, ни у самих
  приложений LMS/SPW.
- **Плюсы/минусы:** плюс — безопаснее и дешевле (меньше платных публичных IP); минус —
  требует смены конфигурации и редеплоя, зависит от таймлайна самого Timeweb (не под нашим
  контролем).
- **Контекст:** решение принято 2026-07-04 при ревью деплой-стратегии на Timeweb Cloud
  (tsk-005, `docs/briefs/deploy-timeweb-cloud.md`) — на момент ревью документация Timeweb
  явно указывала "приватные сети для Apps — план на будущее". Уточнено 2026-07-04 при вопросе
  про отключение платного IPv4 у App Platform-приложений: документация `/docs/public-ip`
  явно перечисляет поддерживаемые сервисы («облачными серверами, базами данных,
  балансировщиками нагрузки, а также приватными сетями») — App Platform в списке нет, то есть
  публичный IP приложения сейчас нельзя ни отвязать, ни заменить приватным без риска разрыва
  связи балансировщик↔приложение.
- **Зависит от / блокируется:** зависит от Timeweb (внешний таймлайн, не отслеживаемый нами
  напрямую) — периодически проверять `https://timeweb.cloud/docs/apps` на предмет апдейта.

### Восстановить корректную начальную Alembic-миграцию (нет создания базовых таблиц)
- **Что:** миграция `add_courses_triggers` (самая первая в цепочке, `down_revision=None`)
  создаёт триггеры на таблицах `user_courses`/`courses`/`course_dependencies`, которые нигде
  в истории миграций явно не создаются через `create_table`. На чистой БД `alembic upgrade
  head` падает с самого начала (`UndefinedTableError`). Похоже, исходная схема когда-то была
  создана напрямую (`Base.metadata.create_all()` в раннем деве) до того, как проект начал
  трекать миграции Alembic.
- **Почему:** любой будущий полностью чистый деплой (новая БД с нуля) упрётся в ту же ошибку.
  Сейчас обойдено вручную: снят `pg_dump --schema-only` с рабочей локальной dev-БД `Learn` и
  применён напрямую к прод-БД, затем `alembic stamp head` (без реального прогона миграций).
  Это работает, но хрупко — обходной путь, не постоянное решение.
- **Плюсы/минусы:** правильный фикс (сгенерировать настоящую baseline-миграцию через
  `alembic revision --autogenerate` от пустой БД) — трудоёмкий, требует сверки со всеми ~30
  миграциями; текущий обходной путь (dump+stamp) быстрый, но не самодокументируемый — новый
  разработчик не сможет поднять чистую БД одним `alembic upgrade head`.
- **Контекст:** обнаружено 2026-07-05 при первом деплое LMS на VPS (tsk-005) — App Platform
  никогда не доходил до реального прогона миграций из-за более ранних проблем с доменами/edge.
- **Зависит от / блокируется:** ничем не блокируется, но стоит сделать до следующего раза,
  когда понадобится поднять ещё одну чистую копию БД (staging, тесты, ещё один регион).

### Расширить CSP (Content-Security-Policy) на весь SPW, не только `/embed/:path*`
- **Что:** сейчас CSP-заголовок в `next.config.ts` (SPW) применяется только к пути
  `/embed/:path*` (для iframe-встраивания, см. Y-5 §6.5.5). Остальные страницы приложения
  (включая `/me`, `/auth`, дашборды) отдаются без CSP вообще.
- **Почему:** независимый ревью (Кодекс, tsk-159) отметил отсутствие site-wide CSP как
  «Significant» (не критичный, но заметный) риск — при XSS-уязвимости где-либо в приложении
  нет второго рубежа защиты в виде ограничения источников скриптов/стилей.
- **Плюсы/минусы:** плюс — дополнительный слой защиты от XSS на всём приложении; минус —
  формирование корректной site-wide CSP-политики для Next.js приложения с внешними
  интеграциями (VK ID виджет, Google Sheets и т.п.) требует аккуратного подбора allow-list —
  наспех сделанная политика рискует сломать легитимные внешние скрипты/стили без заметного
  прироста в защите.
- **Контекст:** найдено 2026-07-06 при независимом ревью Кодекса после миграции на prod VPS
  (tsk-159). Решение — не спешить с точечным патчем, а сделать отдельной задачей с явным
  проектированием allow-list.
- **Зависит от / блокируется:** ничем не блокируется; делать отдельной задачей, когда будет
  время на аккуратный аудит всех внешних источников скриптов/стилей/фреймов, которые
  использует SPW (VK ID, любые сторонние виджеты).

### `deploy.sh`/`rollback.sh` могут потребовать пароль sudo при неинтерактивном запуске (без pty)
- **Что:** на `lms-spw-vds` в sudoers включён `Defaults use_pty` — sudo пытается выделить
  настоящий псевдотерминал для команды, даже когда для неё есть NOPASSWD-правило. При запуске
  `sudo -u app bash deploy.sh` из обычной интерактивной SSH-сессии (`ssh user@host`, затем
  команда с руки) это не проблема — терминал уже есть. Но при запуске неинтерактивно
  (`ssh host "команда"` без `-t`, как делает автоматизация/CI/будущий cron) `sudo systemctl
  restart lms` иногда падает с `sudo: a password is required`, даже с настроенным NOPASSWD.
- **Почему:** обнаружено 2026-07-06 при живом тестировании `rollback.sh`/`deploy.sh` на проде —
  прогон через `ssh -tt` (форсированный псевдотерминал) стабильно проходит, прогон через
  обычный `ssh host "команда"` — падает непредсказуемо на этом шаге.
- **Плюсы/минусы:** `use_pty` — общесистемная настройка sudoers, менять её ради одного скрипта
  не стоит (затрагивает поведение sudo для всех пользователей на сервере). Точечный фикс —
  добавить `Defaults:app !requiretty` / `!use_pty` только для пользователя `app`, либо
  запускать скрипты через `ssh -tt` в любой будущей автоматизации.
- **Контекст:** `docs/qa/2026-07-06-deploy-timeweb-smoke-report.md` — там же зафиксирован факт
  теста; `/etc/sudoers.d/app-deploy` на `lms-spw-vds` — текущее NOPASSWD-правило.
- **Зависит от / блокируется:** не блокирует ручной операторский запуск (там всегда есть pty);
  актуально только если появится неинтерактивная автоматизация деплоя (CI/cron) — тогда сначала
  решить эту проблему, иначе автодеплой будет случайно зависать на рестарте.
