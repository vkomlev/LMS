### Шаг 2: Проектирование учебного плана модуля
1. Декомпозировать цель на 3–7 учебных результатов. Каждый — наблюдаемый глагол + уровень Блума (reference, Часть 1).
2. Выстроить прогрессию: монотонно по Блуму, scaffolding — каждый шаг опирается только на освоенные предпосылки (reference, Часть 3).
3. Карта занятий под формат: что на встрече недели, что на самостоятельной.
4. Точки активного извлечения (retrieval) между шагами; раннее понятие возвращается в поздних заданиях (spaced practice); для спиральных курсов (класс→класс, многоглавных) — retrieval-мостик «Вспомни: …» на прошлый класс/тему (difficulty-and-design Часть 3 п.3). Каждый урок проектировать «конвертом»: цель-действие на входе + итог на выходе (Часть 3 п.7).
5. **Показать черновик структуры оператору, получить правки — интерактивный гейт, не молча.**

### Шаг 3: Генерация заданий (по `assignment-rules.md`)
Задания строятся строго по `assignment-rules.md` и готовятся LMS-импортируемыми.

1. **Количество по узлам** (узлы графа `Courses`, см. lms-wp-export.md) — классифицировать ПО СОДЕРЖАНИЮ, не по форме графа:
   - узел-лист (узкая подтема, один приём/факт) → ровно 3 задания по возрастанию сложности;
   - **содержательная тема (числа/строки/условия/циклы — несколько приёмов) = подкурс 1-го уровня**, даже если в графе пока без детей → 10–15 заданий, микс **базовая 70% / повышенная-средняя 25% / сложная 5%** + **2–3 проектных варианта**. 3 задания на содержательную тему = недодача, брак.
   - **Гейт глубины практики (assignment-rules §3.4):** каждый приём/метод из материала → ≥1 задание на ПРИМЕНЕНИЕ учеником; распространённые приёмы темы не пропускать (для строк — срезы + `count`/`find`/`replace`); объём РАЗНООБРАЗНЫХ заданий (§8 п.8), не 3 однотипных.
2. **Типы** — только enum LMS `SC|MC|SA|SA_COM|TA`. Приоритет быстрых авто-проверок `SC`/`MC`/`SA`; `SA_COM` — **только задачи с кодом/решением** (двухфазно: авто-чек + проверка качества, assignment-rules § 7, НЕ «сочинение»); `TA` — только крупные проектные. Обилие текстовых ответов массой запрещено.
2a. **Вовлечённость (assignment-rules § 8, ОБЯЗАТЕЛЬНО):** каждое задание — миссия с видимым результатом и проверкой через действие (выбери шаг / допиши фрагмент / исправь чужую ошибку / примени в новой формулировке), нарастающая сложность, жизненный контекст, проектные «зажигают». Школьное эссе/пересказ теории = брак.
3. **Сложность** — маппинг бакетов на `difficulties`: базовая→`EASY` (чистый recall→`THEORY`), средняя/повышенная→`NORMAL`, сложная→`HARD`, проектные→`PROJECT`.
4. Каждое задание содержит: `external_uid` (стабильный, схема `{COURSE_UID}-{NN}-{TYPE}`), `course_uid`, `type`, `stem`, `options` (SC/MC), `max_score`, `difficulty`, `solution_rules` (наблюдаемые: `correct_options`/`accepted_answers`/`rubric`). Цепочка AI-наставника + механизм защиты от списывания (assignment-rules § 5) — **только если методика включена оператором** (по умолчанию выкл., отдельный трек tsk-059).
5. Содержание задания — из соответствующего `docs/v2/P*.md`/`F-*.md` (Трек 1: бот, интеграции, парсинг, CRM, дашборды), не «по памяти».

### Шаг 4: Двойная проверка (ОБЯЗАТЕЛЬНЫЙ гейт перед верификацией)
Прогнать черновик плана/заданий двумя взглядами по `coverage-and-review.md`, Часть 2. Это не опция. Любой ответ «нет / не уверен» → доработать и пройти заново.

1. **Глазами ученика** (целевая ЦА × стадия): провести **механическую трассировку предпосылок** (coverage-and-review, Часть 2): для каждого шага выписать требуемые понятия/термины/инструменты и убедиться, что каждый разобран в более раннем шаге (не позже, не параллельно, не «по мере встречи»). Затем: нет ли скачка (перегруз) или тривиальности (недогруз)? хватает ли времени реально? понятно ли, что предъявлять и куда идти при затыке?
2. **Глазами сеньора по теме**: прогрессия без логических дыр? все ключевые вопросы темы закрыты? дан ли полный набор инструментов/материалов (ничего обязательного не выпало)? нет ли поверхностности — каждый результат трассируется к файлу docs/v2?
3. **Анти-поверхностность**: нет формулировок-заглушек («разобрать основы» без действия и источника); глубина соответствует уровню детализации.
4. **Линза освоения (mastery, финальная — по ЗАДАНИЯМ темы):** «Сможет ли ученик САМ сделать эту тему после ЭТИХ заданий? Каждый ли изученный приём он применил руками?» (coverage-and-review, Часть 2, линза освоения). Ответ «нет» = задания дают узнавание, а не умение → доработать задания, не выдавать.

Вывод фиксируется в Контракте результата: что проверено, что переделано после прохода, остаточные риски.

### Шаг 5: Верификация
- [ ] Каждый учебный результат = наблюдаемый глагол + уровень Блума
- [ ] Прогрессия монотонна, нет скачка через непокрытую предпосылку
- [ ] Каждое задание имеет наблюдаемые критерии готовности
- [ ] Сложность каждого задания оценена по 3 параметрам
- [ ] Сложность не превышает потолок заявленной стадии ЦА (иначе — добавить scaffolding-шаг или пометить)
- [ ] *(если методика AI-наставник включена оператором)* цепочка встроена; ни одно задание не решается одним промптом без понимания; у каждого задания есть механизм защиты от списывания. По умолчанию методика выкл. — пункт пропускается
- [ ] Сумма времени заданий ≤ бюджет самостоятельной работы формата
- [ ] **Полнота против инвентаря:** каждый файл docs/v2 классифицирован include/exclude с причиной; обязательный-по-стадии минимум (coverage-and-review, Часть 1) не выпал; есть раздел «Карта покрытия источника»
- [ ] **Двойная проверка (Шаг 4) пройдена**, остаточные риски зафиксированы
- [ ] **Задания по `assignment-rules.md`:** типы только из enum (`TA` лишь у проектных); счёт по узлам (лист=3, подкурс 1-го ур.=10–15 + 2–3 проектных); микс 70/25/5 посчитан фактически; difficulty по маппингу; у каждого задания `external_uid`/`course_uid`/наблюдаемые `solution_rules`
- [ ] **Глубина практики (гейт §3.4 + mastery-линза):** содержательная тема классифицирована как подкурс (10–15 разнообразных), не лист на 3; каждый приём/метод материала применён ≥1 заданием; распространённые приёмы не пропущены; микс типов практики (§8 п.8); mastery-линза Шага 4 пройдена
- [ ] **Профиль прогрессии ЦА (assignment-rules §3.5):** объём/сложность заданий растут от старта к финалу в коридоре профиля; для не-детской ЦА к концу — многострочные программы и ≥ требуемого числа проектных; «детский» одношаговый формат весь курс для не-детской ЦА = брак; многошаговые/проектные авто — `SA_COM`, не `TA`
- [ ] **Две проекции присутствуют:** LMS-импорт (JSON bulk-API + Google Sheets раскладка) и WP-структура «Навигатор»; `course_uid`/`external_uid` общие и стабильные
- [ ] **Конверт урока у КАЖДОГО урока:** цель-действие на входе + итог на выходе (обе проекции, не `lms_skip`); спиральный курс — мостик «Вспомни: …»; первый программирующий урок — подсказка запуска. Абстрактные темы помечены под ASCII-схему внутри материала (coverage-and-review анти-поверхностность)

## Формат вопроса
Интерактивные развилки (Шаг 1, выбор прогрессии, спорная сложность) — через короткий вопрос оператору:
- **Контекст** — что решаем и почему сейчас
- **Суть** — конкретная развилка
- **Рекомендация** — A) … (рекомендую) / B) … / C) …, с обоснованием

## Правила качества
- **Интерактивность:** цель, ЦА, прогрессия — через оператора, не додумывать. Структуру модуля согласовывать до генерации заданий
- **Действие, не знание:** учебный результат и критерий — наблюдаемый глагол, не «знать»/«понимать тему»
- **Прогрессия без скачков:** каждый шаг опирается только на освоенные предпосылки
- **Матчинг уровня:** сложность не выше потолка стадии ЦА без явной пометки и scaffolding
- **AI-наставник опционален** (по умолчанию ВЫКЛ., отдельный трек tsk-059, не блокер). Включается только явным запросом оператора; тогда задание, списываемое у AI одним промптом = брак. По умолчанию не навязывать, `methodology-source: не применяется (tsk-059, отдельный трек)`
- **Методика:** нет финального tsk-059 → baseline + явная пометка, не выдавать baseline как финал
- **Источник содержания:** docs/v2, не «по памяти». Каждый учебный результат трассируется к конкретному файлу docs/v2
- **Бюджет нагрузки:** сумма времени заданий ≤ бюджет формата, считать в часах
- **веб-поиск:** по запросу оператора или новому типу модуля, не каждый прогон; выводы кэшировать в `references/difficulty-and-design.md` (Часть 4)
- **Полнота против инвентаря:** для трек/куррикулум-запроса — классифицировать каждый файл docs/v2 include/exclude с причиной; обязательный-по-стадии минимум не срезать. Для новичка фундамент, базовые понятия (глоссарий), бесплатные альтернативы — не сокращать ради компактности (срез = блокер прохождения)
- **Двойная проверка обязательна:** план/задания не выдаются без прохода «глазами ученика» и «глазами сеньора» (Шаг 4). Поверхностность (заглушки без действия и источника) = брак
- **Понятие раньше инструмента:** служебный/инфраструктурный шаг (счёт токенов, калькулятор, запрос к API, деплой) не идёт раньше шага, где явно разобраны используемые им понятия. Разбор «параллельно» или «по мере встречи» не считается «раньше» — это дефект порядка
- **Типы заданий — строго enum LMS** `SC|MC|SA|SA_COM|TA`. Кода `COM` нет (это `SA_COM`). `SA_COM` = двухфазная проверка задач с кодом (авто-чек ответа + ручная/ИИ оценка качества), НЕ «короткий ответ + сочинение» (assignment-rules § 7). `TA` — только крупные проектные
- **Вовлечённость > школьность:** задание — жизненная миссия с видимым результатом и проверкой через действие (assignment-rules § 8). Эссе/пересказ теории массой, скучные абстрактные задачи, неинтересные проектные = брак, переформулировать
- **Глубина практики по умолчанию (mastery, не «для галочки»):** самоцель — ученик НАУЧИЛСЯ ДЕЛАТЬ тему сам. Содержательная тема = подкурс с 10–15 РАЗНООБРАЗНЫМИ заданиями; каждый приём/метод материала применён ≥1 заданием (assignment-rules §3.4); 3 однотипных задания на содержательную тему = брак
- **Сложность — фиксированный маппинг** на `difficulties` (EASY/THEORY/NORMAL/HARD/PROJECT). Микс считать фактически, не «на глаз»
- **Профиль прогрессии под ЦА (потолок ≠ цель):** объём/многошаговость заданий и доля проектных растут по ходу курса в коридоре профиля ЦА (difficulty-and-design «Профиль прогрессии»). Низкий вход допустим, но для не-детской ЦА финал ОБЯЗАН выйти на многострочные программы и проектные; «детский» одношаговый формат весь курс для предвузовской/взрослой ЦА = брак. Многошаговые/проектные авто-задания — `SA_COM` (конвертер не разворачивает `TA`)
- **Идемпотентность:** `course_uid`/`external_uid` стабильны и едины для LMS- и WP-проекций. Повторный импорт = upsert, не дубль
- **Две проекции обязательны:** план не выдаётся без LMS-проекции (JSON + Sheets) и WP-проекции «Навигатор». `docs/v2` — источник, не переписывать (план — слой нормализации)
- **Структура подкурсами, не плоско (большие/детские курсы):** курс (класс) → глава → тема(лист); лист-подтема = отдельный подкурс со СВОИМИ материалами и заданиями рядом (изучил → сразу применил). Плоский курс ломает LMS-логику «сначала все материалы, потом все задания» — ранняя теория забывается. Граф через `subcourses[]` документа-корня (`lms-wp-export.md` § 1.4)
- **Визуалы по умолчанию (детские курсы — насыщенно):** не одна обложка, а образ под каждое ключевое понятие ВНУТРИ материала (ориентир — презентации Босовой); детский урок = 2–5 визуалов. Идут И в WP, И в материал LMS. Точное (схемы, формулы, таблицы-связи) — ASCII (генеративному ИИ точные данные не доверять); **UI-состояния** (терминал, установка, онбординг, экран приложения, файлы) — детерминированные скриншоты (рендер Claude, `image`), для install/онбординг-уроков обязательны (Codex сюда не годится); иллюстрации/декор — делегировать в Codex через md-промпт (Claude не генерит картинки), у каждого образа место в материале + `alt` + идемпотентное имя. **При проектировании плана — помечать уроки, которым нужны скриншоты интерфейса** (установка, онбординг, работа в приложении) **и абстрактные темы, требующие ASCII-схему ВНУТРИ материала** (логика/таблицы истинности, системы счисления, структуры данных, алгоритмы, функции): абстрактная тема с одной обложкой = визуальный голод, брак (повтор-дефект Информатика 5-11). Для CreateCourses — `docs/ai/visuals-policy.md`, воркфлоу — `/course-screenshots`
- **Конверт урока (ОБЯЗАТЕЛЬНО, обе проекции):** каждый урок открывается целью-действием («после урока сможешь …», глагол Блума — не «узнаешь про») сразу после хука и закрывается итогом («Итог: …», 2–3 пункта); для спиральных — мостик «Вспомни: …»; первый программирующий урок — подсказка запуска (песочница/IDLE). Цель/итог/мостик идут И в WP, И в LMS-материал (не `lms_skip`). Урок без цели или итога = незавершён (difficulty-and-design Часть 3 п.7, рендер — lms-wp-export § L3)
- **Полнота результатов:** непокрытый учебный результат = незавершённый план
