# Правила создания заданий (LMS-адаптированные)

Источник истины для skill `methodist` по заданиям. Подключается в Шаге 0,
применяется в Шаге 3. Вынесен отдельным файлом — переиспользуется при разных
режимах генерации. Все коды/сложности сверены с `d:\Work\LMS` (см.
`lms-wp-export.md`). НИЧЕГО не выдумывать сверх enum LMS.

## 1. Типы заданий (строго enum LMS)

Допустимы **только**: `SC`, `MC`, `SA`, `SA_COM`, `TA`. Кода `COM` не
существует — это `SA_COM`.

| Код | Что | Когда применять | Проверка |
|---|---|---|---|
| `SC` | Single Choice — 1 верный вариант | факт/выбор/различение понятий | авто |
| `MC` | Multiple Choice — неск. верных | «выбери всё, что верно», классификация | авто (partial) |
| `SA` | Short Answer — короткий ответ | термин, значение, имя функции, результат запуска | авто (accepted_answers) |
| `SA_COM` | Авто-ответ (SA) + проверка качества (COM) | **задачи с кодом/решением**: авто-чек правильности + ручная/ИИ оценка качества по рубрике | авто + ручная/ИИ |
| `TA` | Text Answer — развёрнутый, рубрика | **только крупные проектные / защита** | ручная по `rubric` |

**Правильная семантика `SA_COM`** (см. § 7, критично): это НЕ «короткий ответ +
сочинение почему». Это **двухфазная проверка задач с кодом/решением**: сначала
быстрый авто-чек верного результата (SA: тест прошёл / ответ совпал), затем
препод или ИИ оценивает **качество решения/кода** (COM) по рубрике.

**Приоритет типов:** быстрые авто-проверки `SC`/`MC`/`SA` — основная масса
(мгновенная обратная связь, ноль трения). `SA_COM` — для практических задач
**с кодом/решением** (есть и авто-результат, и артефакт для качественной
оценки). `TA` — только крупные проектные/защита.

**Анти-школьность (жёстко):** «напиши словами, почему так» как массовый формат
ЗАПРЕЩЁН — обилие текстовых «сочинений» убивает вовлечённость, ученики
сливаются. Понимание проверять **через действие** (§ 8), не через эссе.

## 2. Сложность → справочник LMS `difficulties`

Маппинг бакетов методиста на коды LMS (фиксированный):

| Бакет (методист) | LMS `difficulty` (uid/code) | weight |
|---|---|---|
| Базовая (recall, чистое воспроизведение) | `theory` / `THEORY` | 1 |
| Базовая (применение по образцу) | `easy` / `EASY` | 2 |
| Повышенная / средняя | `normal` / `NORMAL` | 3 |
| Сложная | `hard` / `HARD` | 4 |
| Проектная | `project` / `PROJECT` | 5 |

Соответствие Блуму: THEORY/EASY ≈ L1–L3, NORMAL ≈ L4, HARD ≈ L5, PROJECT ≈ L6.

## 3. Количество и микс заданий по узлам LMS

LMS не имеет отдельной сущности «подкурс/лист» — это узлы графа `Courses`
(см. lms-wp-export.md). Правила по позиции узла:

**Сначала классифицировать узел ПО СОДЕРЖАНИЮ, не по форме графа.** Тема, на
которой ученик ОСВАИВАЕТ навык (числа, строки, условия, циклы — несколько
приёмов/методов внутри), — это **подкурс 1-го уровня (§3.2)**, даже если в графе
она пока без детей. «Лист» (§3.1, ровно 3) — только узкая подтема с одним
приёмом/фактом, не несущая самостоятельного навыка. Сомнение «лист или подкурс?»
→ решать в пользу подкурса (§3.2). **3 задания на содержательную тему = недодача,
брак** (ученик не научится делать тему с 3 однотипных заданий).

### 3.1. Узел-лист (узкая подтема, один приём/факт)
**Ровно 3 задания** в порядке возрастания сложности:
1. базовое (EASY, при чистом recall — THEORY)
2. повышенное/среднее (NORMAL)
3. сложное (HARD)
Типы: преимущественно `SA_COM` / `SC` / `MC`.

### 3.1a. Урок-тема школьного курса (решение оператора, курс «Информатика 5-11»)
Школьный урок — это **содержательная тема**, а не узкая подтема: ученик осваивает
понятия и должен уметь их применять. Норматив: **10 заданий на урок** — не просто числа,
а роли: **6 простых** на узнавание и прямое применение, **3 средних** на перенос и
комбинирование, **1 проектное** (или мини-практика; `PROJECT`, авто-проектное — `SA_COM`).
Три задания на урок — недодача (узнавание вместо умения), §3.1 сюда не применять.
**Меньше 10 допустимо только для очень короткого урока и только с явным обоснованием.**
Считать по каждому урок-узлу, не в среднем по главе. Проверка — `expert-course-review`
рубрика К4.

**Перенос: объекты вне материалов — норма, а не брак.** В заданиях НУЖНЫ варианты и
объекты, которых в материале не было: ученик обобщает признак понятия и применяет его
к новому объекту (тачскрин, новое устройство, незнакомый файл). Условие одно: **признак,
по которому объект относят к понятию, в материалах дан**. Задание только из объектов
материала проверяет память, не умение. Ревьюеру это не флагать (см. К5).

**Минимум SA/SA_COM среди 6 простых (tsk-176, повтор-дефект).** Из 6 простых заданий
минимум **2 — типа `SA`/`SA_COM`** (впиши ответ), не только `SC`/`MC` (выбор из
вариантов). Причина: банк 10 заданий/урок, целиком собранный из `SC`/`MC`, формально
проходит норматив 6+3+1, но по факту — 9 заданий узнавания-с-подсказкой-варианта и один
изолированный `TA`-проект; ученик ни разу не воспроизводит термин самостоятельно до
проектного уровня. Найдено дубль-треком (Claude+Codex) на курсе «Информатика 5-11»
класс 5 после волны 5C1 — 7 из 16 уроков имели 0 заданий SA/SA_COM. Проверка —
`expert-course-review` К5.

### 3.2. Подкурс первого уровня (прямой ребёнок курса; напр. «Фаза 0», «Фаза 1»)
**10–15 заданий**, микс по сложности:
- **базовая 70%** → EASY (чистый recall → THEORY)
- **повышенная/средняя 25%** → NORMAL
- **сложная 5%** → HARD
Плюс **2–3 варианта проектного задания** (difficulty `PROJECT`; авто-проектное —
тип `SA_COM`, не `TA`, см. §3.5).

Округление: при 10 заданиях ≈ 7 базовых / 2–3 средних / 0–1 сложное (минимум
1 сложное, если узел не вводный); при 15 ≈ 10–11 / 4 / 1.

**Микс зависит от профиля прогрессии ЦА** (difficulty-and-design, «Профиль
прогрессии»). 70/25/5 — базовый ориентир для детского/подросткового профиля. Для
**предвузовского / взрослого** профиля к финальным узлам курса доля NORMAL/HARD и
проектных РАСТЁТ за счёт базовых (низкий вход → крутой подъём): в финальных подкурсах
допустимо ~50/35/15 + ≥2 проектных. Брак — держать 70/25/5 одношаговых заданий весь
курс там, где профиль требует выхода на многострочные программы.

### 3.3. Распределение по листам внутри подкурса
Сначала по 3 задания на каждый лист (3.1), затем добор до 10–15 на уровне
подкурса так, чтобы суммарный микс соблюдал 70/25/5 + 2–3 проектных.

### 3.4. Гейт глубины практики и покрытия приёмов (ОБЯЗАТЕЛЬНО)
Цель скилла — чтобы ученик **НАУЧИЛСЯ ДЕЛАТЬ тему сам**, а не чтобы тема была
формально «покрыта материалом». Перед сдачей заданий темы пройти три проверки:

1. **Покрытие приёмов.** Выписать КАЖДЫЙ приём / понятие / метод из материала темы
   (для строк это, в т.ч., общие срезы + распространённые `count`/`find`/`replace`/
   `upper`/`lower`/`split`). Каждый приём → **≥1 задание, где ученик его ПРИМЕНЯЕТ
   руками** (не «прочитал в материале»). Приём изложен в материале, но нигде не
   применён = дыра, добавить задание. (Повтор-дефект: про срезы строк рассказали —
   ученик срезы ни разу не использовал.)
2. **Не недодавать распространённые приёмы темы.** Если в материале темы пропущен
   общеупотребимый приём (для строк — `count`/`find`/`replace`), сперва **дополнить
   материал**, затем дать на него задание. Тема без своих ходовых приёмов = неполная.
3. **Достаточный объём РАЗНООБРАЗНЫХ заданий** (ориентир §3.2: 10–15 + 2–3
   проектных), микс типов практики по §8 п.8 — не 3 однотипных «предскажи вывод».

Любой «нет» → доработать, не выдавать.

### 3.5. Рост объёма заданий по профилю прогрессии (не-детские ЦА)
Объём кода и многошаговость заданий обязаны РАСТИ по ходу курса в коридоре профиля
ЦА (difficulty-and-design, таблица «Профиль прогрессии»): напр. предвузовский —
4–5 строк в начале → 8–10 в середине → 10–15 в конце, плюс ≥2 проектных. Гейт:
выписать объём (строк кода / шагов) типичного задания по узлам в порядке курса —
если в финале он не вырос относительно старта, профиль нарушен, доработать.

**Многошаговые и проектные авто-задания с детерминируемым результатом — типом
`SA_COM`.** Конвертер ContentBackbone `lms_publish/blocks_to_lms.py` разворачивает
`SC | SA | SA_COM | TA` — начиная с tsk-176 `TA` тоже разворачивается (`_ta_rules`,
уходит в `manual_review_required: true` с рубрикой). Правка (2026-07-10): более
ранняя версия этого файла утверждала «`TA` не разворачивается» — это устарело,
не проверять и не воспроизводить это утверждение. Тем не менее для «напиши
программу целиком» и авто-проектного с ОДНИМ детерминируемым эталоном по-прежнему
предпочтителен `SA_COM` с продиктованной структурой ответа (см. §7): SA-фаза ловит
ключевые идентификаторы/числа/имена методов и их порядок (нормализация
`trim/lower/strip_punctuation/collapse_spaces` стирает символы `= " ( ) { }`, но
СОХРАНЯЕТ слова, идентификаторы и числа), COM-фаза оценивает качество кода — это
даёт мгновенную авто-обратную связь на SA-фазе, которой `TA` не даёт. `TA` —
осознанный выбор для защиты/проекта без детерминируемого эталона (см. §7 про
инвариант поля `answer`), не запасной вариант «раз SA_COM не подошёл». Открытое
«делание» без единого эталона (свой проект/игра) — ручная проверка
(`manual_review_required`) или WP-миссия `lms_skip`, не авто. Так проектная глубина
достигается без ручной проверки там, где эталон есть и её можно избежать.

### 3.6. Интерливинг-узел (опционально, точечно)

Отдельно от spiral-мостика (difficulty-and-design.md, Часть 3 п.3 — старое понятие
возвращается ОДНИМ заданием в новом узле). Interleaving — намеренное чередование
заданий РАЗНЫХ, но похожих тем в одном наборе практики (не блокировать по одной
теме подряд) — доказанно улучшает различение близких понятий, которые ученики
типично путают. Разные приёмы, разное назначение: spiral-мостик = напоминание,
интерливинг-узел = тренировка различения.

**Когда применять (узкий критерий, не по умолчанию):** только если 2-3 узла-листа
подряд внутри одного подкурса (§3.1) держат **легко путаемые** понятия одной
предметной области — близкие по имени/сигнатуре методы (`count` vs `find`,
`append` vs `extend`), похожие структуры (`if/elif` vs вложенный `if`), смежные
термины школьного курса (растр/вектор, файл/путь). Если темы подкурса тематически
далеки друг от друга (не путаются) — интерливинг-узел не нужен, спирального
мостика достаточно.

**Механизм — без нового типа узла, без миграции LMS-схемы.** Граф курса и
публикатор ContentBackbone уже поддерживают узел с несколькими родителями
(ADR-0042 Stage 3 `shared`/`attach_to`/`requires`, реализовано и используется в
проде для переиспользуемых узлов). Тот же механизм даёт «сводную практику»:

1. Авторить новый узел-лист `course_uid` (доп. суффикс, напр. `...-mix`) сразу
   после последнего из 2-3 путаемых листьев в подкурсе.
2. `shared: true`, `attach_to: [uid_листа_1, uid_листа_2, (uid_листа_3)]` —
   привязка к нескольким родителям одновременно.
3. `requires: [те же uid]` — гейт: узел открывается только когда ВСЕ
   источники-листья пройдены (обычный, уже готовый механизм requires §4).
4. Свои 4-6 заданий узла: **вперемешку** ссылаются на понятия из РАЗНЫХ
   источников-листьев не блоками («2 задания темы A, потом 2 темы B»), а
   чередованием (A, B, A, C, B — порядок задаёт разброс, не группировку).
   Каждое задание — про ОДНО понятие, но набор в целом заставляет ученика
   переключаться между темами, а не «доигрывать» одну тему подряд.
5. Типы/сложность — как у обычного листа (§3.1): SC/MC/SA, преимущественно
   NORMAL (цель — различение, не новизна и не пик сложности).

**Не для каждого подкурса.** Точечный приём для мест с реальной путаницей —
не универсальное правило на весь курс. Пилот — на одном модуле, сверка эффекта с
оператором перед распространением на другие курсы (см. `errors.md` CreateCourses,
backlog «Interleaving», tsk-178).

## 4. Структура задания (поля под LMS)

Каждое задание формулируется готовым к импорту (см. lms-wp-export.md §
task_content / solution_rules). Обязательно:
- `external_uid` — стабильный (идемпотентный upsert), схема именования
  `{COURSE_UID}-{NN}-{TYPE}` (напр. `PY-EGE-T1-03-SC`)
- `type` (enum §1), `stem` (формулировка), `options` (для SC/MC), `max_score`
- `difficulty` (uid/code §2)
- `solution_rules`: `correct_options` (SC/MC) / `accepted_answers` (SA/SA_COM) /
  `rubric` (TA) — наблюдаемые критерии, не «знает тему»
- Привязка к узлу курса через `course_uid`
- *(опционально)* цепочка AI-наставника + механизм защиты (ai-mentor-baseline) — только если методика включена оператором

Для `SA`/`SA_COM` сразу указывать не только канонический ответ, но и очевидные
естественные формы, если публикатор/проверка их поддерживают: падежи русского
слова, символ вместо названия, регистр, распространённый синоним. Если список
accepted-ответов технически недоступен, стем обязан диктовать ровно ту форму,
которая лежит в эталоне.

## 5. Защита от списывания по типам (ОПЦИОНАЛЬНО)

> Применяется **только если методика AI-наставник включена оператором**
> (по умолчанию ВЫКЛ., отдельный трек tsk-059, не блокер). По умолчанию раздел
> не применяется; задания валидны и без механизмов защиты.

| Тип | Механизм защиты |
|---|---|
| SC/MC | дистракторы из типичных ошибок понимания; «выбери и объясни» вторым шагом |
| SA | accepted_answers строгие; вопрос на свой случай, не общий |
| SA_COM | обязательный комментарий «почему» — ядро защиты; рубрика комментария |
| TA (проект) | привязка к личному проекту ученика + защита на разборе/демо |

Инвариант: задание, решаемое одним промптом к AI без понимания = брак.
Для SA_COM/TA — комментарий/рубрика проверяют понимание, не итог.

## 6. Чеклист верификации заданий (Шаг 5)

- [ ] Все типы из enum `SC|MC|SA|SA_COM|TA`; `TA` только у проектных
- [ ] Для школьной информатики урок-тема не ушёл в §3.1: есть 10 заданий на урок
      (6 простых + 3 средних + 1 проектное), считается по урок-узлу, не по главе
- [ ] Узел классифицирован по содержанию: содержательная тема (числа/строки/условия/циклы) = подкурс (10–15 + 2–3 проектных), не лист на 3; лист = 3 только для узкой подтемы
- [ ] **Глубина практики (§3.4):** каждый приём/метод материала применён ≥1 заданием; распространённые приёмы темы не пропущены (для строк — срезы + `count`/`find`/`replace`); микс типов практики (§8 п.8), не 3 однотипных
- [ ] Микс сложности соблюдён (посчитать фактически), difficulty по §2; для предвузовского/взрослого профиля к финалу доля NORMAL/HARD/проектных растёт (§3.2)
- [ ] **Профиль прогрессии (§3.5):** объём заданий растёт по коридору профиля ЦА (старт → середина → финал); не «детский» одношаговый формат весь курс для не-детской ЦА; многошаговые/проектные авто с ОДНИМ детерминируемым эталоном — предпочтительно `SA_COM` (мгновенная авто-обратная связь); `TA` — для защиты/проекта без детерминируемого эталона, не запасной вариант
- [ ] **Инвариант `answer` у SA/SA_COM (§7, ОБЯЗАТЕЛЬНО):** короткий детерминированный ответ → `answer` заполнен, `stem` диктует формат ввода; открытый ответ (объяснение/обоснование/генерация) → `answer: null` + критерий «Принято, если…» в `stem`. Не оставлять текст рубрики в поле `answer` — непустой `answer` всегда означает точное совпадение, ручного review не будет
- [ ] У каждого задания: external_uid, course_uid, наблюдаемые solution_rules
- [ ] Приоритет быстрых авто-проверок (SC/MC/SA); `SA_COM` только у задач с кодом/решением (§7); нет «сочинений» массой
- [ ] **Вовлечённость (§8):** каждое задание — действие с видимым результатом, не школьный вопрос; есть нарастающая сложность; проектные «зажигают» (реальный артефакт/кейс)
- [ ] external_uid стабильны (повторный импорт = upsert, не дубль)
- [ ] **Проектный SA_COM (§7):** в `stem` есть видимый блок «Принято, если…» (3–5
      пунктов), не только закрытая рубрика для проверяющего
- [ ] **Подсказка (`hint`) не выдаёт ответ:** если из текста подсказки арифметически/
      логически прямо получается финальное значение из `accepted_answers` — переписать
      (подсказка = метод/формула/направление, не готовый расчёт до числа)

## 7. `SA_COM` — двухфазная проверка (точная семантика)

`SA_COM` создан для **задач с кодом/решением**, где нужно быстро и дёшево:

1. **SA-фаза (авто):** проверить **правильность результата** автоматически —
   тест прошёл / вывод совпал / ответ из `accepted_answers`. Мгновенно,
   без участия человека.
2. **COM-фаза (ручная/ИИ):** препод или ИИ оценивает **качество** решения/кода
   по рубрике (читаемость, структура, без хардкода секретов, идиоматичность).

Это НЕ «короткий ответ + объясни словами почему». COM — это проверка артефакта
(кода), а не эссе ученика. Применять только там, где есть И авто-проверяемый
результат, И артефакт-решение для качественной оценки (обычно — код, конфиг,
запрос). На онбординге/теории без кода `SA_COM` почти не нужен — там `SC/MC/SA`.

`solution_rules` для `SA_COM`: `short_answer.accepted_answers` (SA-фаза) +
`text_answer.rubric` или `manual_review_required: true` (COM-фаза, критерии
качества кода).

**Проектный SA_COM (артефакт крупнее кода — бот/брендбук/пайплайн) — критерии
готовности ОБЯЗАТЕЛЬНЫ в `stem` (не только в закрытой `rubric` для проверяющего).**
SA-фаза здесь часто проверяет тривиальное поле (число шагов, имя модели) — это
не заменяет проверку самого артефакта. Добавить в `stem` видимый ученику ДО сдачи
блок «Принято, если…» (3–5 наблюдаемых пунктов: что именно должно получиться,
какие требования к артефакту). Без этого блока проектное задание не измеряет
заявленную цель «сделать» — ученик не знает, что предъявлять. Повтор-дефект:
65 проектных заданий флагмана «AI-предприниматель» сверяли только факт-пропуск,
без рубрики готовности артефакта.

**Инвариант поля `answer` (ОБЯЗАТЕЛЬНО, любой SA/SA_COM, не только проектный).**
Конвертер (`_short_answer_rules`) кладёт ЛЮБОЙ непустой `answer` целиком в
`accepted_answers` для точного совпадения (после `trim/lower/strip_punctuation/
collapse_spaces`) и одновременно ставит `manual_review_required = not accepted`
— то есть непустой `answer` ВСЕГДА означает «только точное совпадение», ручной
проверки не будет, даже если `answer` — рассуждение на 300 символов, а не короткий
ответ. Правило: если у задания есть ОДИН короткий детерминированный ответ (число,
слово, имя переменной, короткая фраза) — `answer` заполнен этим ответом, `stem`
диктует формат ввода (§9 п.3). Если ответ ОТКРЫТЫЙ (объяснение, обоснование,
сгенерированный список, любое «своими словами») — `answer: null`, а критерий
«Принято, если…» переносится в `stem` (видим ученику до сдачи, как для проектного
SA_COM выше). Смешивать нельзя: открытый ответ с непустым `answer`-рубрикой не
пройдёт ни авто-, ни ручную проверку (rubric-текст в `answer` не совпадёт с ответом
ученика дословно, а `manual_review_required` останется `false`, раз `answer`
непуст). Повтор-дефект (2026-07-10, курс «Тестировщик ПО», tsk-181): 8 из 9
заданий `SA_COM` и 6 заданий `SA` в одной главе несли в `answer` текст рубрики
вместо короткого ответа или `null` — открытые ответы учеников были бы забракованы
без права на ручной review.

## 8. Вовлечённость, игровые механики, жизненность (ОБЯЗАТЕЛЬНО)

Источники: требование оператора + @teachcase #998 (В. Устюгова) — «знание
проявляется не в согласии с объяснением, а в способности действовать без
готовой инструкции».

**Принципы (каждое задание проверять по ним):**

1. **Задание = миссия с видимым результатом, не вопрос.** Ученик ДЕЛАЕТ и
   видит исход: бот ответил, токен получен, путь скопирован, тест позеленел.
   Не «расскажи, что такое X», а «сделай X и покажи результат».
2. **Проверка через действие (teachcase):** «выбери следующий шаг», «допиши
   недостающий фрагмент», «найди и исправь чужую ошибку», «примени подход в
   новой формулировке» — вместо пересказа теории.
   Для классификаций и списков часть заданий должна проверять обобщение: дать
   новый пример, которого не было в тексте, и попросить определить класс по уже
   данному критерию. Если критерия нет, сначала исправить материал.
3. **Управляемая нарастающая сложность:** неполный пример → дополни → сделай
   с нуля → перенеси на свой проект. Внутри узла сложность растёт.
4. **Игровые механики:** явная цель и «зачем это круто»; уровни/прогресс
   (THEORY→EASY→NORMAL→HARD); проектные = «боссы» с реальным артефактом;
   персонализация (на своём кейсе — нельзя списать и интересно); мгновенная
   обратная связь (авто-чек SC/MC/SA).
5. **Жизненность:** контекст из реальной практики (реальный клиент-симуляция,
   свой бот/проект, реальный сервис вроде @BotFather), не абстрактные «задачи
   из учебника».
6. **Проектные «зажигают»:** не реферат/сочинение, а маленький реальный
   результат, которым можно гордиться/показать. Формулировать как вызов.
7. **Написание кода, не только узнавание (с момента, когда ученик уже пишет).**
   Как только в курсе введён синтаксис, в уроке ОБЯЗАТЕЛЬНО ≥1 авто-задание на
   НАПИСАНИЕ кода («напиши команду», «допиши недостающий фрагмент»), а не только
   выбор/предсказание/чтение. Учитывать нормализацию SA (`trim/lower/
   strip_punctuation/collapse_spaces`): символы (`= " ( ) { }`) при сверке стираются,
   а слова и идентификаторы (`print`, `f`, имена переменных, числа) сохраняются и
   упорядочены. Поэтому авто-«напиши код» опирать на слова/идентификаторы и их
   порядок — так надёжно отсекаются «забыл f» и «захардкодил число», а варианты
   кавычек/пробелов принимаются. Свободные программы целиком (нет одного эталона) —
   WP-миссия (`lms_skip`) или ручная проверка (`TA`/manual), не авто-SA.

8. **Разнообразие практики (не один тип массой).** Содержательную тему
   отрабатывать МИКСОМ типов: (а) предсказать вывод; (а2) **трассировка** — что
   лежит в переменной после каждой строки / сколько раз выполнится тело цикла
   (ответ — число/значение, определимо и авто-проверяемо); (б) **написать код каждым
   приёмом** темы (по заданию на приём — см. §3.4); (б2) **допиши пропущенное**
   ключевое слово / имя метода / строку целиком (fill-in, один определимый ответ);
   (в) найти и исправить ошибку в чужом коде; (г) применить приём в НОВОЙ формулировке
   (не копия примера из материала); (д) мини-проект. **Для программирующего урока
   задания на ПРИМЕНЕНИЕ (б/б2/в/г) обязательны — «предскажи вывод» в одиночку даёт
   чтение чужого кода, а не умение писать свой = брак** (повтор-дефект, курс
   Информатика 5-11: вся код-практика была «предскажи вывод»). «Напиши программу с
   нуля» без единого эталона — не авто-SA (строковый чек не тянет): проектное/ручное
   или WP-миссия `lms_skip` (см. §3.5, §9 п.3). Нормализация SA стирает
   символы (`= " ( ) { }`), но СОХРАНЯЕТ слова и идентификаторы — имена методов
   (`upper`, `count`, `replace`, `find`, `split`) survive: поэтому «напиши код
   методом X» надёжно авто-проверяется по имени метода и прямо поощряет практику
   самих приёмов.

**Запрещено:** «напиши эссе почему», «перечисли определение», длинные
текстовые ответы массой, «школьный» тон. Скучное/абстрактное задание = брак,
переформулировать в миссию с результатом.

## 9. Граница «материал ↔ задание» и однозначность (ОБЯЗАТЕЛЬНО, LMS)

В LMS материал статичен и читается ДО заданий, причём при выполнении задания текст
материала не виден. Отсюда два жёстких инварианта (нарушение = повторяющийся брак):

1. **Материал = только статичная теория.** Любой блок, который просит ученика
   ДЕЙСТВОВАТЬ — «упражнение N», «сделай / поменяй / запусти / выбери / попробуй /
   ответь» — это НЕ материал, это **задание** (или квиз). Инструкции-упражнения,
   «выбор пути/трека», код «поменяй и запусти» — выносить в задачи, не в материал.
   Перед сдачей: прочитать каждый материал-блок и спросить «требует ли он действия
   ученика?» Если да → перенести в задание. (Повтор-дефект: «Выбери путь» и
   «Упражнение 1/2» оставались в материалах.)
2. **Self-contained + однозначность.** Задание не ссылается на то, что только в
   материале (вшивать исходник в задание). **Исходник — в поле `context`** (рендерится
   и на WP, и в LMS после tsk-130), а НЕ в `stem` и НЕ ссылкой «программа выше/ниже»:
   дублирование кода в `stem` при заполненном `context` даёт код дважды. **Никогда не спрашивать «на какой строке»**
   фрагмента — комментарии и пустые строки делают нумерацию неоднозначной. Ссылаться
   на **элемент кода**: имя переменной/функции, ключевое слово, процитированную строку
   целиком. ❌ «какую строку поменять, чтобы…» → ✅ «как называется переменная, которая
   задаёт число углов?» (ответ — `uglov`) или «впиши строку целиком, которую изменишь».
3. **Определимый ответ + явный формат ввода (для авто-проверки SA/SA_COM/SC/MC).**
   У задания должен быть ОДИН ответ, который ученик может однозначно вывести из условия,
   и в стеме **прямо сказано, ЧТО и в каком виде вводить** (одно слово / число / имя
   переменной / выбрать вариант). Если ученик не понимает, что писать — брак.
   Для русского языка формат ввода включает падеж. «Впиши название знака» требует
   именительный (`точка`), «чем отделяется?» провоцирует творительный (`точкой`).
   Нельзя смешивать две формы в одном стеме, если `accepted_answers` не принимает обе.
   Перед сдачей мысленно ввести естественный ответ ученика и проверить, будет ли он
   засчитан.
   **Открытое творческое действие** («собери свою игру», «придумай бота», «сделай свой
   проект») НЕ годится для SA-автопроверки (нет единого ответа, поле «краткий ответ»
   бессмысленно): либо переформулировать в проверяемый вопрос о результате/коде
   (❌ «собери игру и выиграй» → ✅ «как называется переменная с фразой победы?» → `pobeda`),
   либо оформить как проектное задание с ручной проверкой (`TA` / `manual_review_required`),
   а само «делание» оставить в инструкции на витрине (WP/`lms_skip`), не в авто-задаче.
4. **Disclaimer природы проверки — не одна шаблонная фраза на все задания.** Если
   несколько заданий подряд (например, серия «часть 2» с творческим файлом-ответом)
   генерируются по общему шаблону текста, disclaimer о характере проверки («ответ
   файлом, проверка не мгновенная» / «сверяется по числу на выходе» / «проверяется
   вручную по критериям») ОБЯЗАН отражать РЕАЛЬНУЮ природу проверки КОНКРЕТНОГО
   задания, а не копироваться дословно с соседнего задания другой природы.
   Проверить при генерации серии: у задания с детерминируемым числовым/строковым
   выводом текст не должен утверждать «ручная проверка», и наоборот. Повтор-дефект:
   серия 13-16 ОГЭ информатика — текст банка 16 ошибочно скопирован с 13/15.
5. **Безопасность миссий.** Задание не должно просить ученика вводить, показывать,
   пересылать или проверять настоящий пароль, токен, логин, персональные данные,
   чужой аккаунт или чужой секрет. Для таких тем давать вымышленные примеры,
   чеклист оценки или безопасную симуляцию. Миссия не должна противоречить правилу
   из этого же урока: если урок говорит «пароль никому не показывай», задание не
   может просить «проверь пароль друга».
