# Оценка сложности и принципы учебного дизайна

Источник истины для skill `methodist`. Подключается в Шаге 0. Здесь — модель оценки
сложности задания, шкала уровня ученика и базовые принципы инструкционного дизайна.
Секция «Кэш веб-практик» — целевое место для выводов веб-поиск (по запросу оператора).

## Часть 1: Модель оценки сложности задания

Сложность задания = три независимых параметра. Оцениваются отдельно, не усредняются
в один балл — каждый отвечает на свой вопрос.

### Параметр 1 — когнитивный уровень (таксономия Блума)

| Уровень | Глагол-маркер | Что делает ученик | Пример (Python/Трек 1) |
|---|---|---|---|
| 1. Запомнить | назвать, перечислить | воспроизводит факт | назвать типы данных Python |
| 2. Понять | объяснить своими словами | пересказывает смысл | объяснить, зачем нужен цикл |
| 3. Применить | написать, использовать | решает по образцу | написать бот-эхо по примеру |
| 4. Анализировать | разобрать, найти причину | находит ошибку, делит на части | найти, почему бот не отвечает |
| 5. Оценить | сравнить, выбрать | обосновывает выбор | выбрать: вебхук или polling, почему |
| 6. Создать | спроектировать, собрать | собирает новое из частей | спроектировать бота под свой кейс |

Правило: учебный результат и задание формулируются глаголом одного уровня. «Понимать
рекурсию» — не результат (не наблюдается). «Объяснить рекурсию на своём примере» —
результат уровня 2.

### Параметр 2 — глубина предпосылок

Сколько ранее освоенных понятий нужно держать в голове одновременно, чтобы сделать
задание.

- **0–1 предпосылка** — изолированное новое понятие (низкая нагрузка)
- **2–3 предпосылки** — связывание нового с освоенным (средняя)
- **4+ предпосылок** — интеграция нескольких тем (высокая; для новичка — риск перегруза)

Считать буквально: выписать список понятий, без которых задание не решается. Если
список длиннее стадии ученика (см. Часть 2) — задание выше уровня, пометить.

### Параметр 3 — оценка времени

Реалистичное время на самостоятельное выполнение для целевой стадии (не для эксперта).
Считать в часах с округлением до 0.5. Для летнего формата (1 встреча/нед +
самостоятельная) суммарное время заданий модуля не должно превышать заявленный бюджет
самостоятельной работы.

### Итоговая запись сложности задания

```
Сложность: Блум L<N> (<глагол>) · предпосылок <k> · ~<t> ч · стадия <целевая>
```

## Часть 2: Шкала уровня ученика

Уровень = ЦА × стадия. **Нумерация ЦА — строго из `docs/v2/02. Три ЦА.md`**
(источник истины). Стадия — из интервью оператора (Шаг 1).

| ЦА (курсовая) | Кто | Опорная лексика | Потолок без пометки |
|---|---|---|---|
| ЦА-1 | джуны / студенты (флагман Трека 1) | базовый код понимают | Блум L1–L5, ≤3 предпосылок |
| ЦА-2 | руководители / предприниматели | бизнес-ракурс, минимум кода | Блум L1–L4, ≤2 предпосылок |
| ЦА-3 | маркетологи | бытовые аналогии, без кода в голове | Блум L1–L4, ≤2 предпосылок |

Стадия (уточняется у оператора):
- `новичок` — первый контакт с темой. Потолок: Блум L3, ≤1 предпосылка.
- `с-базой` — освоил предыдущий модуль. Потолок: Блум L4, ≤3 предпосылки.
- `уверенный` — закрепляет, готов к проектной работе. Блум L5–L6 допустим.

Матчинг: сложность задания не должна превышать потолок стадии. Если по плану нужен
шаг выше — это сигнал добавить промежуточный scaffolding-шаг, а не давать задание
выше уровня.

### Профиль прогрессии по типу ЦА (целевая траектория ≠ потолок)

Потолок (выше) защищает от перегруза в **точке входа**. Он НЕ задаёт, КУДА курс
ведёт. Второй параметр уровня — **профиль прогрессии**: тип и назначение ЦА задают
целевую ТРАЕКТОРИЮ роста по курсу (объём кода в задании, доля «напиши программу
целиком», обязательное число проектных, крутизна подъёма по Блуму). Один и тот же
«новичок» у детского кружка и у предвузовской инженерной подготовки требует разной
траектории: низкий старт может совпадать, выход — нет.

| Профиль ЦА | Объём кода в задании: старт → середина → финал | «Напиши программу целиком» (многошаговые) | Проектные (обязательно) | Подъём по Блуму |
|---|---|---|---|---|
| **Дети / младшие школьники** | 1–2 → 2–4 → 3–5 строк | мало, опционально | 0–1 мини-проект | пологий L1→L3 (L4 редко) |
| **Подростки** | 2–3 → 4–6 → 6–10 строк | часть заданий со 2-й трети | 1–2 | L1→L4 |
| **Предвузовский / абитуриент** | 3–5 → 8–10 → 10–15 строк | растущая доля, к финалу значимая | **≥2** | крутой L1→L5 |
| **Взрослый новичок-профессионал** | 4–6 → 10–15 → 15–30+ / многофайловые | доминируют к концу | **≥2–3** | L1→L6 |

**Инвариант «низкий вход ≠ низкий выход».** Для НЕ-детских профилей (подростки,
предвузовский, взрослый) низкий старт допустим — слабый ученик начинает с 1–2 команд,
— но к концу курса ОБЯЗАТЕЛЕН выход на многострочные программы и проектные задания
по профилю. Профиль определяется **назначением курса** (куда ведёт), а не только
текущим уровнем ученика. **«Детский» одношаговый формат на весь курс для не-детской
ЦА = брак** (программа предвузовская, а сделана как для подростков-новичков —
повтор-дефект, курс tsk-133).

Профиль фиксируется в интервью (SKILL Шаг 1) и проверяется в Шаге 5 / mastery-линзе.
Микс сложности по узлу (assignment-rules §3) и рост объёма заданий подстраиваются под
профиль: у предвузовского/взрослого доля сложных и проектных к финалу выше базовых
70/25/5.

## Часть 3: Принципы инструкционного дизайна (база)

Применять при проектировании прогрессии модуля (Шаг 2).

1. **Scaffolding (опоры).** Каждый шаг опирается только на освоенные предпосылки.
   Прогрессия монотонна по Блуму: запомнить/понять → применить → анализ/создание.
   Скачок через уровень без опоры = переделать.
2. **Retrieval practice (активное извлечение).** Между шагами — короткая точка, где
   ученик воспроизводит освоенное по памяти (мини-задача, объяснение своими словами),
   а не перечитывает материал.
3. **Spaced practice + спиральный мостик (распределение).** Понятие из раннего шага
   возвращается в заданиях поздних шагов в новом контексте, а не закрывается один раз.
   Для **спиральной программы** (тема повторяется на новом витке — школьный курс
   класс→класс, многоглавный): урок, опирающийся на ранее пройденное, ОБЯЗАН нести
   короткий retrieval-мостик «Вспомни: это было в … (класс/тема)» перед новым материалом
   — иначе поздний урок «телепортируется» и предполагает знание без напоминания
   (повтор-дефект, курс Информатика 5-11). То же правило — для **реюз-пары
   задание↔задание** внутри одного курса (два задания опираются на общий узел-тему
   через multi-parent граф): у второго задания-«соседа» по реюзу тоже должен быть
   короткий мостик-ссылка «ты уже видел это в задании N», не только техническое
   переиспользование узла без явной связки (повтор-дефект, ОГЭ информатика). Рендерится
   и в WP, и в LMS-материал.
4. **Формулировка через действие.** Учебный результат = наблюдаемый глагол (написать,
   объяснить, разобрать), не «знать»/«понимать».
5. **Наблюдаемые критерии готовности.** Что ученик предъявляет (рабочий код, устное
   объяснение, разбор чужой ошибки), по чему видно освоение. Не «усвоил тему».
6. **Бюджет нагрузки.** Сумма времени заданий ≤ бюджет самостоятельной работы
   формата. Перегруз срывает освоение сильнее, чем недогруз.
7a. **Индукция закономерности перед формулой (для количественных/формульных понятий,
   ОБЯЗАТЕЛЬНО).** Когда материал вводит общую формулу/закономерность (степенная
   зависимость, пропорция, любое «N = f(k)»), запрещено давать формулу сразу с одним
   примером после неё. Порядок: (1) взять МАЛЫЙ параметр (напр. 1 бит) и явно
   перечислить конкретные варианты (0 или 1 → 2 варианта; примеры «чёрный/белый»,
   «да/нет»); (2) увеличить параметр на шаг (2 бита → 00/01/10/11 → 4 варианта) —
   опять явно перечислить; (3) повторить для 3–4 параметра (не только сказать число,
   первые 1-2 раза — перечислить сами варианты); (4) явный retrieval-вопрос ученику
   «какую закономерность вы видите?» — ПЕРЕД тем, как назвать общую формулу; (5) только
   теперь дать формулу как обобщение уже увиденного, не как новый факт с потолка.
   Один расчётный пример ПОСЛЕ формулы не заменяет индукцию — ученик должен своими
   глазами увидеть 3+ конкретных случая и рост, из которого формула следует, а не
   запоминать её как given. Повтор-дефект: тема «Измерение информации» (Информатика
   5-11) — формулы `N=2^i`/`I=K·i` даны сразу, единственный пример после формулы
   не показывает роста/закономерности; новичок теряет нить.
7. **Урок-конверт: цель на входе, итог на выходе (ОБЯЗАТЕЛЬНО).** Каждый урок-материал
   ОТКРЫВАЕТСЯ целью-действием сразу после жизненного хука: «после урока ты сможешь …»
   с глаголом Блума (§1), не «узнаешь про X» и не «понять X». И ЗАКРЫВАЕТСЯ кратким
   итогом «Итог: …» (2–3 пункта — что именно освоено). Цель ставит рамку и мотив,
   итог закрепляет (эффект края). Оба блока идут И в WP-урок, И в LMS-материал
   (визуал/цель/итог **не** `lms_skip`). В программирующем уроке, где ученик впервые
   пишет/запускает код, добавляется self-contained подсказка запуска (где и как
   выполнить: онлайн-песочница / IDLE) — рендер-спец в `lms-wp-export.md` § L3.
   Повтор-дефект (курс Информатика 5-11): 133 урока без цели, 103 без итога.

## Часть 4: Кэш веб-практик

веб-поиск по инструкционному дизайну запускать **по запросу оператора или при новом
типе модуля**, не каждый прогон. Выводы (с источником и датой) дописывать сюда —
дальше skill читает кэш, не ищет заново.

Формат записи:

```
### <дата> — <тема запроса>
- Источник: <URL>, <дата обращения>
- Вывод (1–3 строки, применимо к курсу):
- Куда применили (модуль / параметр):
```
