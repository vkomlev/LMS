## Часть 1. LMS-проекция (подтверждено кодом d:\Work\LMS)

### 1.1. Иерархия
Одна модель `Courses`, иерархия — граф `course_parents` (many-to-many,
произвольная глубина, порядок `order_number` управляется триггером БД — не
задавать). «Лист» = курс без детей. «Подкурс 1-го уровня» = прямой ребёнок.
Зависимости — `course_dependencies` (`course_id` требует `required_course_id`).
Идемпотентность: курсы — по `course_uid`, задания — по `external_uid`,
материалы — по `(course_id, external_uid)`, сложность — `difficulties.uid/code`.

> Иерархию родителя читать через `GET /courses/{id}` (by-id) или `/courses/{id}/children`:
> `by-code/{code}` возвращает `parent_course_ids: []` даже у вложенных узлов (tsk-147).

### 1.1.1. Плоский курс vs подкурсы — порог и «почему» (tsk-148, enforced)
**Как LMS подаёт курс.** Внутри узла сначала идут ВСЕ его материалы, затем ВСЕ его задания.
Если у узла есть подкурсы — ученик проходит их по очереди `order_number`, и у каждого
подкурса свои материалы и задания рядом. Значит длинная сплошная теория в одном узле → к
заданиям ученик уже забыл раннюю теорию.

**Инвариант (enforce-ит публикатор):** один курс-узел-**лист** несёт ≤ 20 сущностей
(`материалы + задания`). Плоский курс уместен ТОЛЬКО для **мини-курса** (≤20). Курс с >20
суммарно обязан быть графом подкурсов (глава → урок-лист, каждый лист ≤20), иначе
`lms-publish-lesson` отклонит публикацию (в dry-run и write; обход `--max-node-items 0`).

**Практика.** Курс >20 сущностей = ОДИН граф-документ `kind:"article"` + `subcourses[]`
(не N плоских уроков-доков в один `course_slug` по отдельности). Границы узлов — по
фазам/модулям/разделам, не механически по 20. Билдер дерева —
`ContentBackbone/monolith/lms_publish/graph_authoring.build_graph_doc`; полный контракт
полей узла — `ContentBackbone/docs/wp-content-contract.md` § «Структура подкурсами».

### 1.2. Артефакты выгрузки (генерировать ОБА)

**A. JSON под bulk-API (основной).**
- Курсы: список `CourseCreate` — `title`, `course_uid`, `description`,
  `access_level` (`self_guided|auto_check|manual_check|group_sessions|personal_teacher`),
  `is_required`, иерархия через `parent_course_uid`, зависимости —
  `required_courses_uid`.
- Задания: массив `TaskUpsertItem` под `POST /api/v1/tasks/bulk-upsert`:
```json
{
  "external_uid": "PY-EGE-T1-03-SC",
  "course_uid": "PY-EGE-T1",
  "difficulty": "EASY",
  "max_score": 10,
  "task_content": {
    "type": "SC",
    "stem": "Что выведет print(2**3)?",
    "options": [
      {"id": "A", "text": "6", "is_active": true},
      {"id": "B", "text": "8", "is_active": true}
    ]
  },
  "solution_rules": { "max_score": 10, "correct_options": ["B"], "scoring_mode": "all_or_nothing" }
}
```
  Для `SA/SA_COM` — `solution_rules.short_answer.accepted_answers`; для `TA` —
  `solution_rules.text_answer.rubric` (список `{id,title,max_score}`).
- Материалы: `course_uid`, `external_uid`, `title`, `type`
  (`text|video|audio|image|link|pdf|office_document|script|document`),
  `content` (для text → `{text, format: markdown}`), `order_position`.

**B. Google Sheets раскладка (дополнительно, для ручного импорта).**
- Лист `Courses`: `course_uid | title | access_level | description |
  parent_course_uid | order_number | required_courses_uid | is_required`
- Лист `Tasks`: `external_uid | type | stem | options ("A: .. | B: .. [пояснение]") |
  correct_answer ("A" / "A,B") | accepted_answers ("вар:10 | вар2:5") |
  max_score | course_uid | difficulty_code | code | title | prompt`
- Лист `Materials`: `course_uid | external_uid | title | type | url |
  description | caption | order_position | is_active`

### 1.3. Difficulty (фиксированный справочник)
`theory/THEORY(1)`, `easy/EASY(2)`, `normal/NORMAL(3)`, `hard/HARD(4)`,
`project/PROJECT(5)`. Маппинг бакетов — см. `assignment-rules.md` § 2.

### 1.4. Структура подкурсами (большие/детские курсы — НЕ плоско)

Логика LMS: **сначала изучаются ВСЕ материалы курса, потом все задания**. У плоского
курса (вся теория сотнями материалов → потом сотни заданий) ранняя теория «протухает» к
моменту практики. Поэтому большой курс (школьный класс, многоглавный) проектировать
**графом подкурсов**, а не одним плоским узлом:

**курс (класс) → глава → тема (лист)**. Каждый **лист-подтема** = отдельный подкурс
**со своими материалами И заданиями рядом** (изучил → сразу применил). Промежуточные
узлы (курс, глава) держат только вводный материал; материалы+задания живут в листьях.

Реализация при публикации через ContentBackbone: документ-корень (`kind:"article"`)
несёт поле `subcourses[]` — рекурсивный список узлов `{course_uid, title, order,
requires, blocks, subcourses}`. У листа `blocks` = теория + `control_question`/`task`
этой подтемы; конвертер режет их на материалы (по H2) и авто-задачи **того же узла**.
Точная спецификация и пример — `D:\Work\ContentBackbone\docs\wp-content-contract.md`,
раздел «Структура подкурсами (граф класс → глава → тема)» + `_build_course_node` в
`lms_publish/blocks_to_lms.py`. `course_uid` стабильны (идемпотентный upsert); при смене
структуры старые узлы деактивировать (иначе сироты).

## Часть 2. WP-проекция — модель «Навигатор» (подтверждено на victor-komlev.ru)

Сайт = WordPress + Elementor, **LMS-плагина нет**. «Курс» = ручная разметка:
посты + страница-навигатор. Зрелая модель (по образцу ЕГЭ-курса):

**L1 — страница-навигатор (оглавление курса)**
- H1: `«<Курс>. Навигатор по курсу.»`
- slug: `/<kurs>-navigator-po-kursu/`
- **«Что нужно перед стартом» (ОБЯЗАТЕЛЬНО, если курс/трек опирается на внешний
  инструмент вне браузера/ОС — n8n, IDE, КуМир и т.п.):** список инструментов/
  аккаунтов + либо ссылка на онбординг-урок ВНУТРИ этого трека, либо явная пометка
  «устанавливается в Треке N, урок M» со ссылкой. Трек, продающийся отдельно, не
  должен предполагать инструмент из соседнего трека без точки входа (повтор-дефект:
  Трек 3 «AI-предприниматель» требовал n8n с урока 2.2 без онбординга).
- линейный пронумерованный список разделов (анкор + кнопка «Перейти»)
- финальный CTA «Следующий урок» → первый раздел

**L2 — страница темы/раздела** — slug `/<kurs>-tema-{N}-{slug}/`, фикс. порядок
блоков:
1. H1 `«<Курс>. Тема N. <название>»`
2. **Краткий план раздела** (маркированный список целей)
3. 📖 **Текстовые уроки** (ссылки на L3-посты, допустим deep-link `#anchor`)
4. 👀 **Видеоуроки**
5. ❓ **Контрольные вопросы** (SC/SA из заданий)
6. 💻 **Задания** (практика — сюда задания с критериями)

   **Пункты 5 и 6 — ссылка ОБЯЗАНА вести на конкретный урок + якорь конкретного
   вопроса/задания** (`{url-урока-с-этим-вопросом}#{anchor-этого-вопроса}`), НЕ на
   первый урок темы общей ссылкой без якоря. У разных пунктов списка 5/6, относящихся
   к разным урокам, — РАЗНЫЕ href. Якорь на L3 — `#cq-{external_uid}` (контрольный
   вопрос) / `#task-{external_uid}` (задание), см. якоря L3 ниже. Повтор-дефект: все
   пункты «Контрольные вопросы» и «Задания» темы вели на один и тот же первый урок
   темы без якоря (Информатика 5-11) — ученик не попадает к нужному вопросу.
7. **Итоги раздела**
8. Нижняя навигация: **«Назад» / «Вперёд» / «На главную»** + счётчик
   **«Раздел N из M»** (M фиксируется в плане заранее, заполняется вручную)

**L3 — пост-урок** (обычный WP-пост `/{slug}/`), фикс. «конверт» урока:
заголовок → жизненный хук → **цель «Чему научишься»** (callout, действие «после
урока сможешь …») → *(для спиральных курсов — мостик «Вспомни: …» на прошлый
класс/тему)* → теория → примеры кода → скриншоты → *(первый программирующий
урок — подсказка «Как запустить»: онлайн-песочница / IDLE)* → «Задание» →
**«Итог»** (callout, 2–3 пункта) → опц. CTA «Изучить с наставником» → видео.
Якоря `#section` на ключевые секции (для deep-link из L2). **Каждый `control_question`
и `task`-блок урока получает свой стабильный якорь** — `#cq-{external_uid}` /
`#task-{external_uid}` (идемпотентно, тот же external_uid что и у задания в LMS) — это
то, на что должны ссылаться пункты 5/6 L2-навигатора (см. выше), а не общий якорь
урока.

> **Конверт урока идёт в ОБЕ проекции.** Цель, итог, мостик, подсказка запуска —
> это учебные блоки материала, а НЕ WP-навигация: в LMS-материал они входят
> (**не** `lms_skip`). `lms_skip` — только для блоков навигации/перелинковки
> (Назад/Вперёд, SEO-ссылки). Правило дизайна — `difficulty-and-design.md`
> Часть 3 п.7; рендер блоков — digital-copywriter `wp-publish`/`lms-publish`.

**Обязательно учесть:** нет авто-навигации/прогресса — порядок и общее M
зафиксировать в плане; breadcrumbs курса через `/shkolnye-programmy/...` (не
блоговый рубрикатор); single-page (мини-курс) — только для коротких узких тем;
плоский «лендинг+посты» без возврата к программе — не использовать для новых.

### L3 — навигация и перелинковка (ОБЯЗАТЕЛЬНО для многостраничных курсов)

1. **Блок навигации в конце каждого урока:** `← Назад · ↑ В начало урока ·
   ⌂ В начало курса · Вперёд →`. «Назад/Вперёд» ведут по урокам **сквозь весь
   курс** (конец темы → первый урок следующей; на краях — страница-навигатор
   курса). Цель — ученик идёт дальше, не возвращаясь в навигатор; страницы держать
   небольшими.
2. **SEO-перелинковка:** в тексте урока — контекстные ссылки на смежные материалы
   сайта (понятия: условия, циклы, функции, словари, git, обработка ошибок; и
   профильные курсы). Только на **реально существующие** страницы (URL сверять,
   не выдумывать), без переспама — концентрировать на обзорных уроках (чеклист,
   карта, финал) + по одной там, где понятие центральное.
3. **Новая вкладка:** внешние и межстраничные ссылки (перелинковка) открывать в
   новой вкладке — `target="_blank" rel="noopener"`. Внутренняя навигация курса —
   в той же вкладке.
4. **Внутренние ссылки — фактические пермалинки.** WP строит вложенные адреса
   `/<навигатор>/<тема>/<урок>/`; навигацию и ссылки между уроками/темами строить
   по ним, не по плоским slug (иначе переходы не работают). Те же вложенные адреса
   — в section-ссылках навигаторов (L1 items, L2 📖/❓/💻) и в `prev/next`.

## Часть 3. Таблица соответствия LMS-узел ↔ WP-страница

| План (узел) | LMS | WP |
|---|---|---|
| Корневой курс | `Courses` root, `course_uid` | страница-навигатор L1 |
| Подкурс 1-го уровня (Фаза/Модуль) | child `Courses` + `course_parents` | раздел в навигаторе + L2 страница темы |
| Лист (тема/урок) | child `Courses` (без детей) | L2-блок 📖 + L3 пост-урок |
| Материал | `materials` (text→markdown) | контент L3-поста |
| Задание | `tasks` (task_content/solution_rules) | L2-блок 💻 / ❓ |
| Зависимость прохождения | `course_dependencies` | порядок «Раздел N из M» + Назад/Вперёд |

Один узел плана → одновременно строка LMS-выгрузки и страница WP. `course_uid` /
`external_uid` — общий ключ обеих проекций (стабильный, идемпотентный).
