# Pre-Landing ревью PR

Ты проводишь `/pr-review`. Анализируй diff текущей ветки относительно базовой на структурные проблемы, которые не ловят тесты.

---

## Шаг 0: Определить базовую ветку

1. Проверь, есть ли уже PR для этой ветки:
   `gh pr view --json baseRefName -q .baseRefName`
   Если успешно — используй это имя как базовую ветку.

2. Если PR нет — определи ветку по умолчанию:
   `gh repo view --json defaultBranchRef -q .defaultBranchRef.name`

3. Если оба упали — используй `main`.

Во всех последующих командах `git diff`, `git log`, `git fetch` подставляй найденную базовую ветку.

---

## Шаг 1: Проверить ветку

1. `git branch --show-current`
2. Если на базовой ветке → "Нечего проверять — ты на базовой ветке." и стоп.
3. `git fetch origin <база> --quiet && git diff origin/<база> --stat` — проверь что есть diff.

---

## Шаг 2: Получить diff

```bash
git fetch origin <база> --quiet
git diff origin/<база>
```

Это включает и закоммиченные, и незакоммиченные изменения относительно последней базовой ветки.

---

## Шаг 3: Двухпроходной обзор

### Проход 1 — КРИТИЧЕСКИЙ (проверить в первую очередь):

**SQL и безопасность данных:**
- Сырые строки SQL без параметризации (SQL-инъекция)
- Bulk UPDATE/DELETE без WHERE с явным ограничением
- Миграции с потенциальной блокировкой таблицы (ADD COLUMN NOT NULL без DEFAULT, полное пересоздание индекса)
- Запросы к БД вне транзакций при нужде в атомарности

**Гонки состояний и параллелизм:**
- check-then-act без пессимистичной блокировки (read + conditional write без блокировки)
- Множественные обновления записей без WHERE на проверку состояния
- Фоновые задачи создающие дубликаты без идемпотентности

**Граница доверия LLM:**
- LLM-вывод используется напрямую в SQL/командах без санитизации
- Структурированный LLM-вывод (JSON) используется без валидации схемы
- LLM-вывод записывается в БД без проверки типа
- LLM-вывод отображается пользователям без экранирования

**Полнота enum и значений:**
- Новое enum-значение/статус/константа добавлены, но не все switch/case/if-цепочки обновлены
- **ТРЕБУЕТ ЧТЕНИЯ КОДА ВНЕ DIFF:** используй Grep чтобы найти все файлы ссылающиеся на соседние значения, прочитай их и проверь обработку нового значения

### Проход 2 — ИНФОРМАЦИОННЫЙ:

**Условные побочные эффекты:**
- Внешние вызовы (email, webhook, запись в БД) внутри условий, которые могут выполниться неожиданно
- Необратимые действия без идемпотентной защиты

**Магические числа и строковая связанность:**
- Хардкод значений которые должны быть константами
- Дублированные строки вместо ссылок на константы

**Мёртвый код и консистентность:**
- Импорты/переменные/методы объявленные но неиспользуемые
- Закомментированный код который должен быть удалён

**Проблемы LLM-промптов:**
- Промпты с инструкциями противоречащими системным инструкциям
- Промпты открытые для инъекции через пользовательский ввод

**Пробелы в тестах:**
- Новые кодпути без тест-покрытия
- Отсутствие тестов на граничные случаи для изменённой логики

## Шаг 4: Fix-First ревью

**Каждая находка получает действие — не только критические.**

Вывести заголовок: `Ревью перед мержем: N проблем (X критических, Y информационных)`

### Шаг 4a: Классифицировать каждую находку

**AUTO-FIX** если:
- Механический баг с очевидным исправлением (мёртвый код, хардкод → константа)
- Стиль/консистентность без трейдоффов
- Простое добавление недостающей проверки

**СПРОСИТЬ** если:
- Несколько валидных подходов к исправлению
- Исправление затрагивает дизайн или API
- Неясны намерения автора
- Критические находки (гонки, SQL-инъекции) — всегда спрашивать

### Шаг 4b: Автоисправить все AUTO-FIX элементы

Примени каждое исправление напрямую. Для каждого вывести:
`[АВТО-ИСПРАВЛЕНО] [файл:строка] Проблема → что сделано`

### Шаг 4c: Батч-вопрос по СПРОСИТЬ элементам

Если есть СПРОСИТЬ элементы, представь их в **одном** вопросе пользователю:

```
Автоисправил 5 проблем. 2 требуют твоего решения:

1. [КРИТИЧЕСКОЕ] app/models/order.py:42 — Гонка состояний при смене статуса
   Исправление: Добавить WHERE status = 'pending' в UPDATE
   → A) Исправить  B) Пропустить

2. [ИНФОРМАЦИОННОЕ] app/services/gen.py:88 — LLM-вывод не валидируется перед записью в БД
   Исправление: Добавить валидацию JSON-схемы
   → A) Исправить  B) Пропустить

РЕКОМЕНДАЦИЯ: Исправить оба — #1 реальная гонка, #2 предотвращает молчаливое повреждение данных.
```

Если СПРОСИТЬ элементов <= 3, можно использовать индивидуальные вопросы вместо батча.

## Шаг 5: Кросс-референс TODOS.md

Прочитай `TODOS.md` в корне репозитория (если существует):
- **Закрывает ли этот PR открытые TODO?** Если да — отметить в выводе.
- **Создаёт ли этот PR работу для нового TODO?** Флагировать.
- **Есть ли связанные TODO дающие контекст?** Ссылаться при обсуждении.

---

## Шаг 6: Проверка устаревшей документации

Для каждого `.md` файла в корне репо (README.md, ARCHITECTURE.md, CLAUDE.md и т.д.):
1. Затрагивают ли изменения в diff фичи/компоненты описанные в этом файле?
2. Если файл документации **не** обновлён в этой ветке — флагировать как ИНФОРМАЦИОННОЕ:
   "Документация может устареть: [файл] описывает [фичу/компонент] но код изменился."

Информационно только — никогда не критично.

---

## Важные правила

* Никогда не флагировать изменения которые чётко реализованы корректно
* Никогда не предлагать рефакторинг ради рефакторинга в рамках ревью
* Сосредоточиться на структурных проблемах: безопасность, корректность, молчаливые сбои
* Если находка неоднозначная — спросить, а не автоисправлять
* Для enum — всегда проверять файлы ВНЕ diff (через Grep)
