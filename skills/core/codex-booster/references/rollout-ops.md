# Rollout Operations

## Packaging Source of Truth
Managed high-value skills should use the packaging manifest:
- [skill-packaging-manifest.json](/d:/Work/IDE_booster/Docs/ai-booster/skill-packaging-manifest.json)
- [skill-template-sources.json](/d:/Work/IDE_booster/Docs/ai-booster/skill-template-sources.json)

Preferred packaged rollout path:

```powershell
python d:\Work\IDE_booster\scripts\package-skills.py --dry-run
```

Package selected managed skills:

```powershell
python d:\Work\IDE_booster\scripts\package-skills.py --skills qa-report qa-fix release-prep
```

For generated managed skills, `package-skills.py` also rebuilds `SKILL.md` from template sources before syncing mirrors.
Generated `SKILL.md` files contain an `AUTOGENERATED` guard comment and should not be edited manually; update the template source instead.
The packaging step also regenerates project-level `AGENTS.md` from `skills/core` so Codex session discovery sees the current project skill catalog.
It also installs the managed global instructions from `Docs/ai-booster/codex-global-AGENTS.md` to `C:/Users/user/.codex/AGENTS.md`.

## Preferred Path (Scripted)
Use fleet sync script with dry-run first:

```powershell
powershell -ExecutionPolicy Bypass -File d:\Work\IDE_booster\scripts\sync-skills.ps1 -DryRun
```

Sync selected skills:

```powershell
powershell -ExecutionPolicy Bypass -File d:\Work\IDE_booster\scripts\sync-skills.ps1 -SkillNames ai-orchestrator,codex-booster
```

Sync all skills:

```powershell
powershell -ExecutionPolicy Bypass -File d:\Work\IDE_booster\scripts\sync-skills.ps1
```

## Manual Path (When Needed)
1. Copy skill folder from `d:/Work/IDE_booster/skills/<skill-name>`.
2. Paste into `<project>/skills/core/`.
3. Verify `<project>/skills/core/<skill-name>/SKILL.md` exists.

Use manual copy only as fallback when the packaging script cannot be used.

## Local Codex Installation
Copy skill folders to:
- `C:/Users/user/.codex/skills/`

Then restart Codex session to refresh available skills.

## Codex Project Binding

When importing project setup from Claude, apply [codex-project-binding.md](codex-project-binding.md):

1. Keep `d:/Work/IDE_booster/skills` as source-of-truth for managed skills.
2. Use `scripts/package-skills.py` to sync `C:/Users/user/.codex/skills`, project `skills/core`, and generated `AGENTS.md`.
3. Treat `C:/Users/user/.codex/config.toml` as runtime config for trusted projects and MCP servers, not as a committed project artifact.
4. Never copy Claude `permissions.allow` / `permissions.deny`, hooks, or credentials directly into Codex; map the intent to Codex tools, sandbox, MCP, automation, or project trust.

## Skill Packaging Preflight (Mandatory)
Before rollout/troubleshooting for any new or updated skill:

1. Parse `SKILL.md` frontmatter as valid YAML (`name`, `description` required).
2. Run validator:

```powershell
python C:\Users\user\.codex\skills\.system\skill-creator\scripts\quick_validate.py C:\Users\user\.codex\skills\<skill-name>
```

3. If UI list/chips visibility is required, verify:
- `C:\Users\user\.codex\skills\<skill-name>\agents\openai.yaml` exists;
- `display_name`, `short_description`, `default_prompt` are set.

## Verification Checklist
- Skill exists in booster source.
- Skill synced to project `skills/core`.
- Project root `AGENTS.md` regenerated and includes the target skill.
- Global `C:/Users/user/.codex/AGENTS.md` matches `Docs/ai-booster/codex-global-AGENTS.md`.
- `SKILL.md` frontmatter parsed successfully (no YAML errors).
- `quick_validate.py` passed for the target skill.
- `agents/openai.yaml` exists when UI visibility is expected.
- Skill visible in local Codex skill list after restart.
- One sample prompt triggers the expected skill behavior.
