# Как прочитать курс из источника

Skill проверяет **готовый** курс. Поддерживаемые источники: LMS (граф `Courses`),
живой сайт WordPress (модель «Навигатор») и исходники CreateCourses
`courses/<slug>/`. Цель — получить линейную траекторию, в которой ученик реально
идёт по курсу, и затем пройти её атомами.

Кэш фактов схемы LMS и структуры сайта — источник истины у методиста:
`references/lms-wp-export.md`. При расхождении с актуальным кодом/сайтом —
перепроверить и обновить тот файл в профильном контуре, не выдумывать.

---

## Источник A. LMS (граф Courses, read-only)

**Только чтение.** Никаких INSERT/UPDATE/DELETE. Доступ — как в `/db-check`
(MCP PostgreSQL или `psql` по строке подключения из окружения LMS,
`d:\Work\LMS`). Если доступа к БД нет — сообщить оператору, не выдумывать
содержимое.

**Контур обязателен.** Для опубликованного курса LMS проверять prod только через
MCP alias `learn_prod_db` или живой `learn.victor-komlev.ru` /
`api.learn.victor-komlev.ru`. Alias `postgres` и `learn_public_db` считать
локальным/dev, даже если схема похожа. В `audit.md` отдельно писать, что было
проверено: `LMS prod DB`, `LMS dev DB`, `WP live`, `local export`. Если prod
alias недоступен, вывод формулировать как «prod не проверен».

**Модель (подтверждено кодом d:\Work\LMS):**
- Одна таблица `courses`, иерархия — граф `course_parents` (many-to-many,
  порядок `order_number` ведёт триггер). «Лист» = курс без детей.
- Материалы — `materials` (`course_id`, `external_uid`, `title`, `type`,
  `content`, `order_position`).
- Задания — `tasks` (`external_uid`, `course_id`, `task_content`,
  `solution_rules`, `difficulty`).
- Зависимости прохождения — `course_dependencies`
  (`course_id` требует `required_course_id`).

**Как собрать траекторию:**
1. Найти корневой курс по `course_uid` или названию.
2. Обойти `course_parents` от корня вглубь, соблюдая `order_number`; получить
   порядок листьев.
3. Логика LMS: внутри узла сначала все материалы (`order_position`), потом все
   задания. Собрать для каждого листа `материалы[] -> задания[]`.
4. Наложить `course_dependencies`: если узел X требует Y, Y должен идти раньше.
5. Для заданий смотреть `solution_rules`: есть ли критерий правильного ответа.

Читать `content` материалов — это текст, который видит ученик; по нему идёт
атомный проход.

**Правильные поля блочного JSON.** Публикации и исходные блоки хранят текст по
типу блока:
- проза (`paragraph`, `callout`, `list`, `intro`, `desc`) -> `html`, `text` или
  `items` по фактической структуре;
- вопросы и задания (`control_question`, `task`) -> `stem`, варианты в `options`;
- поле `text` у ряда блоков может быть пустым и не доказывает пустой контент.

Перед P0 «проза пропала / вопрос пустой» проверить правильное поле, read-only
запрос, исходник или живой рендер.

## Источник B. Живой сайт WordPress

Сайт victor-komlev.ru: WP + Elementor, LMS-плагина нет. Курс = ручная разметка
(посты + страница-навигатор). Для сложной верстки/JS использовать браузер.

**Структура:**
- L1 — навигатор курса (`/<kurs>-navigator-po-kursu/`): пронумерованный список
  разделов.
- L2 — страница темы (`/<kurs>-tema-{N}-{slug}/`): план раздела, уроки, видео,
  контрольные вопросы, задания, итоги, навигация.
- L3 — пост-урок (`/{slug}/`): теория, примеры, скриншоты, задания, видео,
  якоря `#section`.

**Как собрать траекторию:**
1. Открыть L1 и получить порядок разделов.
2. По каждому L2 вытащить ссылки на L3-уроки, вопросы и задания.
3. Пройти L3-посты в порядке из L2.
4. Проверить ссылки «Назад/Вперёд», deep-link `#anchor`, «Раздел N из M».
5. Отсутствие цели урока, критерия задания или итогов раздела — находка по
   полноте.

**Визуалы проверять рендером, а не только JSON/HTML.** Если аудит касается
картинок, инфографики, скриншотов, ASCII, якорей или адаптивности, открыть живую
страницу и зафиксировать факт: изображение загрузилось, не обрезано, не смещено,
текст читается, визуал стоит рядом с нужным понятием. Для детских курсов
проверять наличие визуала под ключевые понятия внутри материала, а не только
обложку.

Для LMS-публикации отдельно проверить, как урок будет разрезан на материалы:
первый H2 не должен дословно повторять `title` урока, а соседние материалы не
должны иметь одинаковый заголовок при разном содержании. Такой дубль считать
структурным дефектом даже тогда, когда исходный WP-блок выглядит «нормально».

Визуальное покрытие считать по смысловым материалам/H2, а не по уроку целиком:
если раздел стал отдельным LMS-материалом, у него должен быть свой обучающий
визуал или явная причина отсутствия. Обложка не закрывает материал. Для
абстракций, безопасности, интерфейсов, памяти, программ, сети и терминов-ловушек
отсутствие обучающего визуала — W6 до доказательства обратного.

Если живой рендер недоступен, явно написать в `audit.md`: «визуальный слой не
проверен». Нельзя выдавать общий вердикт по визуалам только по исходному JSON.
Для ASCII-схем сделать скриншот или визуальный осмотр: `font-family: monospace`,
HTTP 200 и отсутствие горизонтального скролла не доказывают, что рамки, стрелки
и подписи не разъехались.

## Источник C. Исходники CreateCourses

Если курс ещё не опубликован или оператор просит проверить исходники, читать:
- `courses/<slug>/README.md`;
- `courses/<slug>/plan/`;
- `courses/<slug>/materials/`;
- `courses/<slug>/assignments/`;
- `courses/<slug>/exports/`, если нужно сравнить LMS/WP-проекции.

Порядок брать из плана, экспортов или структуры курса. Если порядок нельзя
восстановить, задать один короткий вопрос оператору. Review-артефакты сохранять
в `courses/<slug>/reviews/<segment>/`.

Для блочных материалов выделять атомы по блокам: `paragraph`, `callout`, `list`,
`image`, `ascii`, `task`, `control_question` и соседние смысловые блоки. Текст
искать в правильных полях (`html`, `text`, `items`, `stem`) по фактической
структуре блока.

Для блоков `task` и `control_question` читать не только `stem`, но и
`answer`/`accepted_answers`/`solution_rules`, если они есть в экспорте. SA и
SA_COM проверять как ученик: какой естественный правильный ответ он введёт и
будет ли он засчитан после нормализации.

---

## Если источник не указан

Спросить оператору одним коротким вопросом: какой курс/сегмент проверяем и
откуда — `course_uid` в LMS, URL навигатора на сайте или путь `courses/<slug>/`.
Без конкретного курса проверка невозможна.

## Смешанный случай

Если курс живёт и в LMS, и на сайте, проверить обе проекции и отдельно отметить
рассинхрон: тема есть на сайте, но нет в LMS, материал в LMS есть, но визуал
потерялся на WP, ссылки ведут в разные места и т.п.
