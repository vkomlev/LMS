# Приоритеты, вердикты, грейд и формат отчёта

## Шкала приоритета находки

- **P0 — блокер:** методический дефект, который срывает обучение по своей цели (цель не
  оценивается вовсе; ключевой приём нигде не отрабатывается; термин-стена без объяснения;
  заявленная тема не раскрыта; битая обязательная навигация). Курс по этому месту не учит.
- **P1 — серьёзно:** ученик дойдёт, но с высоким риском не освоить или бросить (скачок
  сложности; задание без критерия; сухой демотивирующий блок; практика «на узнавание»).
- **P2 — заметно:** трение, снижение эффективности, риск ошибки; исправимо без переделки.
- **P3 — косметика:** мелкая шероховатость формулировки/оформления, обучению не мешает.

## Вердикт по критерию (К1–К11)

- **ОК** — критерий выполнен, находок нет или только P3.
- **ОГОВОРКИ** — есть P1/P2, но нет P0 по этому критерию.
- **ПРОВАЛ** — есть хотя бы один P0 по этому критерию.
- **н/п** — критерий не применим к данному материалу (не выдумывать находку).

## Общий грейд A–F (свёртка)

Опорные критерии: **К1 (выравнивание), К3 (полнота), К4 (практика/освоение)** — весомее.

- **A** — все критерии ОК; ноль P0/P1; курс методически образцовый.
- **B** — нет P0; единичные P1 по неопорным критериям; опорные — ОК.
- **C** — нет P0; есть P1 (в т.ч. по опорным) или множественные P2; проходит с доработкой.
- **D** — один P0 по **неопорному** критерию, ИЛИ множественные P1 по опорным; требует
  серьёзной переработки перед запуском.
- **F** — хотя бы один P0 по **опорному** критерию (К1/К3/К4), ИЛИ два+ P0 суммарно; курс
  не готов методически.

Правило: грейд не завышать «из вежливости». Один P0 по опорному критерию = F, сколько бы
ни было удачных мест. В обосновании грейда назвать критерии, которые его определили.

## Формат находки

```
[P1][К4 Практика/освоение][Merrill: применение]
Место: LMS course_uid=py-strings-01, узел «Срезы строк», задания 1–3
Что нарушено: тема «срезы» вводит 4 приёма (индексация, срез с шагом, обратный срез,
  срезы + len), но все 3 задания — на один приём (простой срез). Приёмы «шаг» и
  «обратный срез» показаны в материале, но не отработаны учеником ни разу.
  Принцип Merrill (application): каждый показанный приём требует применения руками;
  без этого — узнавание, а не умение (линза mastery не пройдена).
Предложение правки: расширить банк до 8–10 разнообразных заданий с покрытием всех
  4 приёмов по нарастанию: (1–2) простой срез → (3–4) срез с шагом → (5–6) обратный
  срез → (7–8) комбинация срез+len в жизненной задаче (обрезать телефон/маскировать
  карту) → (9–10) проектное. Микс сложности 70/25/5.
```

Поля: `[Приоритет][Критерий Kn Название][Нарушенный принцип]` · `Место` (course_uid /
URL / узел-раздел) · `Что нарушено` (факт + **ссылка на методический принцип**) ·
`Предложение правки` (конкретный текст/задание/шаг — не «улучшить»).

Для находок по материалам дополнительно указывать: `Термины/понятия` — какие новые
понятия проверялись, какие ранее введённые термины требуют отсылки, какая опора есть
сейчас (`визуал`, `развёрнутый пример`, `однословный пример`, `нет`). Это не отдельная
длинная таблица ради таблицы, а доказательство, что материал пройден по обязательным
микрогейтам.

Для находок по заданиям дополнительно указывать: `Банк заданий` — сколько заданий в
уроке, распределение по сложности, есть ли проектное; `Подсказки` — не сливают ли
готовый ответ; `Перенос` — если в задании есть объект/вариант вне материала, указать,
дан ли в материале признак для обобщения. Не флагать сам факт нового объекта как дефект.
Для школьной информатики ориентир урока: 6 простых + 3 средних + 1 проектное задание.

## Шаблон итогового отчёта

```markdown
# Профессиональный аудит курса «<название>»

Источник: <LMS course_uid=... | URL сайта>
Заявленная цель / ЦА: <что ученик должен уметь / для кого>
Дата: <YYYY-MM-DD>

## Общий грейд: <A–F>
Определили грейд: <перечислить критерии-виновники, кратко почему>.

## Сводка по критериям
| # | Критерий | Рамка | Вердикт | P0 | P1 | P2 | P3 |
|---|---|---|---|---|---|---|---|
| К1 | Выравнивание целей | Biggs | ПРОВАЛ | 1 | 1 | 0 | 0 |
| К2 | Последовательность/нагрузка | Sweller | ОГОВОРКИ | 0 | 2 | 1 | 0 |
| К3 | Полнота | — | ОК | 0 | 0 | 1 | 0 |
| К4 | Практика/освоение | Merrill | ПРОВАЛ | 1 | 0 | 0 | 0 |
| К5 | Оценивание/обратная связь | — | ОГОВОРКИ | 0 | 1 | 1 | 0 |
| К6 | Мотивация/вовлечённость | ARCS | ОГОВОРКИ | 0 | 2 | 0 | 1 |
| К7 | Понятность | — | ОК | 0 | 0 | 1 | 2 |
| К8 | Доступность | UDL | ОГОВОРКИ | 0 | 1 | 1 | 0 |
| К9 | Структура/навигация | — | ОК | 0 | 0 | 2 | 1 |
| К10 | Нагрузка/темп | — | ОК | 0 | 0 | 0 | 0 |
| К11 | Точность/актуальность | — | ОГОВОРКИ | 0 | 1 | 0 | 0 |

## Чинить в первую очередь (P0 → P1)
1. [P0][К1] ...
2. [P0][К4] ...
3. [P1][К6] ...

## Находки по критериям
### К1. Выравнивание целей
<находки по формату выше, P0→P3>
### К2. Последовательность и когнитивная нагрузка
...
### (остальные критерии; критерии без находок — «ОК, замечаний нет» или «н/п»)

## Карта покрытия цели
- Заявленная цель курса: <...>
- Что курс реально даёт: <...>
- Разрыв: <чего не хватает, чтобы цель достигалась>

## Что уже сильно
<2–3 методически удачных решения курса, которые стоит сохранить>
```

Правило: отчёт — на русском, простыми словами (по AGENTS.md); методические термины — с
короткой расшифровкой в скобках. Каждая находка имеет нарушенный принцип и предложение
правки. Skill **не редактирует** исходник — только предлагает.
