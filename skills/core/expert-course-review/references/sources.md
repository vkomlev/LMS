# Как прочитать собранный курс для аудита

Skill аудирует **готовый** курс. Два источника: LMS (граф `Courses`) и живой сайт
(WordPress, «Навигатор»). Цель — восстановить три вещи для рубрики: **структуру**
(подкурсы/разделы, зависимости), **линейную траекторию** «материал → задание» и
**банк заданий** (типы, сложность, критерии `solution_rules`).

Кэш фактов схемы LMS и структуры сайта — источник истины у методиста:
`references/lms-wp-export.md`. При расхождении с актуальным
кодом/сайтом — перепроверить и обновить тот файл (не выдумывать). Способ чтения
согласован с `naive-learner-review/references/sources.md` — источник тот же, отличается
только линза анализа (рубрика вместо прохода персоны).

---

## Источник A. LMS (граф Courses, read-only)

**Только чтение.** Никаких INSERT/UPDATE/DELETE. Доступ — как в `/db-check`
(MCP PostgreSQL или `psql` по строке подключения из окружения LMS, `d:\Work\LMS`).
Нет доступа к БД → сообщить оператору (ветка Б operator-handoff), не выдумывать.

**Контур обязателен.** Для опубликованного курса LMS проверять prod только через
MCP alias `learn_prod_db` или живой `learn.victor-komlev.ru` /
`api.learn.victor-komlev.ru`. Alias `postgres` и `learn_public_db` считать
локальным/dev. В отчёте разделять `LMS prod DB`, `LMS dev DB`, `WP live`,
`local export`; если prod alias недоступен, писать «prod не проверен».

**Модель и механика чтения** — не дублируем: источник истины по схеме `courses` /
`course_parents` / `materials` / `tasks` / `course_dependencies` — методистов
`references/lms-wp-export.md`; способ восстановления
линейной траектории — `C:/Users/user/.codex/skills/naive-learner-review/references/sources.md`,
Источник A. Ниже — только то, что специфично для **этого** аудита: что извлечь под
каждый критерий рубрики.

**Что собрать для рубрики:**
1. **Структура (К9):** обойти граф `course_parents` от корня (`course_uid` даёт оператор
   или ищем по названию), зафиксировать иерархию и `order_number`. Плоский большой курс
   без подкурсов = находка К9.
2. **Траектория (К2):** внутри узла порядок LMS — сначала все материалы (`order_position`),
   потом все задания. Собрать для каждого листа `материалы[] → задания[]`. Наложить
   `course_dependencies`: узел X требует Y ⇒ Y раньше; нарушение = находка К2.
3. **Банк заданий (К4, К5):** для каждого `tasks` — `difficulty`, тип (из `task_content`/
   схемы), `solution_rules`. Пустой/размытый критерий = находка К5. Однотипность и
   непокрытие приёмов материала = находка К4.
4. **Содержание (К7, К11):** читать `content` материалов (для `type=text` — markdown) —
   это текст, что видит ученик; по нему проверяется понятность и точность.
5. **Цели (К1):** искать явные формулировки целей урока/модуля в материалах; их отсутствие
   или «знаниевость» = находка К1.

> **Правильные поля блочного JSON (иначе — ложный P0 «пустой контент»).** Проза лежит в
> `html`, вопросы/задания — в `stem`; поле `text` часто пустое и контент НЕ отражает.
> Детали полей — `naive-learner-review/references/sources.md`, Источник A. Подозрение на
> «тело отсутствует» подтверждать фактом (перечитать поле / запрос к `content_hub` / рендер
> живой L3) ДО объявления P0 — см. SKILL Шаг 3 «Верификация перед P0».

## Источник B. Живой сайт (WordPress, модель «Навигатор»)

Сайт victor-komlev.ru: WP + Elementor, **LMS-плагина нет**. Курс = ручная разметка
(посты + страница-навигатор). Забор страниц — `веб-запрос` по URL; для сложной вёрстки/JS —
браузер (браузер Codex).

> **Обязательно догружать L3.** `веб-запрос` по L1/L2 отдаёт только оглавление и цели, а не
> тела уроков. Критерии К2/К7/К11 (термин раньше объяснения, понятность, точность) и
> глубина К4 оцениваются **только по L3-урокам** — сначала вытащить их точные ссылки с L2,
> затем прочитать сам урок. Вывод по L2 без L3 = недоаудит (выявлено на smoke tsk-143).

**Структура «Навигатор» (L1 навигатор / L2 тема / L3 пост-урок)** — не дублируем:
модель, фикс. порядок блоков L2 и slug-паттерны описаны в
`C:/Users/user/.codex/skills/naive-learner-review/references/sources.md`, Источник B, и в
методистовом `lms-wp-export.md`, Часть 2. Ниже — что извлечь под критерии рубрики.

**Что собрать для рубрики:**
1. **Структура и навигация (К9):** L1 → порядок разделов и M. По каждому L2 проверить
   наличие «Краткого плана» и «Итогов», блок навигации в конце L3, счётчик «Раздел N из M»,
   работу deep-link `#anchor`. Битая/уводящая ссылка или неверный счётчик = находка К9.
   Ссылки проверять как фактические вложенные пермалинки `/<навигатор>/<тема>/<урок>/`.
2. **Траектория (К2):** порядок L3 из L2; термин раньше объяснения — по тексту уроков.
3. **Практика и оценивание (К4, К5):** блоки 💻 задания и ❓ вопросы — есть ли критерий,
   отрабатывают ли приёмы уроков, разнообразны ли, есть ли обратная связь.
4. **Содержание (К7, К11):** тексты L3 — понятность, точность, актуальность скриншотов.
5. **Мотивация и доступность (К6, К8):** старт раздела/урока (цепляет?), связь с жизнью,
   визуалы на понятия, несколько каналов подачи (текст/видео/схема).
6. **Цели (К1):** «цель урока»/«чему научитесь» в начале L2/L3; отсутствие = находка К1.

## Если источник не указан
Спросить оператора одним `короткий вопрос оператору`: какой курс аудируем и откуда — `course_uid`
в LMS или URL навигатора на сайте — **и** заявленную цель + ЦА курса (нужны для К1
выравнивания). Без конкретного курса аудит невозможен (не выдумывать материал).

## Смешанный случай
Один курс живёт и в LMS, и на сайте (общий ключ `course_uid`/`external_uid`, таблица
соответствия «LMS-узел ↔ WP-страница» в методистовом `lms-wp-export.md`, Часть 3). Если
оператор просит — аудировать обе проекции и отдельно отметить **рассинхрон** (тема есть
на сайте, но нет в LMS, или наоборот) как находку К3 (полнота) / К9 (структура).

## Парный запуск с обывателем
Если нужен полный обзор курса — методика **и** восприятие — запускать этот skill вместе
с `naive-learner-review`: эксперт даёт рубричные разрывы со ссылкой на принцип, обыватель
— живые точки, где реальный новичок спотыкается. Отчёты не дублируются: разные линзы.
