# /retro — Еженедельная инженерная ретроспектива

Генерирует комплексную инженерную ретроспективу анализируя историю коммитов, рабочие паттерны и метрики качества кода. Команда-осведомлённый: определяет пользователя запускающего команду, затем анализирует каждого контрибьютора с персональной похвалой и возможностями роста.

## Аргументы
- `/retro` — по умолчанию: последние 7 дней
- `/retro 24h` — последние 24 часа
- `/retro 14d` — последние 14 дней
- `/retro 30d` — последние 30 дней
- `/retro compare` — сравнить текущий период с предыдущим периодом той же длины
- `/retro compare 14d` — сравнить с явным окном

Если аргумент не распознан:
```
Использование: /retro [окно]
  /retro              — последние 7 дней (по умолчанию)
  /retro 24h          — последние 24 часа
  /retro 14d          — последние 14 дней
  /retro 30d          — последние 30 дней
  /retro compare      — этот период vs предыдущий
  /retro compare 14d  — сравнить с явным окном
```

## Шаг 0: Определить ветку по умолчанию

```bash
gh repo view --json defaultBranchRef -q .defaultBranchRef.name
```

Если упало — использовать `main`.

---

## Шаг 1: Сбор сырых данных

Определи текущего пользователя:
```bash
git fetch origin <ветка_по_умолч.> --quiet
git config user.name
git config user.email
```

Имя из `git config user.name` — это **"ты"** (человек читающий эту ретро). Все остальные авторы — твои коллеги.

Запусти **все эти команды параллельно** (они независимы):

```bash
# 1. Все коммиты в окне с временными метками, хешами, авторами
git log origin/<ветка> --since="<окно>" --format="%H|%aN|%ae|%ai|%s" --shortstat

# 2. Разбивка тест/продакшн LOC по коммитам
git log origin/<ветка> --since="<окно>" --format="COMMIT:%H|%aN" --numstat

# 3. Временны́е метки для определения сессий
git log origin/<ветка> --since="<окно>" --format="%at|%aN|%ai|%s" | sort -n

# 4. Самые часто меняемые файлы (hotspot-анализ)
git log origin/<ветка> --since="<окно>" --format="" --name-only | grep -v '^$' | sort | uniq -c | sort -rn

# 5. Количество коммитов по авторам
git shortlog origin/<ветка> --since="<окно>" -sn --no-merges

# 6. Hotspot-файлы по авторам
git log origin/<ветка> --since="<окно>" --format="AUTHOR:%aN" --name-only

# 7. TODOS.md бэклог (если есть)
cat TODOS.md 2>/dev/null || true
```

## Шаг 2: Вычислить метрики

| Метрика | Значение |
|---------|---------|
| Коммитов в main | N |
| Контрибьюторов | N |
| Всего добавлений | N |
| Всего удалений | N |
| Чистые LOC добавлено | N |
| Тест LOC (добавления) | N |
| Доля тест LOC | N% |
| Активных дней | N |
| Обнаружено сессий | N |
| Среднее LOC/час-сессии | N |

Затем **таблица лидеров по авторам:**

```
Контрибьютор           Коммиты   +/-            Основная область
Ты (имя)                    32   +2400/-300     src/
alice                       12   +800/-150      tests/
bob                          3   +120/-40       docs/
```

Сортировать по коммитам убыванием. Текущий пользователь всегда первый, помечен "Ты (имя)".

## Шаг 3: Анализ качества кода

**Тест-дисциплина:**

Вычисли долю тестового LOC (тесты / всего LOC):
- ≥ 30%: Отличная дисциплина тестирования
- 15-30%: Умеренная тестовая дисциплина
- < 15%: Слабое покрытие тестами — флагировать

**Hotspot-анализ:**

Топ-5 самых изменяемых файлов — они файлы с высоким риском:
- > 5 коммитов за период: горячая точка изменений
- Изменяются несколькими авторами: потенциальная связанность
- Примечание: высокая активность может быть хорошей (активная разработка) или плохой (нестабильный код)

**Паттерны технического долга:**

Пройдись по diff и найди:
- Коммиты с TODO/FIXME в diff
- Откаты (коммиты "revert" или "fix bug introduced in")
- Паттерны "быстрого исправления" в сообщениях коммитов

## Шаг 4: Анализ команды

Для **каждого** контрибьютора создай персональную секцию:

```
### {Имя} (Ты!) / {Имя} (коллега)
Коммиты: N  |  Добавлено: +X  |  Удалено: -Y  |  Тест LOC: Z%

🌟 ПОХВАЛА:
- {конкретное достижение из коммитов}
- {паттерн качества который они демонстрировали}
- {импакт их работы на проект}

📈 ВОЗМОЖНОСТИ РОСТА:
- {конкретное наблюдение основанное на данных, не общая критика}
- {предложение с конкретным примером}

Основные области: {топ-3 директории/модуля}
```

Правила:
- Хвали конкретно (ссылайся на реальные коммиты/файлы)
- Точки роста основывай на данных (доля тестов, hotspot-файлы, паттерны откатов)
- Никогда не называй точку роста без конструктивного предложения
- Для себя ("Ты") — честнее и конкретнее

## Шаг 5: Состояние бэклога и прогресс TODO

Прочитай `TODOS.md` (уже получен в Шаге 1, команда 7). Если нет — пропустить секцию.

Вычислить:
- Открытых TODO: N
- Выполненных в этом периоде: N (из коммит-сообщений или изменений TODOS.md)
- Добавленных новых TODO: N
- Чистое изменение бэклога: ±N

```
Бэклог TODO: {N} открытых
  Завершено в этом периоде: {N} ✓
  Добавлено новых:          {N}
  Чистое изменение:         {+/-N}
```

Флагировать если бэклог растёт > 5 пунктов в неделю.

## Шаг 6: Режим сравнения (только при `/retro compare`)

Собери те же метрики для предыдущего периода той же длины:

```bash
git log origin/<ветка> --since="<2*окно> ago" --until="<окно> ago" ...
```

Показать таблицу сравнения:

```
Метрика              Этот период    Предыдущий    Изменение
-------------------  -----------    ----------    ---------
Коммитов             32             24            +33% ↑
Контрибьюторов        3              2            +50% ↑
Добавлено LOC        2400          1800           +33% ↑
Доля тест LOC         28%           22%           +6pp ↑
Откаты                1              3            -67% ↑
```

```
{1-3 конкретных предложения для следующей недели}

```

---

## Правила форматирования
* Используй конкретные примеры из реальных коммитов — никаких обобщений
* Данные сначала, интерпретация потом
* Похвала раньше точек роста
* Рекомендации должны быть выполнимы в течение следующей недели
* Никогда не публикуй этот отчёт без разрешения пользователя — данные по производительности личные
